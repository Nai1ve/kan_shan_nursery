# 看山小苗圃-算法与模块详细设计

本文档专门沉淀看山小苗圃后续开发需要的算法设计和模块详细设计。

目标是解决三个问题：

1. 每个模块到底负责什么、不负责什么。
2. 每个核心算法输入是什么、怎么计算、输出到哪里。
3. 后续交给不同 AI Coding 工具时，任务边界足够清晰，不需要对方自行发挥产品逻辑。

本文档以 `docs/看山小苗圃-产品文档.md` 和 `docs/看山小苗圃-技术文档.md` 为产品依据，以 `docs/看山小苗圃-接口功能与数据流文档.md` 为接口依据。

---

## 1. 全局闭环

系统主链路：

```text
用户画像 / Memory
↓
今日看什么：输入筛选与卡片推流
↓
用户轻量表态：认同 / 反对 / 有疑问 / 加入种子 / 基于它写
↓
观点种子库：沉淀观点、疑问、浇水材料
↓
今日发芽：历史种子 × 今日热点 / 新材料 / 关注流
↓
写作苗圃：Memory 注入、观点确认、蓝图、初稿、圆桌审稿
↓
模拟发布 / 用户发布
↓
历史反馈：评论摘要、二次种子、Memory 更新建议
```

核心对象流转：

```text
ZhihuContentItem
→ ContentSource
→ WorthReadingCard
→ IdeaSeed
→ WateringMaterial / SeedQuestion
→ SproutOpportunity
→ WritingSession
→ FeedbackArticle
→ MemoryUpdateRequest
```

## 2. 模块详细设计

## 2.1 frontend

### 定位

前端是用户工作台，不是业务规则的最终来源。P0 可以保留 mock/localStorage，但后续应逐步接入 `api-gateway`。

### 页面模块

| 页面 | 目标 | 关键状态 |
| --- | --- | --- |
| 登录 / 画像采集 | 建立用户身份和 Memory | ProfileData |
| 今日看什么 | 展示当前输入分类的值得看卡片 | categories, cards, categoryRefreshState |
| 种子库 | 管理观点种子和浇水材料 | seeds, questions, wateringMaterials |
| 今日发芽 | 用户主动触发发芽计算 | sproutRun, opportunities |
| 写作苗圃 | 写作流程和 Memory 注入可见 | writingSession, memoryOverride |
| 历史反馈 | 发布后反馈与二次种子 | feedbackArticles, memoryUpdateRequests |
| 个人画像 | Memory 展示、编辑、确认更新 | globalMemory, interestMemories |

### 前端必须遵守

- 只通过 `api-gateway` 调后端。
- 不直接调用 `zhihu-adapter`。
- 不直接调用 `llm-service`。
- 不在前端计算成熟度、发芽分等核心业务分数，只展示服务返回结果。
- 原始来源在卡片内固定高度滚动展示，不只跳转链接。
- Memory 注入必须可见，并允许用户编辑当前写作 session 的 `memoryOverride`。

---

## 2.2 api-gateway

### 定位

前端唯一入口。

### 负责

- 统一 `request_id`。
- 注入 demo user 上下文。
- 代理内部服务。
- 简单聚合多服务响应。
- 统一错误格式。
- 对前端隐藏内部服务地址。

### 不负责

- 不计算推荐分。
- 不计算成熟度。
- 不调用知乎 API。
- 不调用直答 Agent。
- 不保存业务主状态。

### 成功响应 envelope

```json
{
  "request_id": "req-xxx",
  "data": {},
  "meta": {}
}
```

### 错误响应 envelope

```json
{
  "request_id": "req-xxx",
  "error": {
    "code": "SERVICE_NOT_READY",
    "message": "Service is not ready: content",
    "detail": {}
  }
}
```

### 服务未就绪策略

未完成服务不做假代理，直接返回：

```text
SERVICE_NOT_READY
```

这样前端和联调能明确知道当前阻塞点。

## 2.3 profile-service / memory 子模块

### 定位

管理用户画像和 Memory。P0 不拆独立 memory-service。

### 负责

- demo 用户资料。
- 全局 Memory。
- 兴趣分类 Memory。
- Memory 注入摘要。
- Memory 更新请求。
- Memory 版本记录。

### 不负责

- 不生成内容卡片。
- 不创建种子。
- 不写作。
- 不直接读取知乎接口。

### 核心数据

```text
ProfileData
GlobalMemory
MemorySummary
MemoryInjectionSummary
MemoryUpdateRequest
```

### Memory 注入算法

输入：

```text
interestId
ProfileData.globalMemory
ProfileData.interestMemories[]
```

流程：

```text
1. 根据 interestId 找到兴趣 Memory。
2. 读取全局 Memory。
3. 拼接可展示摘要 displaySummary。
4. 返回 interestMemory + globalMemory + displaySummary + editable=true。
```

输出：

```ts
MemoryInjectionSummary {
  interestId: string;
  interestMemory: MemorySummary;
  globalMemory: GlobalMemory;
  displaySummary: string;
  editable: true;
}
```

关键规则：

- 如果兴趣不存在，返回明确错误。
- 前端允许用户在写作 session 中修改副本。
- 修改副本不回写长期 Memory。
- 需要长期更新时，创建 `MemoryUpdateRequest`。

### Memory 更新请求算法

输入：

```text
interestId
targetField
suggestedValue
reason
```

流程：

```text
1. 创建 pending 请求。
2. 不修改 profile memory。
3. 用户 apply 后，按 targetField 写入 global 或 interest memory。
4. 用户 reject 后，只更新请求状态。
5. 保存旧版本快照，便于后续回滚。
```

状态：

```text
pending → applied
pending → rejected
```

## 2.4 zhihu-adapter

### 定位

知乎开放能力唯一出口。

### 负责

- 三类 Client：Community / OAuth / Data Platform。
- 三类鉴权隔离。
- Redis 短缓存。
- quota 计数。
- 官方 raw 字段映射到 `ZhihuContentItem`。
- mock/live provider。

### 不负责

- 不生成 WorthReadingCard。
- 不生成观点种子。
- 不写用户画像。
- 不决定内容是否值得写。

### 缓存算法

输入：

```text
endpoint
normalized parameters
userId
```

流程：

```text
1. 根据 endpoint 和参数生成 Redis key。
2. 查缓存。
3. 命中：返回缓存，不增加 quota。
4. 未命中：检查 quota。
5. quota 未超：回源调用知乎 API。
6. 回源成功：写缓存，增加 quota。
7. 回源失败：不增加 quota，返回错误或 fallback mock。
```

关键 Redis key：

```text
zhihu:hot_list:{limit}
zhihu:zhihu_search:{query_hash}:{count}
zhihu:global_search:{query_hash}:{count}
zhihu:direct_answer:{model}:{messages_hash}:stream_false
zhihu:ring_detail:{ring_id}:{page_num}:{page_size}
zhihu:comment_list:{content_type}:{content_token}
zhihu:story_list
zhihu:story_detail:{work_id}
zhihu:user_moments:{user_id}
quota:{endpoint}:{user_id}:{yyyyMMdd}
```

### quota 算法

```text
used = Redis.get(quotaKey)
if used >= limit:
  return QUOTA_EXCEEDED
else:
  call upstream
  if success:
    Redis.incr(quotaKey)
```

默认限制：

| 能力 | 限额 |
| --- | --- |
| hot_list | 100 / day |
| zhihu_search | 1000 / day |
| global_search | 1000 / day |
| direct_answer | 100 / day |

### 字段归一化算法

所有业务服务只消费：

```text
ZhihuContentItem
```

映射规则：

- 热榜无官方 ID，用 URL hash 生成 `sourceId`。
- 搜索 `RankingScore` 映射为 `relevanceScore`。
- 全网搜索 `ContentText` 可能含 `<em>`，保留 raw html，同时生成去标签 summary。
- 故事内容必须带 `usageNotice`。
- 关注流映射 actor、action、target。

## 2.5 content-service

### 定位

解决“今日看什么”。它是创作输入流服务。

### 负责

- 输入分类：兴趣小类、关注流、偶遇输入。
- Query Plan 生成。
- 调用 zhihu-adapter 获取候选。
- 调用 llm-service 做摘要、争议、角度。
- 聚合多来源为 WorthReadingCard。
- 卡片评分、去重、多样性。
- 当前分类刷新。
- 来源完整展示。

### 不负责

- 不创建种子。
- 不计算种子成熟度。
- 不执行今日发芽。
- 不直接调用知乎 API。

### Query Plan 生成算法

输入：

```text
categoryId
categoryKind
interestMemory
globalMemory
historicalSeeds
refreshState
```

流程：

```text
1. 如果 categoryKind=interest：
   - 从 interestMemory.preferredPerspective 提取主题词。
   - 从 writingReminder 提取风险词和反方词。
   - 从 historicalSeeds 提取近期主张关键词。
   - 拼出 zhihu_search / global_search query。

2. 如果 categoryKind=following：
   - 使用关注动态接口。
   - 可补充当前主兴趣关键词。
   - 重点过滤观点密度和可补种子能力。

3. 如果 categoryKind=serendipity：
   - 从用户主兴趣找相邻主题。
   - 使用热榜、全网搜索、故事 API。
   - 加入 novelty / remoteAssociation 约束。
```

输出示例：

```ts
QueryPlan {
  categoryId: "ai-coding";
  keywords: ["AI Coding", "程序员成长", "研发工作流"];
  sources: ["hot_list", "zhihu_search", "global_search"];
  count: 10;
}
```

### 候选召回算法

输入：QueryPlan。

流程：

```text
1. 根据 sources 并行请求 zhihu-adapter。
2. 获取 ZhihuContentItem[]。
3. 按 sourceType 分桶。
4. 去除明显空摘要、空标题、重复 URL。
5. 保留 raw，用于来源详情展示。
```

### 多来源卡片聚合算法

目标：一张 WorthReadingCard 可以由 1-3 个来源组成。

流程：

```text
1. 选择主来源：最高 relevance / heat / ranking 的内容。
2. 找支持来源：标题或摘要相近，能支撑主来源观点。
3. 找反方来源：评论、全网搜索或争议文本中含反方信号。
4. 组成 originalSources[]。
5. 调 llm-service 生成 contentSummary / controversies / writingAngles。
6. 计算 cardScore。
```

合并条件：

```text
sameTopic = titleSimilarity > threshold
或 keywordOverlap >= 2
或 source.searchHashId 相同
```

### 卡片评分算法

当前基础公式：

```text
relevanceScore =
兴趣匹配 35%
+ authorityLevel / rankingScore 20%
+ VoteUpCount / CommentCount 15%
+ 争议点数量 15%
+ EditTime / 热榜排序 10%
+ 可写角度质量 5%
```

建议落地公式：

```text
cardScore =
interestFit * 0.28
+ sourceQuality * 0.18
+ freshness * 0.12
+ controversy * 0.14
+ writingPotential * 0.16
+ memoryFit * 0.08
+ diversityGain * 0.04
```

各项计算：

```text
interestFit = keywordOverlap(card, interestMemory) / maxKeywords
sourceQuality = normalized(authorityLevel, rankingScore, likeCount, commentCount)
freshness = timeDecay(publishedAt) 或 hotRankScore
controversy = min(100, controversies.length * 30 + selectedComments反方信号)
writingPotential = writingAngles.length * 25 + 是否能形成明确主张
memoryFit = evidencePreference / preferredPerspective 匹配度
diversityGain = 与已选卡片不相似时加分
```

### 多样性去重算法

```text
selected = []
for candidate in sortedByScore:
  if duplicateSource(candidate): skip
  if tooSimilarToSelected(candidate): skip
  if sourceTypeQuotaExceeded(candidate): skip
  selected.append(candidate)
  if len(selected) == targetCount: break
```

P1 可升级为 MMR：

```text
mmrScore = lambda * relevance - (1 - lambda) * maxSimilarity(candidate, selected)
```

### 刷新输入算法

输入：categoryId。

流程：

```text
1. 读取 categoryRefreshState。
2. 不重新创建用户资产。
3. 从候选池轮换 visibleCardIds。
4. 轻微重排卡片顺序。
5. 更新 refreshCount / refreshedAt。
```

约束：刷新不影响种子库、写作 session、用户画像、历史反馈。

### 偶遇输入算法

目标：打破信息茧房，但不能随机。

计算：

```text
serendipityScore =
remoteAssociation * 0.35
+ novelty * 0.25
+ explainability * 0.20
+ writingPotential * 0.15
- forcedAnalogyRisk * 0.20
```

解释：

- `remoteAssociation`：与用户兴趣的远端类比强度。
- `novelty`：和用户近期内容不相似。
- `explainability`：能说清楚为什么推荐。
- `forcedAnalogyRisk`：类比过度则扣分。

故事内容特殊规则：

- 只进入创意表达参考或偶遇输入。
- 必须展示署名和活动限制。
- 不作为事实论证证据。

## 2.6 seed-service

### 定位

观点种子状态中心。

### 负责

- 从卡片创建或更新种子。
- 用户 reaction。
- 疑问线程。
- 四格浇水材料。
- 成熟度计算。
- 合并相似种子。

### 不负责

- 不搜索内容。
- 不生成内容卡片。
- 不生成完整文章。
- 不更新长期 Memory。

### 从卡片生成种子算法

输入：

```text
WorthReadingCard
reaction
userNote
```

幂等规则：

```text
如果 createdFromCardId 已存在：
  更新 userReaction / userNote / updatedAt
  不重复创建 seed
否则：
  新建 seed
```

字段规则：

```text
title = card.writingAngles[0] || card.title
coreClaim = card.writingAngles[0] || card.title
sourceSummary = card.contentSummary
possibleAngles = card.writingAngles
counterArguments = card.controversies
requiredMaterials = ["补充个人经验", "补充反方回应", "补充可引用来源"]
```

初始材料：

```text
evidence:
  card.originalSources[0].summary 或 fullContent 摘录
  adopted = true

counterargument:
  card.controversies[0]
  reaction=disagree 时 adopted = true
```

### 疑问线程算法

输入：

```text
seedId
question
parentQuestionId?
```

流程：

```text
1. 创建 SeedQuestion。
2. 如果 parentQuestionId 存在，继承 threadId。
3. 调 llm-service answer-seed-question。
4. 写入 open_question material。
5. 根据回答类型写入 evidence 或 counterargument material。
6. 重新计算 maturityScore。
```

状态处理：

```text
answered: Agent 已回答，等待用户判断
resolved: 用户确认已解决，open_question adopted=true
needs_material: 仍需补材料，open_question 保留未采纳
```

### 浇水材料算法

四类材料：

```text
evidence
counterargument
personal_experience
open_question
```

新增材料：

```text
material.id = stable unique id
material.adopted = 用户选择或 Agent 默认
插入 wateringMaterials
recalc maturityScore
```

Agent 补证据：

```text
seed + existing evidence + sources
→ llm-service supplement-material(type=evidence)
→ 新 evidence material
```

Agent 找反方：

```text
seed + coreClaim + existing counterArguments
→ llm-service supplement-material(type=counterargument)
→ 新 counterargument material
```

个人经验：

- Agent 只能提问引导。
- 不允许编造用户经历。

### 成熟度算法

当前公式：

```text
maturityScore =
min(96, 32 + 已采纳材料类型数 * 14 + 疑问总数 * 3 + 已解决疑问数 * 4)
```

状态流转：

```text
if status in [writing, published]: keep
else if maturityScore >= 70: sproutable
else: water_needed
```

后续可扩展：

```text
maturityScore =
base
+ adoptedMaterialTypeCoverage
+ sourceQuality
+ resolvedQuestionBonus
+ personalExperienceBonus
+ counterResponseBonus
- unresolvedCriticalQuestionPenalty
```

### 合并相似种子算法

输入：targetSeedId, sourceSeedId。

流程：

```text
1. 读取 target 和 source。
2. 合并 possibleAngles 去重。
3. 合并 counterArguments 去重。
4. 合并 requiredMaterials 去重。
5. 合并 wateringMaterials。
6. 合并 questions，保留 threadId。
7. userNote 添加合并说明。
8. 删除或归档 source。
9. 重算 maturityScore。
```

相似种子推荐算法：

```text
similarity = titleSimilarity * 0.35
+ coreClaimSimilarity * 0.35
+ interestIdMatch * 0.15
+ sourceOverlap * 0.15
```

P0 可以用关键词交集，P1 可用向量相似度。

## 2.7 llm-service

### 定位

LLM 任务执行层。

### 负责

- prompt 模板。
- task schema。
- mock provider。
- 直答 Agent wrapper。
- LLM 缓存。
- fallback。
- 输出校验。

### 不负责

- 不保存业务主状态。
- 不直接读取知乎 API。
- 不决定 seed / writing 状态。

### 任务类型

```text
summarize-content
extract-controversies
generate-writing-angles
answer-seed-question
supplement-material
sprout-opportunities
argument-blueprint
draft
roundtable-review
feedback-summary
```

### LLM 缓存算法

输入：

```text
taskType
input
promptVersion
schemaVersion
```

缓存 key：

```text
llm:{task_type}:{input_hash}:{prompt_version}:{schema_version}
```

流程：

```text
1. validate request。
2. 生成 input_hash。
3. 查缓存。
4. 命中直接返回。
5. miss 后调用 provider。
6. validate output。
7. 写缓存。
```

### provider fallback 算法

```text
if provider_mode=mock:
  run mock task
else:
  call zhihu-adapter /zhihu/direct-answer
  if success and valid JSON:
    return parsed output
  else:
    return mock fallback with fallback=true
```

### 输出校验原则

- 每个 task 有 required keys。
- 不允许直接把自然语言长文塞给前端渲染，除非 task 本来就是 draft。
- 失败时返回明确 `fallback=true` 或错误。

### 成本控制

- 高频任务必须缓存：摘要、争议点、角度。
- 昂贵任务用户触发：发芽、圆桌审稿、初稿。
- P0 默认非流式。
- 相同输入不重复调用直答。

## 2.8 sprout-service

### 定位

今日发芽服务，负责把历史种子和今日新信息碰撞。

### 负责

- 用户主动触发发芽。
- 候选种子筛选。
- 新信息召回。
- seed-topic 匹配。
- activationScore。
- 发芽机会生成。
- 发芽动作：补资料、换角度、暂时不写、开始写作。

### 不负责

- 不生成完整文章。
- 不维护长期 Memory。
- 不直接调用知乎 API。

### 发芽运行算法

输入：

```text
userId
interestId?
seedIds?
timeRangeHours
includeHotList
includeZhihuSearch
includeGlobalSearch
includeFollowingFeed
```

流程：

```text
1. 读取用户 Memory。
2. 读取未发布 seeds。
3. 过滤候选种子。
4. 拉取热榜 / 搜索 / 关注流缓存数据。
5. 生成 seed topic 向量或关键词。
6. 生成新信息 topic 向量或关键词。
7. 计算匹配和新信息增益。
8. 计算 activationScore。
9. 调 llm-service 生成发芽解释、标题、角度。
10. 返回 Top N opportunities。
```

### 候选种子筛选

```text
status in [water_needed, sproutable, high_timeliness]
not published
maturityScore >= 45
```

加权优先：

- maturityScore 高。
- 近 14 天有更新。
- 有已采纳 evidence。
- 有反方材料。
- 有未解决问题但问题清晰。

### seed-topic 匹配算法

P0 关键词版：

```text
seedKeywords = tokenize(seed.title + seed.coreClaim + possibleAngles + materials)
topicKeywords = tokenize(hot.title + summary + comments)
matchScore = overlap(seedKeywords, topicKeywords) / len(seedKeywords)
```

P1 向量版：

```text
matchScore = cosine(embedding(seed), embedding(topic))
```

### 新信息增益算法

```text
newInfoGain =
newEvidenceGain * 0.4
+ newCounterArgumentGain * 0.3
+ newCaseGain * 0.2
+ newTimelinessGain * 0.1
```

判断依据：

- 今天出现新案例。
- 今天出现新反方观点。
- 今天有新热点或事件。
- 今天的内容能补齐种子缺失材料。

### Activation Score

```text
activationScore =
seedMaturity * 0.25
+ hotRelevance * 0.25
+ timeliness * 0.20
+ newInfoGain * 0.15
+ writingPotential * 0.15
- duplicatePenalty
- evidencePenalty
```

惩罚项：

```text
duplicatePenalty = 最近发布过同主题 ? 15 : 0
evidencePenalty = 无 adopted evidence ? 12 : 0
```

阈值建议：

| 分数 | 结果 |
| --- | --- |
| >= 80 | 强烈建议写 |
| 65-79 | 可以写，但建议补材料 |
| 50-64 | 适合继续浇水 |
| < 50 | 不展示 |

### 发芽动作算法

| 动作 | 落点 |
| --- | --- |
| 补充资料 | 写入 seed wateringMaterials |
| 换个角度 | 更新 opportunity.suggestedAngle，不改 seed 主张 |
| 暂时不写 | opportunity.status=dismissed |
| 开始写作 | 调 writing-service 创建 session |

## 2.9 writing-service

### 定位

写作苗圃服务，把成熟观点种子转成可编辑文章。

### 负责

- WritingSession。
- Memory 注入和 memoryOverride。
- 核心观点确认。
- 文章类型、语气、读者收益。
- 论证蓝图。
- 初稿。
- 圆桌审稿。
- 定稿。
- 模拟发布。

### 不负责

- 不直接读取知乎 API。
- 不创建基础内容卡片。
- 不直接覆盖 profile Memory。
- 不绕过用户确认发布真实内容。

### session 创建算法

输入：

```text
seedId
articleType
tone?
```

流程：

```text
1. 读取 seed。
2. 校验 seed.status != published。
3. 根据 seed.interestId 读取 memory injection。
4. 创建 writingSession。
5. memoryOverride = injection.interestMemory 副本。
6. seed.status = writing。
```

### 观点确认算法

输入：seed.coreClaim, userAdjustment, tone。

输出：

```text
coreClaim
takeaway
articleThesis
riskNotes
```

规则：

- “更犀利”可以提高立场明确度，但不能编造证据。
- “更稳健”应增加边界条件和反方承认。
- “补充个人经历”只能引导用户填写，不能编造。

### 论证蓝图算法

输入：

```text
seed
materials
questions
memoryOverride
articleType
```

输出：

```text
centralClaim
background
arguments[]
counterArgument
responseStrategy
suggestedPersonalExperience[]
```

蓝图必须先于初稿生成。

### 初稿生成算法

输入：argumentBlueprint, memoryOverride, tone。

输出：title, body, aiDisclosureSuggestion。

规则：

- 必须引用 seed materials。
- 不把 open_question 当事实证据。
- 缺个人经验时保留占位提醒。
- 保留 AI 辅助声明建议。

### 圆桌审稿算法

输入：draft, blueprint, seed, memoryOverride。

并行或串行执行四个 reviewer：

```text
logic_reviewer
human_editor
opponent_reader
community_editor
```

每个 reviewer 输出：

```text
summary
problems[]
suggestions[]
severity
```

审稿聚合：

```text
high severity 问题必须展示在顶部
同类建议合并
用户可选择采纳 / 忽略
采纳建议进入下一版 draft
```

### 模拟发布算法

流程：

```text
1. writingSession.status = published
2. seed.status = published
3. seed.maturityScore = 96
4. 创建 feedbackArticle mock
5. 如果需要真实发布，必须走用户确认和 zhihu-adapter publish
```

## 2.10 feedback-service

### 定位

历史反馈服务，把发布表现转化为新观点资产。

### 负责

- 已发布文章列表。
- 指标摘要。
- 评论摘要。
- 二次文章建议。
- 二次种子 payload。
- Memory 更新建议。

### 不负责

- 不直接更新 Memory。
- 不直接发布文章。
- 不修改原文。

### 反馈分析算法

输入：

```text
article
metrics
comments
linkedSeed
```

流程：

```text
1. 分析指标：阅读、点赞、收藏、评论。
2. 评论聚类：支持、反方、补充材料、无关。
3. 提取高价值评论。
4. 生成 performanceSummary。
5. 生成 secondArticleIdeas。
6. 生成 seed payload。
7. 生成 memory_update_request payload。
```

### 评论聚类算法

P0 规则版：

```text
if comment contains 支持/认同/确实: positive
if comment contains 但是/不一定/问题是/反过来: counterargument
if comment contains 案例/数据/补充/链接: supplementary
else: neutral
```

P1 LLM 版：

```text
comments[] → llm-service feedback-summary → structured clusters
```

### Memory 更新建议算法

触发条件：

- 多条评论指出同一写作缺陷。
- 评论补充了稳定的新证据偏好。
- 某兴趣分类下多篇文章反馈相同。

输出：

```text
MemoryUpdateRequest {
  interestId;
  targetField;
  suggestedValue;
  reason;
  status: pending;
}
```

## 3. 端到端关键算法组合

### 3.1 今日看什么完整链路

```text
profile-service: 读取 Memory
↓
content-service: 生成 Query Plan
↓
zhihu-adapter: 读取缓存 / 回源
↓
content-service: 候选去重、聚合来源
↓
llm-service: 摘要、争议、可写角度
↓
content-service: 评分、多样性选择
↓
frontend: 展示当前分类卡片和原始来源
```

### 3.2 有疑问完整链路

```text
frontend: 用户输入 question
↓
seed-service: 创建或找到 seed
↓
seed-service: 创建 SeedQuestion
↓
llm-service: answer-seed-question
↓
seed-service: 写入 open_question 和 evidence/counterargument
↓
seed-service: 重算 maturityScore
↓
frontend: 显示 answered/resolved/needs_material
```

### 3.3 今日发芽完整链路

```text
frontend: 用户点击开始今日发芽
↓
sprout-service: 读取 Memory + seeds
↓
sprout-service: 过滤候选种子
↓
zhihu-adapter/content-service: 获取今日新信息
↓
sprout-service: seed-topic 匹配 + newInfoGain
↓
sprout-service: activationScore
↓
llm-service: 生成机会解释和角度
↓
frontend: 展示 SproutOpportunity
```

### 3.4 写作完整链路

```text
frontend: 开始写作
↓
writing-service: 读取 seed
↓
profile-service: 读取 memory injection
↓
writing-service: 创建 session + memoryOverride
↓
llm-service: argument-blueprint
↓
llm-service: draft
↓
llm-service: roundtable-review
↓
writing-service: finalize / publish mock

```

## 4. 评测指标

### 4.1 内容输入质量

| 指标 | 含义 |
| --- | --- |
| 来源覆盖率 | 每张卡片是否有可追溯来源 |
| 多来源率 | 重点卡片是否有 2-3 个来源 |
| 争议点有效率 | 是否能形成真实正反讨论 |
| 可写角度有效率 | 用户是否能从角度进入种子 |
| 偶遇解释率 | 偶遇输入是否能解释远端关联 |

### 4.2 种子质量

| 指标 | 含义 |
| --- | --- |
| 种子重复率 | 同 cardId 是否重复建种子 |
| 材料覆盖率 | evidence/counterargument/personal/open_question 覆盖 |
| 成熟度提升率 | 浇水后 maturityScore 是否合理上升 |
| 已解决疑问率 | question thread 是否闭环 |

### 4.3 发芽质量

| 指标 | 含义 |
| --- | --- |
| 激活准确率 | 展示机会是否和种子真实相关 |
| 新信息增益 | 是否不是旧观点重复包装 |
| 写作转化率 | 用户是否点击开始写作 |
| 补资料率 | 用户是否认可缺失材料判断 |

### 4.4 写作质量

| 指标 | 含义 |
| --- | --- |
| Memory 可见率 | 写作时是否展示注入 Memory |
| 观点确认率 | 用户是否确认或调整核心观点 |
| 审稿有效率 | 圆桌建议是否被采纳 |
| AI 味风险 | 是否缺个人经历、过度模板化 |

### 4.5 成本指标

| 指标 | 含义 |
| --- | --- |
| 缓存命中率 | 热榜、搜索、LLM 结果命中情况 |
| quota 使用率 | 每日知乎能力消耗 |
| LLM 调用次数 | 每条链路消耗 |
| fallback 率 | provider 失败或输出不合法比例 |

## 5. 开发顺序建议

在额度和人力有限时，不要同时追求所有服务完整。

推荐顺序：

1. 先让 `api-gateway + profile + seed + llm + zhihu-adapter` 跑通。
2. 做 `content-service` 的规则版推流，不急着做向量。
3. 做“有疑问”从卡片到种子的完整闭环。
4. 做 `sprout-service` 的规则版 activationScore。
5. 做 `writing-service` 的 Memory 注入和状态机。
6. 做 `feedback-service` 的 mock 反馈和 Memory 更新请求。
7. 最后再接真实知乎 live 和算法调优。

每一步都要有 mock 数据和测试，不要依赖真实接口才能演示。
