# 看山小苗圃-产品文档

## 一、产品一句话说明

看山小苗圃是一个面向知乎创作者的读写一体 AI 创作 Agent，帮助你

**看到好内容，形成好观点，写出好文章。**

它不是一个从空白页开始的 AI 创作工具，而是帮助你完成完整的创作链路：

```
看到好内容
↓
产生想法（记录）
↓
沉淀观点种子（辩论，搜集资料）
↓
等待合适时机发芽（想要写作｜有热点关联到先前的种子提示你可以写作）
↓
进入写作流程
↓
经过圆桌审稿（不同身份的AI进行探讨）
↓
形成可修改、可发布的文章草稿
↓
发布后根据历史反馈继续优化（AI进行数据分析整理的展示）
```

---

## 二、适合哪些用户？

看山小苗圃适合这些用户：

- 经常看知乎、收藏内容，但很少真正写出来的人；
- 有一些想法，但表达能力或结构能力不足的人；
- 想写技术分析、经验复盘、观点评论、知乎回答的人；
- 希望 AI 帮忙整理资料、提炼观点、打磨逻辑，但不希望 AI 替自己决定立场的人；
- 希望长期积累自己的观点库、写作风格和创作 Memory 的内容创作者。

---

## 三、产品核心原则

```
看到好内容，形成好观点，写出好文章。
```

这句话对应产品的三个阶段：

| 阶段 | 产品能力 | 用户获得什么 |
| --- | --- | --- |
| 看到好内容 | 今日看什么 | 不再盲刷信息流，看到更适合自己创作的内容 |
| 形成好观点 | 种子库 / 今日发芽 | 把阅读反应变成可写作的观点种子 |
| 写出好文章 | 写作苗圃 / 圆桌审稿 | 把观点发展成结构清晰、有个人味的文章 |

## 四、页面设计

## 1. 登录与个人画像

首次使用时，用户可以选择两种方式进入系统：

### 方式一：关联知乎账号

如果允许关联知乎账号，系统可以读取：

- 关注流；
- 圈子互动；
- 热榜偏好；
- 搜索与阅读偏好；
- 历史发布内容和反馈。

系统会基于这些信息快速建立用户的创作画像。

### 方式二：首次画像采集

如果暂时不关联知乎账号，用户可以通过问卷建立本地画像。

用户需要填写：

- 昵称；
- 身份背景；
- 兴趣小类；
- 不希望 AI 做什么；
- 写作风格问卷。

写作风格问卷会围绕这些方面建立画像：

- 是否重视逻辑；
- 是否愿意表达鲜明立场；
- 是否喜欢加入个人经历；
- 是否希望系统主动提出反方质疑；
- 标题更克制还是更有传播性；
- 文章更像知乎回答还是公众号长文；
- 希望 AI 帮到大纲、段落、初稿、修改还是发布建议。

---

## 2. 个人画像 Memory

系统会为用户维护一个可见、可编辑的画像 Memory。

它不是隐藏在后台的黑箱，而是用户可以直接查看和修改的内容。

画像包括：

```
长期背景
内容偏好
写作风格
推荐策略
写作风险提醒
```

例如：

```
长期背景：
有 Java 后端开发经验，做过金融软件与复杂数据同步项目，目前关注 AI / LLM / Agent 工程化落地。

内容偏好：
偏好工程复盘、问题拆解、反方质疑、真实案例。更重视“为什么这样设计”，而不是单纯罗列概念。

写作风格：
清晰、克制、偏工程师复盘视角；允许有观点锋芒，但避免标题党和情绪煽动。
```

### 重要设计：画像按兴趣分类绑定

用户不是在所有领域里都保持同一种状态。

例如同一个用户：

| 兴趣分类 | 用户状态可能不同 |
| --- | --- |
| Agent 工程化 | 更专业，更关注系统设计和工程落地 |
| 程序员成长 | 更愿意加入个人经历和职业判断 |
| 金融市场 | 可能更谨慎，需要更多事实依据 |
| 医学 AI | 可能更重视论文、指标和临床风险 |
| 内容创作 | 可能更关注传播、读者反馈和表达方式 |

所以系统的用户画像不应该只有一份总画像，而应该是：

```
全局用户画像
+
按兴趣分类维护的局部画像
```

## 3. 今日看什么

“今日看什么”不是普通信息流推荐。

它的目标不是让用户看更多内容，而是帮用户找到：

```
值得看
值得想
值得收藏
值得写
```

的内容。

### 展示逻辑

页面会按照“输入分类”进行切换展示。

输入分类分为三类：

1. 兴趣小类；
2. 关注流；
3. 偶遇输入。

关注流和偶遇输入不是挂在某个兴趣小类下面的子模块，而是与兴趣小类平级的入口。

例如：

```
Agent 工程化
AI Coding
RAG / 检索
后端工程
程序员成长
金融风控
医学 AI
产品设计
内容创作
关注流
偶遇输入
```

用户点击某个输入分类后，页面只展示该分类下的内容卡片。

### 重要调整

当前设计需要记录为：

> **兴趣小类、关注流、偶遇输入应作为平级入口切换展示，不应该在一个页面同时塞满所有分类的全部内容。**
>

正确逻辑是：

```
用户点击「AI Coding」
↓
页面展示 AI Coding 下的 3～5 张值得看卡片

用户点击「Agent 工程化」
↓
页面切换展示 Agent 工程化下的 3～5 张值得看卡片

用户点击「关注流」
↓
页面切换展示关注流精选内容

用户点击「偶遇输入」
↓
页面切换展示远端关联内容
```

## 4. 值得看卡片

每一张值得看卡片都不是简单摘要，而是为“形成观点种子”服务。

标准结构如下：

```
标题：
AI 编程工具会不会改变程序员的成长路径？

推荐理由：
这个话题近期讨论热度较高，并且和你过往关注的 Agent、编程、职业转型内容相关。

原始内容来源：
1. 知乎热榜问题：AI 编程工具普及后，初级程序员的成长路径会被改写吗？
   - 来源类型：热榜
   - 原文要点：讨论初级开发是否会被工具替代，以及新人训练机会是否减少
   - 对本卡片贡献：提供时效背景和职业焦虑入口
2. 知乎高赞回答：AI 降低的是机械编码门槛，不是问题抽象能力
   - 来源类型：知乎搜索 / 高赞回答
   - 原文要点：代码实现会被加速，但需求拆解、边界判断、复杂度分析仍依赖工程经验
   - 对本卡片贡献：支撑正方论据
3. 圈子评论：如果模型能读完整代码库，上下文还会是壁垒吗？
   - 来源类型：圈子评论 / 反方材料
   - 原文要点：长上下文模型和企业知识库可能削弱工程上下文壁垒
   - 对本卡片贡献：提供反方回应入口

内容摘要：
当前讨论主要集中在 AI 是否会压缩初级程序员岗位，以及程序员是否还需要传统编码训练。

主要争议：
1. AI 是否会替代初级开发？
2. 编码能力和工程能力是否会分化？
3. 程序员学习路径是否需要重构？

可写角度：
- AI 时代，程序员真正需要训练的是什么？
- 会写代码和能交付系统之间的差距会不会变大？
- 初级程序员的成长路径是否正在被改写？

操作：
[总结一下] [我认同] [我反对] [有疑问] [加入种子库] [基于它写一篇]
```

用户不需要马上写文章，只需要做轻量动作：

- 我认同；
- 我反对；
- 有疑问；
- 加入种子库；
- 基于它写一篇。

这些动作会帮助系统理解用户立场，并沉淀为观点种子。

### 原始来源展示要求

值得看卡片不能只展示 AI 摘要。每张卡片都必须保留原始内容来源，并且来源可能有多条。

来源需要分别展示：

- 来源类型：热榜、知乎搜索、全网搜索、关注流、圈子、评论；
- 原始标题；
- 作者 / 发布时间 / 热度等元信息；
- 原文要点；
- 该来源对当前卡片结论的贡献；
- 来源链接或可追溯 id。

原型里第一张卡片应作为完整展开样例，展示全部来源；后续卡片可以保持紧凑样式。

## 5. 偶遇输入

为了避免信息茧房，系统会保留少量“偶遇输入”。

偶遇输入不是随机内容，而是和用户已有知识结构存在远端关联的内容。

例如：

```
偶遇卡片：

今天的技术热点：
AI 编程工具进入企业协作场景。

远端关联：
这和金融风控中的“风险暴露”很像。单次代码生成效果并不是全部，真正重要的是极端场景下是否可控、可追责、可回滚。

可能产生的观点：
Agent 产品的护城河，不在单次任务完成率，而在持续上下文、风险控制和工作流闭环。

是否加入种子库？
[加入] [忽略] [再解释一下]
```

偶遇输入的目标是：

```
打破同温层
激发跨领域观点
帮助用户形成更有辨识度的表达
```

## 6. 我的种子库

种子库不是收藏夹。

收藏夹保存的是内容。

种子库保存的是：

```
我看完之后产生的想法
我认同什么
我反对什么
我觉得哪里有问题
这件事未来能不能写成文章
```

标准种子卡片如下：

```
观点种子：AI 编程工具的护城河可能不在代码生成

来源：
知乎热榜 / 搜索结果 / 用户手动记录

原始内容摘要：
近期讨论集中在 AI 编程工具是否会替代程序员，以及代码生成能力是否会快速同质化。

我的反应：
我认同代码生成会越来越普遍，但不认为这就是最终壁垒。

核心观点：
单纯代码生成工具的护城河很浅，真正的壁垒可能在上下文积累、工作流入口、团队协作和企业数据闭环。

可写方向：
1. AI 编程工具为什么会快速同质化？
2. 企业真正需要的是代码，还是可控的工程交付？
3. Agent 产品的护城河到底在哪里？

需要补充：
- AI 编程产品案例
- 企业研发流程
- 自己做复杂工程项目的经历

当前状态：
可发芽

操作：
[继续浇水] [今日发芽] [开始写作] [合并相似种子]
```

## 7. 今日发芽

“今日发芽”是看山小苗圃的核心差异化功能。

它会把：

```
历史观点种子
×
今日热点
×
知乎搜索
×
关注流
×
用户兴趣画像
```

进行碰撞，找出今天最值得写的机会。

### 重要调整

今日发芽不应该自动频繁计算。

因为它涉及：

- 热榜拉取；
- 搜索请求；
- 历史种子匹配；
- 语义检索；
- LLM 判断；
- 发芽指数计算；
- 写作机会生成。

这些都会带来 Token 消耗和接口调用成本。

所以正确交互应该是：

```
页面中出现一个小刘看山角色
↓
小刘看山问：
“今天要不要给你的种子浇浇水？我可以帮你看看有没有值得发芽的想法。”
↓
用户点击「今日发芽」
↓
系统开始计算
↓
计算完成后展示发芽机会卡片
```

### 发芽机会卡片

```
今日发芽机会：

被激活的种子：
单纯代码生成工具的护城河很浅。

触发热点：
某 AI 编程工具宣布被收购 / 发布企业版 / 引发讨论。

为什么值得写：
这个热点验证了你之前的判断：当代码生成能力逐渐同质化，产品竞争重点会转向工作流入口、上下文管理和企业协作数据。

建议标题：
AI 编程工具的护城河，可能不在“会写代码”

建议角度：
从“代码生成能力商品化”切入，讨论 AI 编程产品真正的长期壁垒。

建议补充：
你过去做复杂系统时可以强调，真正困难的不是写代码，而是需求边界、状态变化、系统约束和交付责任。

操作：
[开始写作] [补充资料] [换个角度] [暂时不写]
```

## 8. 写作苗圃

当用户决定写一篇文章时，会进入“写作苗圃”。

写作不是一步生成，而是一步一步推进。

进入写作苗圃时，系统必须根据观点种子的 `interestId` 匹配对应的兴趣分类画像，并将该 Memory 注入写作上下文。

用户需要能感知到这件事，例如页面中明确展示：

```
已匹配兴趣分类画像：AI Coding
Memory 已注入：
- 偏好视角：工程交付、研发工作流、程序员成长
- 证据偏好：案例优先
- 写作提醒：不要只讲工具趋势，需要补充具体工程场景和使用 AI 编程时的真实判断
```

这能让用户理解为什么系统会推荐某种文章类型、论证方式、反方问题和补充材料。

流程如下：

```
选择观点种子
↓
确认核心观点
↓
选择文章类型
↓
生成论证蓝图
↓
生成表达初稿
↓
圆桌审稿会
↓
定稿草案
↓
发布与反馈
```

## 9. 核心观点确认

系统先帮用户确认自己到底想表达什么。

示例：

```
我理解你想表达的是：

核心观点：
单纯代码生成能力很难成为 AI 编程工具的长期护城河。真正有价值的是上下文积累、工程工作流入口、团队协作数据和企业级可控交付。

文章基调：
不是否定 AI 编程工具的价值，而是讨论它的长期竞争壁垒在哪里。

你想让读者带走的判断：
AI 编程产品的竞争，将从“谁更会写代码”转向“谁更懂工程现场”。

请确认：
[确认] [调整观点] [更犀利一点] [更稳健一点] [补充个人经历]
```

## 10. 论证蓝图

系统不会直接上来写全文，而是先生成论证蓝图。

```
中心观点：
AI 编程工具的长期护城河，不在代码生成，而在工程上下文和工作流闭环。

背景：
代码生成能力正在快速普及，单次生成效果的差距会逐渐缩小。

分论点一：
代码生成能力容易被模型升级追平。

需要说明：
- 基础模型能力会持续提升
- 常规代码模式可被大量数据覆盖
- 单点生成能力难以形成长期壁垒

分论点二：
工程上下文更难迁移。

需要说明：
- 每个团队的代码规范、业务规则、历史债务不同
- 真正影响效率的是上下文理解和持续协作
- 工程现场的信息高度碎片化

分论点三：
企业用户真正关心的是可控交付。

需要说明：
- 生成代码只是开始
- 代码需要测试、审查、集成、回滚、追责
- 工具必须嵌入研发流程

反方观点：
如果模型足够强，是否也能理解上下文？

回应方向：
模型能力会提升，但企业上下文、团队流程和组织数据不是开放通用能力，需要长期积累和系统接入。

建议加入的个人经验：
你过去做 FileSync3 或 Java 后端项目时，可以补充“真正困难的不是写代码，而是数据链路、状态变化、边界条件和客户需求变化”。
```

## 11. 圆桌审稿会

逻辑压测和人味检查会以“圆桌会议”的形式出现。

参与讨论的 Agent 包括：

| Agent | 职责 |
| --- | --- |
| 逻辑审稿 Agent | 检查论证跳跃、未证明前提、概念偷换 |
| 人味编辑 Agent | 检查是否有真实经历、个人态度、具体细节 |
| 反方读者 Agent | 从反对者视角攻击文章漏洞 |
| 社区传播 Agent | 检查标题、开头、传播性和知乎语境 |

用户觉得可以后，可以点击：

```
[我觉得可以，进入定稿]
```

系统会整理出最终草稿。

---

## 12. 定稿草案

定稿草案不是自动发布内容。

系统会明确提醒：

```
这是最终草稿，但仍需要你修改后再发布。
```

这样可以避免用户丧失作者主体性，也可以降低纯 AI 内容感。

---

## 13. 历史文章监控反馈

文章发布后，系统可以监控历史表现，并把反馈重新沉淀为用户资产。

监控内容包括：

- 阅读完成率；
- 收藏率；
- 评论数；
- 评论争议度；
- 高赞评论；
- 反方观点；
- 读者补充材料；
- 是否适合写二次文章。

例如：

```
文章：AI 编程工具的护城河，可能不在“会写代码”

表现摘要：
阅读完成率较高，收藏率高于平均值。评论区主要围绕“上下文是否真的构成壁垒”展开。

评论反馈提取：
1. 支持观点：企业研发流程确实比单次生成更重要。
2. 反方观点：未来长上下文模型可能会弱化这个壁垒。
```
