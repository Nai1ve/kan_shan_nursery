# 看山小苗圃-核心流程校准方案

本文档是 24 小时冲刺阶段的执行口径。优先保证 Demo 的创新性、流程完整性和可讲清楚，不追求复杂算法一次到位。

本轮三条主流程：

```text
今日看什么：系统缓存高质量内容 → 按用户画像现场组卡 → 用户操作沉淀为种子
今日发芽：仅手动触发 → 用热点缓存和用户种子找写作机会 → 补充材料给用户确认
写作苗圃：种子 → 蓝图 → 大纲与引用 → 圆桌讨论 → 定稿 → 发布反馈
```

---

## 1. 今日看什么

### 1.1 目标

今日看什么不是直接把热榜展示给用户，而是把系统预先准备好的内容池，结合用户兴趣和画像，现场组装成少量高价值阅读卡片。

用户每次只看 3 条，避免信息流继续制造负担。

### 1.2 核心流程

```text
后台定时任务
→ 维护一份与用户无关的内容缓存
→ 用户请求今日看什么
→ content-service 根据用户兴趣、画像、已展示 id 现场选 3 张卡片
→ 前端展示卡片
→ 用户刷新时带回已展示 id
→ content-service 返回未展示过的下一批 3 张
→ 用户认同/反对/有疑问/加入种子/基于它写
→ seed-service 持久化卡片快照和用户反应，转化为观点种子
```

### 1.3 系统内容缓存

这份缓存与用户状态无关，由后台维护。

缓存内容来源：

- 知乎热榜：提供公共热点和时效信号。
- 知乎搜索：围绕系统预置主题和热门扩展词获取站内优质讨论。
- 全网搜索：补充外部资料、反方材料和偶遇输入。
- 关注流：OAuth 可用后再接入；不可用时不影响主流程。

缓存中保存的是标准候选内容：

```text
ContentCandidate
- id
- sourceType
- sourceId
- title
- summary
- fullContent
- author
- url
- publishedAt
- tags
- heat / ranking / authority / interaction signals
```

提供能力：

| 服务 | 提供什么 |
| --- | --- |
| `zhihu-adapter` | 拉取热榜、知乎搜索、全网搜索，并做 Redis 短缓存 |
| `content-service` | 定时构建系统级内容候选池 |
| `llm-service` | 对候选内容做轻量摘要、争议点、可写角度提取 |

### 1.4 用户现场组卡

用户请求时，`content-service` 读取：

```text
userId
categoryId
用户兴趣标签
全局 Memory
兴趣 Memory
已展示 cardIds
系统级内容候选池
```

输出：

```text
3 张 WorthReadingCard
本次 servedCardIds
refreshToken / cursor
```

组卡原则：

- 优先匹配用户兴趣和兴趣 Memory。
- 每批 3 张里尽量包含不同视角。
- 不重复返回用户已经看过的卡片。
- 单卡可以包含多个来源，但 Demo 阶段不强求每张都是 3 来源。
- 卡片必须能展示原始来源内容，不只给链接。

### 1.5 刷新机制

前端刷新时传入：

```text
categoryId
servedCardIds / excludeCardIds
limit = 3
```

后端返回：

```text
未展示过的 3 张卡片
新的 servedCardIds
```

刷新不应该重新清空用户状态，也不应该影响：

- 种子库。
- 写作 session。
- 用户画像。
- 历史反馈。

### 1.6 用户动作沉淀为种子

任何对卡片的强操作都应持久化用户关系：

| 用户动作 | 后端处理 |
| --- | --- |
| 认同 | 创建或更新种子，记录 `reaction=agree` |
| 反对 | 创建或更新种子，记录 `reaction=disagree`，优先生成反方材料 |
| 有疑问 | 创建或更新种子，创建 question thread |
| 加入种子库 | 保存卡片快照为种子 |
| 基于它写一篇 | 保存种子并进入写作苗圃 |

写入 seed-service 的信息：

```text
card snapshot
originalSources
用户 reaction
用户 note / question
categoryId / interestId
createdFromCardId
```

这样卡片不是一次性信息流，而是观点种子的来源。

### 1.7 最小接口

```text
GET /api/v1/content/cards?category_id={id}&limit=3&exclude_ids=a,b,c
POST /api/v1/content/categories/{category_id}/refresh
POST /api/v1/seeds/from-card
POST /api/v1/seeds/{seed_id}/questions
```

## 2. 今日发芽

### 2.1 目标

今日发芽只回答一个问题：

```text
我过去沉淀的哪些种子，今天因为热点、材料或用户近期阅读，值得继续写？
```

取消所有主动触发。只能由用户手动点击“开始今日发芽”。

### 2.2 手动触发流程

```text
用户点击开始今日发芽
→ sprout-service 读取用户种子库
→ 读取热点缓存和今日看什么卡片记录
→ 读取用户画像和兴趣 Memory
→ 找到可能被激活的种子
→ 为 Top 机会补充信息
→ 返回给用户
```

输入：

```text
userId
可选 interestId
用户种子列表
热点缓存
今日看什么已提供/已操作卡片
用户画像 Memory
```

输出：

```text
SproutOpportunity[]
- 关联 seed
- 触发它的热点/卡片
- 为什么今天值得写
- 建议标题
- 建议角度
- 建议补充材料
- 缺什么材料
```

### 2.3 各服务提供能力

| 服务 | 提供什么 |
| --- | --- |
| `seed-service` | 用户种子、材料、疑问、成熟度、状态 |
| `content-service` | 今日卡片、用户已看卡片、系统热点缓存 |
| `profile-service` | 用户画像、兴趣 Memory |
| `sprout-service` | 发芽机会匹配与机会状态 |
| `llm-service` | 生成“为什么值得写”、建议角度、补充材料 |

### 2.4 匹配原则

Demo 阶段不做复杂算法，按可解释规则即可：

- 种子没有发布、没有正在写作。
- 种子和热点/今日卡片主题相关。
- 种子材料不是完全空白。
- 今日输入能补充新证据、反方或时效背景。
- 用户 Memory 显示这个方向是用户关心的。

### 2.5 用户动作

| 动作 | 处理 |
| --- | --- |
| 补充资料 | 把发芽机会里的建议材料写回四格浇水板 |
| 换个角度 | 带用户输入重新生成标题和角度 |
| 暂时不写 | 仅记录本次机会 dismissed |
| 开始写作 | 创建 writing session，进入写作苗圃 |

### 2.6 最小接口

```text
POST /api/v1/sprout/start
GET /api/v1/sprout/opportunities
POST /api/v1/sprout/opportunities/{id}/supplement
POST /api/v1/sprout/opportunities/{id}/switch-angle
POST /api/v1/sprout/opportunities/{id}/dismiss
POST /api/v1/sprout/opportunities/{id}/start-writing
```

## 3. 写作苗圃

### 3.1 目标

写作苗圃要体现“逐步细化”，不是一键生成文章。

核心体验：

```text
观点来自用户
结构由系统辅助
材料可追溯
圆桌可讨论
最终由用户控制发布
```

### 3.2 主流程

```text
选择种子
→ 确认核心观点
→ 生成蓝图
→ 用户编辑或带要求重新生成蓝图
→ 确认蓝图
→ 生成大纲和每部分引用材料
→ 用户微调或重新生成大纲
→ 生成草稿
→ 圆桌审稿会
→ 作者发言，Agent 讨论和辩论
→ 用户控制进入定稿
→ 定稿草案
→ 模拟发布 / 发布后反馈
```

### 3.3 阶段一：选择种子与核心观点

输入：

```text
seed
seed materials
seed questions
用户 Memory
```

输出：

```text
coreClaim
写作目标
文章类型
Memory 注入摘要
```

要求：

- Memory 注入必须让用户看见。
- 用户可以改核心观点。
- 改动只影响当前 writing session，不直接覆盖种子。

### 3.4 阶段二：蓝图

蓝图是文章的“论证路线”，不是详细大纲。

输入：

```text
coreClaim
seed materials
用户写作目标
文章类型
Memory
用户补充要求
```

输出：

```text
blueprint
- 文章主线
- 关键论证步骤
- 必须回应的反方
- 需要补充的个人经验
- 风险提醒
```

用户能力：

- 直接编辑蓝图。
- 输入“重新生成要求”后重新生成。
- 确认后进入大纲。

### 3.5 阶段三：大纲与引用材料

确认蓝图后，系统生成更细的大纲。

输入：

```text
confirmed blueprint
seed materials
originalSources
用户补充材料
```

输出：

```text
outline sections
- section title
- section purpose
- key points
- referenced material ids
- missing material hints
```

用户能力：

- 编辑章节顺序和标题。
- 删除或新增章节。
- 为某一节重新生成。
- 补充某节需要引用的材料。

创新展示点：

```text
每一节都能看到“引用哪些材料”，避免 AI 空写。
```

### 3.6 阶段四：草稿

输入：

```text
confirmed outline
referenced materials
Memory
用户写作风格
```

输出：

```text
draft
AI 辅助声明建议
需要用户补真实经历的位置
```

要求：

- 不编造用户个人经历。
- 缺少个人经验时保留提示。
- 可支持按章节重新生成，不必一次重写全文。

### 3.7 阶段五：圆桌审稿会

圆桌不是一次性 review，而是可由用户控制的讨论。

参与者：

| 角色 | 作用 |
| --- | --- |
| 逻辑审稿 Agent | 查论证漏洞 |
| 人味编辑 Agent | 查 AI 味和个人经验不足 |
| 反方读者 Agent | 从反对者角度攻击文章 |
| 社区传播 Agent | 查标题、开头、知乎语境和 AI 声明 |
| 作者 | 用户本人，可以发言、解释、反驳或要求重审 |

流程：

```text
用户点击开始圆桌
→ Agent 各自给第一轮意见
→ 用户选择某条意见追问或回应
→ Agent 基于作者发言继续讨论
→ 用户决定采纳哪些建议
→ 用户点击进入定稿
```

输入：

```text
draft
outline
materials
Memory
roundtable messages
author message
```

输出：

```text
review messages
suggested changes
adopted suggestions
```

前端最小实现：

- 显示四个 Agent 的意见。
- 作者输入一段发言。
- 点击“继续讨论”追加一轮。
- 点击“采纳建议”写入 session。
- 点击“进入定稿”结束圆桌。

### 3.8 阶段六：定稿、发布、反馈

定稿输入：

```text
draft
adopted suggestions
用户最后要求
```

输出：

```text
finalDraft
publishChecklist
aiDisclosureSuggestion
```

发布：

- Demo 阶段可做“模拟发布”。
- 真实发布必须由用户确认。
- 不允许自动批量发布。

反馈：

```text
发布后文章
评论
点赞/收藏/阅读数据
→ feedback-service 摘要
→ 二次文章建议 / Memory 更新建议
```

## 4. 24 小时冲刺优先级

### P0 必须完成

1. 今日看什么按“用户请求 3 条 + exclude ids 刷新”跑通。
2. 卡片操作能转种子。
3. 今日发芽只保留手动触发。
4. 发芽机会能带补充材料进入种子或写作。
5. 写作苗圃支持蓝图编辑/重新生成。
6. 确认蓝图后生成大纲和引用材料。
7. 圆桌支持作者发言和继续讨论。
8. 定稿和模拟发布闭环。

### P1 可以简化

- 复杂评分公式。
- 多天趋势追踪。
- 自动定时发芽。
- 真实发布。
- 真实 OAuth 关注流。

### P2 截止后再做

- 高级个性化推荐权重。
- 完整多轮 Agent 辩论状态机。
- 反馈驱动长期自动优化。
- 更复杂的缓存失效策略。

---

## 5. Demo 讲述重点

1. 好内容不是刷出来的，是系统从缓存内容池中按用户画像筛出来的。
2. 卡片不是一次性消费，用户一表态就变成观点种子。
3. 今日发芽不是自动打扰，而是用户主动触发的“旧观点遇到新热点”。
4. 写作不是一键代写，而是蓝图、大纲、引用、圆桌、定稿的逐步细化。
5. Agent 不是替用户表达，而是在关键节点帮助用户想清楚、说清楚。
