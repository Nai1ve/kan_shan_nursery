# 看山小苗圃-前端入口与用户管理方案

本文档只描述前端入口、注册登录、用户状态、知乎账号关联提示、兴趣与写作风格采集、LLM 配置入口，以及这些页面需要调用的后端接口。OAuth 数据汇总和画像生成算法见：`docs/看山小苗圃-OAuth数据汇总与画像生成算法.md`。

---

## 1. 当前入口问题

当前前端入口仍然接近演示原型：

```text
打开页面
→ 关联知乎账号 / 首次画像 / 直接演示模式
→ 可绕过注册、登录、用户身份和 OAuth 绑定状态
```

这个入口不适合真实平台，因为：

- OAuth 授权结果无法绑定到某个本地用户。
- 无法区分新用户、老用户、未完成 onboarding 用户。
- 无法记录用户是否配置了自己的 LLM。
- 无法追踪临时画像、增强画像、待确认 Memory 更新。
- 直接进入工作台会让“用户画像驱动推荐和写作”的产品逻辑失效。

---

## 2. 目标入口流程

正式用户流程：

```text
新用户访问平台
→ 注册 / 登录
→ 提示关联知乎账号
→ 配置 LLM 使用方式
→ 选择兴趣领域和写作风格
→ 生成临时画像
→ 进入工作台
→ OAuth 数据和 LLM 增强画像在后台生成
→ 用户确认增强画像后写入长期 Memory
```

需要保留两个降级路径：

```text
知乎 OAuth 不可用
→ 用户跳过关联
→ 先用填写内容生成临时画像
→ 后续在个人画像页重新关联知乎
```

```text
用户未配置自己的 LLM
→ 使用平台免费接口
→ 受每日额度和频率限制
→ 超限后提示配置自己的 LLM 或次日继续
```

演示模式仍可保留，但应放在开发/路演入口，不作为普通用户主路径。

## 3. 用户状态机

建议前端新增这些状态枚举：

```ts
type AuthStatus =
  | "guest"
  | "registered"
  | "authenticated"
  | "logged_out";

type ZhihuBindingStatus =
  | "not_started"
  | "authorizing"
  | "bound"
  | "failed"
  | "skipped"
  | "unavailable"
  | "expired";

type LlmConfigStatus =
  | "not_configured"
  | "platform_free"
  | "user_configured"
  | "invalid"
  | "quota_limited";

type OnboardingStatus =
  | "not_started"
  | "preferences_pending"
  | "provisional_ready"
  | "completed";

type ProfileGenerationStatus =
  | "not_started"
  | "queued"
  | "collecting_oauth_data"
  | "analyzing"
  | "draft_ready"
  | "applied"
  | "failed";
```

组合后的平台状态：

| 状态 | 说明 | 页面 |
| --- | --- | --- |
| `guest` | 未注册/未登录 | 注册登录页 |
| `registered` | 注册成功但未设置 | 知乎关联提示页 |
| `zhihu_pending` | OAuth 进行中或失败 | 授权状态页，允许跳过 |
| `llm_pending` | 未选择 LLM 使用方式 | LLM 配置页 |
| `preferences_pending` | 未选择兴趣和写作风格 | 兴趣与风格页 |
| `provisional_ready` | 临时画像可用 | 可进入工作台 |
| `enrichment_running` | OAuth 增强画像后台生成中 | 工作台提示条 + 个人画像进度 |
| `enrichment_review` | 增强画像草稿待确认 | 画像确认页 |
| `ready` | 完整画像可用 | 工作台 |

## 5. 注册 / 登录页

### 5.1 注册

前端请求：

```ts
RegisterRequest {
  nickname: string;
  email?: string;
  username?: string;
  password: string;
}
```

接口：

```text
POST /api/v1/auth/register
```

成功响应：

```ts
AuthMeResponse {
  user: CurrentUser;
  setupState: UserSetupState;
  profile?: ProfileData;
  zhihuBinding?: ZhihuBindingViewModel;
  llmConfig?: LlmConfigViewModel;
}
```

注册成功后进入 `/onboarding/zhihu`。

### 5.2 登录

请求：

```ts
LoginRequest {
  identifier: string;
  password: string;
}
```

接口：

```text
POST /api/v1/auth/login
GET  /api/v1/auth/me
POST /api/v1/auth/logout
```

登录后根据 `setupState` 决定进入哪一步。

## 6. 知乎账号关联页

页面目标：说明为什么关联，以及 OAuth 不可用时如何继续。

展示文案应清楚说明会读取：

- 当前授权用户基本信息。
- 关注列表。
- 粉丝列表。
- 关注动态。
- 如后续开放用户回答/文章接口，则读取用户公开创作样本。

按钮：

```text
[关联知乎账号]
[暂时跳过，先建立本地画像]
[查看将使用哪些数据]
```

接口：

```text
GET    /api/v1/auth/zhihu/authorize
GET    /api/v1/auth/zhihu/callback
GET    /api/v1/auth/zhihu/binding
DELETE /api/v1/auth/zhihu/binding
```

前端不应直接跳 `zhihu-adapter`，而应走 `api-gateway`。原因：OAuth `state` 必须和本地用户会话绑定。

### 6.1 当前本地 OAuth 回调策略

当前开发和赛前调试阶段，OAuth 回调统一使用本地前端地址：

```text
http://127.0.0.1:3000/auth/zhihu/callback
```

原因：

- Cloudflare 临时域名会失效，授权完成后容易出现“无法访问此网站”。
- Cloudflare 页面和本地 `127.0.0.1` 页面不同源，弹窗 `postMessage`、`localStorage` 票据桥接和登录态同步都容易受跨域影响。
- 本地调试时前端 `/api/v1/*` 已通过 Next rewrite 转发到 `api-gateway`，因此回调页可以同源完成 `GET /api/v1/auth/zhihu/callback`。

当前要求：

```text
services/config.yaml
zhihu.oauth.redirect_uri = "http://127.0.0.1:3000/auth/zhihu/callback"
```

修改该配置后需要重启 `zhihu-adapter`，否则授权链接仍会使用旧回调地址。

### 6.2 上线前 OAuth 处理

上线前再做正式改造，不在当前开发阶段展开：

- 如果线上域名和前端同源可用，则将 `zhihu.oauth.redirect_uri` 改为线上同源地址，例如 `https://<domain>/auth/zhihu/callback`。
- 知乎开放平台后台登记的 redirect URI 必须和 `services/config.yaml` 完全一致。
- 前端授权成功页不得强制从线上域名跳回 `127.0.0.1`。
- 如果上线环境暂不支持稳定同源 OAuth，则隐藏“关联知乎账号”入口，只保留“暂时跳过，先建立本地画像”。
- 上线检查时再补充 `NEXT_PUBLIC_ENABLE_ZHIHU_OAUTH=false` 或等价开关，控制线上是否展示 OAuth 入口。

当前知乎鉴权未开放时：

```text
status = unavailable 或 failed
页面提示：知乎授权能力暂未开放，当前先使用你填写的信息生成临时画像。
用户可以继续进入下一步。
```

## 7. LLM 配置页

### 7.1 为什么需要这一页

项目不能假设平台无限提供 LLM 能力。实际策略应是：

```text
优先使用用户自己的 LLM 配置
→ 如果用户没有配置，则使用平台提供的免费 API 接口
→ 平台免费接口必须限流
→ 超限后提示用户配置自己的 LLM 或稍后再试
```

这里的“平台免费接口”可以是知乎直答 Agent 或平台自己接入的免费额度，但必须按用户、任务、日期做额度控制。

### 7.2 页面选项

建议提供三种模式：

```text
1. 使用平台免费额度
   - 零配置
   - 有每日次数和频率限制
   - 适合试用和演示

2. 配置我自己的 LLM
   - 支持 OpenAI-compatible base URL
   - 支持自定义 model
   - API Key 存服务端，不进浏览器 localStorage
   - 可以设置每日预算和最大调用频率

3. 稍后配置
   - 当前只使用非 LLM 或 mock 能力
   - 需要 LLM 的功能会提示补配置或使用免费额度
```

### 7.3 用户自有 LLM 配置字段

```ts
UserLlmProviderConfig {
  providerId: string;
  providerType: "openai_compatible" | "zhihu_direct" | "custom";
  displayName: string;
  baseUrl?: string;
  apiKeySecretRef: string;
  model: string;
  enabledTasks: LlmTaskType[];
  dailyBudgetCents?: number;
  dailyCallLimit?: number;
  rpmLimit?: number;
  status: "untested" | "valid" | "invalid" | "disabled";
  lastTestedAt?: string;
}
```

前端只展示脱敏信息：

```ts
LlmConfigViewModel {
  status: LlmConfigStatus;
  activeProvider: "platform_free" | "user_provider" | "none";
  displayName?: string;
  maskedKey?: string;
  model?: string;
  quota?: LlmQuotaViewModel;
  errorMessage?: string;
}
```

### 7.4 LLM 配置接口

建议 Gateway 接口：

```text
GET    /api/v1/llm/config/me
POST   /api/v1/llm/config/me/providers
PATCH  /api/v1/llm/config/me/providers/{providerId}
DELETE /api/v1/llm/config/me/providers/{providerId}
POST   /api/v1/llm/config/me/providers/{providerId}/test
POST   /api/v1/llm/config/me/select
GET    /api/v1/llm/quota/me
```

安全规则：

- API Key 只提交一次，服务端加密保存。
- 前端永远不读取明文 API Key。
- 导出日志时不包含 key。
- 用户删除配置后，后端删除 secret 或标记不可用。

### 7.5 平台免费接口限流展示

前端应展示：

```text
今日平台免费额度：
- 画像生成：1/3
- 内容摘要：12/50
- 疑问回答：4/20
- 写作草稿：0/5
```

当触发限流：

```text
平台免费额度已用完。你可以配置自己的 LLM，或明天继续使用免费额度。
```

## 8. 兴趣领域与写作风格页

该页在注册后必须执行，即使用户跳过 OAuth 或 LLM 配置也要执行。

### 8.1 兴趣领域

```ts
SelectedInterest {
  interestId: string;
  selected: boolean;
  selfRatedLevel: "beginner" | "intermediate" | "advanced";
  intent: "read" | "write" | "both";
}
```

候选领域：

- Agent 工程化
- AI Coding
- RAG / 检索
- 后端工程
- 程序员成长
- 金融风控
- 医学 AI
- 产品设计
- 内容创作

注意：

- `关注流精选` 和 `偶遇输入` 是输入来源，不是用户主动兴趣。
- 不应放入长期兴趣 Memory 主分类。

### 8.2 写作风格

当前 12 个问卷可以保留，但保存时要结构化。

```ts
WritingStyleSurvey {
  logicDepth: 1 | 2 | 3 | 4 | 5;
  stanceSharpness: 1 | 2 | 3 | 4 | 5;
  personalExperienceWillingness: 1 | 2 | 3 | 4 | 5;
  expressionSharpness: 1 | 2 | 3 | 4 | 5;
  preferredFormat: "zhihu_answer" | "balanced" | "long_article" | "column" | "draft";
  evidenceVsJudgment: "evidence_first" | "balanced" | "judgment_first";
  wantsCounterArguments: boolean;
  openingStyle: "direct" | "balanced" | "story";
  titleStyle: "restrained" | "balanced" | "spreadable";
  uncertaintyTolerance: 1 | 2 | 3 | 4 | 5;
  emotionalTemperature: "cold" | "balanced" | "emotional";
  aiAssistanceBoundary: "outline" | "paragraph" | "draft" | "polish" | "publish_ready";
}
```

接口：

```text
POST /api/v1/profile/onboarding
```

返回：

```ts
OnboardingResponse {
  profile: ProfileData;
  profileStatus: "provisional";
  enrichmentJob?: ProfileEnrichmentJob;
}
```

## 9. 画像生成进度页

提交兴趣和写作风格后，前端应立即进入画像生成页。

如果用户未配置 LLM，且选择平台免费额度：

- 检查平台免费额度。
- 如果额度可用，生成临时画像。
- 如果额度不可用，用规则模板生成最低限度画像，并提示“未使用 LLM 增强”。

展示进度：

```text
已完成：
✓ 保存兴趣领域
✓ 保存写作风格
✓ 生成临时画像

后台进行中：
- 读取知乎用户信息
- 汇总关注列表
- 汇总粉丝列表
- 分析关注动态
- 生成增强 Memory
```

按钮：

```text
[进入工作台]
[查看临时画像]
[稍后关联知乎]
[配置自己的 LLM]
```

## 10. 主应用内状态提示

工作台顶部或个人画像页应显示：

```text
画像状态：临时画像 / OAuth 增强中 / 增强画像待确认 / 已应用增强画像
知乎账号：未关联 / 已关联 / 授权失效 / 暂不可用
LLM：平台免费额度 / 已配置自己的 LLM / 未配置 / 已限流
```

个人画像页新增三个区块：

1. 授权与数据来源
2. LLM 配置与额度
3. Memory 草稿与更新请求

## 11. 前端最小实现切片

### Phase A：入口状态机

- 新增注册/登录页面。
- 新增 `AuthMeResponse` 获取和页面分流。
- 保留演示模式，但放到开发入口。

### Phase B：知乎关联提示

- 前端从直接跳 adapter 改成跳 gateway OAuth。
- 支持 unavailable/failed/skipped 状态。

### Phase C：LLM 配置页

- 支持平台免费额度选择。
- 支持用户自有 OpenAI-compatible 配置表单。
- 支持测试连接和额度展示。

### Phase D：兴趣与写作风格

- 结构化保存兴趣和问卷。
- 生成 provisional profile。

### Phase E：画像状态反馈

- 工作台和个人画像页显示画像状态、OAuth 状态、LLM 状态。
- 增强画像草稿待确认时显示提示。

---

## 12. 验收清单

1. 新用户不能直接进入正式工作台。
2. 注册成功后进入知乎关联提示页。
3. OAuth 不可用时可以跳过。
4. 用户可以选择平台免费 LLM 或配置自己的 LLM。
5. 用户自有 API Key 不进入 localStorage。
6. 平台免费额度能展示剩余次数。
7. 平台免费额度超限时有明确提示。
8. 兴趣领域和写作风格结构化保存。
9. 完成 onboarding 后生成临时画像并可进入工作台。
10. 工作台能展示画像状态、知乎状态、LLM 状态。
11. 个人画像页能重新关联知乎、配置 LLM、查看增强画像草稿。
