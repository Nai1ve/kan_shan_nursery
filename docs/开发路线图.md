# 开发路线图

本文档描述看山小苗圃从原型到可演示产品的阶段路线。**v0.1-v0.4 已完成**，剩余开发合并为 5 个版本（v0.5-v0.9），每个版本对应一条可演示的能力切片，避免颗粒度过细带来联调成本。

## v0.5 LLM 基座与多视角 Agent 框架

合并自旧 v0.5（核心 Agent 基座）与旧 v0.7（统一网关与本地联调）。聚焦把 `llm-service` 从单一 mock+zhihu_direct 路由扩展为可注册多 provider、可按 task 路由、可在单 task 内并发多 persona 的 harness 框架，**不改下游业务服务**。

目标：让"哪个 Agent 用哪个 LLM 视角"成为配置而非代码。

### 交付（v0.5）

- `services/llm-service/app/providers/` 抽象基类 + mock + zhihu_direct 两个真实 provider。
- `services/llm-service/app/registry.py` provider 注册表 + per-task 路由 + persona 配置。
- `services/llm-service/config/providers.yaml` 配置入口（mock + zhihu_direct，OpenAI 兼容预留 stub）。
- `services/llm-service/prompts/v1/personas/` 圆桌审稿四个 reviewer 的 persona prompt。
- 多 persona 模式：roundtable-review 路由到 `multi_persona`，并发执行后聚合。
- LLM trace：每次调用结构化日志（task / provider / persona / cache hit / fallback / latency）。
- 单测：路由表、fallback、persona 聚合、trace 字段完整性。

### 不交付（v0.5）

- 不接入 OpenAI 兼容外部 provider（保留 stub，等密钥再启用）。
- 不改 content / sprout / writing / feedback 调 llm-service 的逻辑。这部分留到 v0.7-v0.8 各自版本。

### 验收（v0.5）

- 每个 task 在 mock 模式下输出仍满足 `validators.TASK_REQUIRED_KEYS`。
- 通过 `LLM_PROVIDER_MODE=zhihu` 切换主 provider，下游不感知；失败时 fallback 到 mock 并打 trace。
- roundtable-review 在 mock 路由下返回 4 个 reviewer 结果，每个 reviewer 标注其 provider + persona。
- 同输入二次调用命中 cache（hit=true）。
- `output/llm-trace/*.jsonl` 至少包含一条记录，字段齐全。

## v0.6 知乎能力接入

v0.6 把 zhihu-adapter 从 skeleton 升级到可真实联调的状态，并捎带完成一次跨服务基础设施升级（共享配置 + 结构化日志），为后续版本评测打下基础。

### 交付（v0.6）

- **三类 Client 拆开**：`CommunityClient` / `OAuthClient` / `DataPlatformClient` 独立文件，各自的鉴权、base URL、错误归一化不再在 service.py 里用 if 分支区分。
- **完整 OAuth 流程**：`/zhihu/oauth/authorize`（支持 `?redirect=true` 直接 302）+ `/zhihu/oauth/callback`（exchange code → access_token，渲染人读 HTML 指示用户把 token 写入 config.yaml）；前端登录页"关联知乎账号"按钮改为触发真实跳转。
- **Redis 缓存 + quota 计数**：命中不扣额度，miss 成功才 +1；quota 限额从 config.yaml 读取（默认 hot 100 / search 1000 / global 1000 / direct-answer 100）。
- **字段归一化**：热榜 URL hash → sourceId；搜索 RankingScore → relevanceScore；全网搜索 `<em>` raw html + 去标签 summary；故事内容 usageNotice；关注流 actor/action/target。
- **错误码归一化**：`ZHIHU_AUTH_FAILED` / `ZHIHU_RATE_LIMITED` / `ZHIHU_INVALID_REQUEST` / `ZHIHU_RING_NOT_WRITABLE` / `ZHIHU_UPSTREAM_ERROR` / `ZHIHU_UNAVAILABLE`，HTTP 401/429/400/502 分别映射。
- **圈子白名单**：发布/评论/点赞前校验 `ring_id in writable_ring_ids`，否则抛 `ZHIHU_RING_NOT_WRITABLE`。
- **共享配置模块** `packages/shared-python/kanshan_shared`：`load_config()` 从 `services/config.yaml` + 环境变量读取凭证，dataclass 化；兼容旧 flat schema（`zhihu.ZHIHU_APP_KEY`）。
- **共享结构化日志**：jsonl 落 `output/logs/<service>-YYYY-MM-DD.jsonl` + 控制台人读；9 个服务 main.py 统一在启动时 `configure_logging(service_name, cfg.logging)`；关键事件（zhihu_call_started / cache_hit / quota_exceeded / call_failed / oauth_exchanged）在 adapter 内埋点，gateway proxy 事件埋点。

### 验收（v0.6）

- `provider_mode=live` 时三类 Client 按对应鉴权头发出请求；`provider_mode=mock` 时继续返回 fixture。
- 缓存命中 `quota.usedToday` 不增；miss 成功才 +1。
- 超限返回 `ZHIHU_RATE_LIMITED` (429)，错误 detail 包含 endpoint 和 limit。
- 全网搜索 `summary` 不含 `<em>`，`rawExcerptHtml` 保留 `<em>`。
- 故事详情 `usageNotice` 指向盐言故事署名与活动限制。
- `POST /zhihu/publish/mock-or-live` 传入不在白名单的 ring_id 时 400 `ZHIHU_RING_NOT_WRITABLE`。
- OAuth authorize → callback 能跑通；前端点击"关联知乎账号"跳到知乎授权页，授权回来 adapter 渲染含 access_token 的提示 HTML。
- `LLM_PROVIDER_MODE=zhihu` 时 llm-service 通过 zhihu-adapter 调直答能拿到 OpenAI 兼容格式的 `content` / `reasoning_content`。
- 任意服务启动后在 `output/logs/<service>-<date>.jsonl` 留下至少一条请求日志；控制台输出人读行。
- 仓库全量单元测试：9 服务 + shared-python 合计 80 ok / 0 fail。

## v0.7 输入推流与种子浇水强化

合并自旧 v0.8（推流算法）与旧 v0.9（种子浇水强化）。两者都围绕"看 → 种子"链路：content-service 把推流升级到规则版多来源聚合 + 卡片评分 + 多样性去重；seed-service 把疑问线程多轮、Agent 补证据/找反方完整接入 llm-service。前端开始从 `/api/mock/*` 切到 `/api/v1/*`。

### 交付（v0.7）

- content-service Query Plan 生成（按 categoryKind 分支：interest / following / serendipity）。
- 候选召回 + 多来源卡片聚合（1-3 个来源/卡）。
- cardScore：interestFit / sourceQuality / freshness / controversy / writingPotential / memoryFit / diversityGain。
- 多样性去重（P0 规则版，去重 sourceId、相似标题合并、按 sourceType 配额）。
- 偶遇输入 serendipityScore（远端关联 + novelty + explainability - forcedAnalogyRisk）。
- seed-service 调 llm-service answer-seed-question 真实 task（替换 mock 文本）。
- seed-service 调 llm-service supplement-material 实现"Agent 补证据"和"Agent 找反方"。
- 前端把 today / seeds / sprout / feedback / profile 五条接口切到 gateway。

### 验收（v0.7）

- 每个兴趣分类至少 2-3 张卡片，重点卡片有 2-3 个来源。
- 刷新当前分类不影响种子库、写作 session、用户画像、历史反馈。
- 偶遇输入能解释远端关联，不是随机内容。
- 同 cardId 不重复创建种子；reaction 幂等更新。
- 疑问线程：`answered → resolved` 后 thread 闭环；`needs_material` 保留待解决。
- 个人经验材料只能由用户填，不能由 Agent 编造。
- 前端可只配 gateway 地址跑 Demo，下游错误统一格式。

## v0.8 发芽算法与写作苗圃

合并自旧 v0.10（发芽算法）与旧 v0.11（写作苗圃与圆桌审稿）。这两个能力都依赖成熟的 seed 与 llm-service 多视角框架，放到一个版本一起做。

### 交付（v0.8）

- sprout-service activationScore（seedMaturity / hotRelevance / timeliness / newInfoGain / writingPotential - duplicatePenalty - evidencePenalty）。
- 候选种子筛选 + 优先级排序。
- 新信息召回（通过 zhihu-adapter 缓存）+ seed-topic 关键词匹配。
- 发芽机会必须解释：哪颗种子、什么触发、为什么现在值得写。
- 阈值过滤（< 50 不展示，50-64 建议浇水，65-79 可写但补料，>= 80 强烈建议）。
- writing-service 调 llm-service argument-blueprint / draft / roundtable-review 真实 task。
- 圆桌审稿真正用 v0.5 多 persona 模式执行 4 个 reviewer，输出按 severity 排序聚合。
- AI 辅助声明建议进入定稿。

### 验收（v0.8）

- 发芽必须用户主动触发，不自动跑。
- 同 interest 同日命中缓存（v0.4 已实现，本版本仍保留）。
- activationScore 低于阈值不展示；展示的机会都附带 newInfoGain 解释。
- writingSession 创建时注入 interest Memory，前端可见且可改 memoryOverride。
- memoryOverride 修改不污染 profile Memory。
- 圆桌审稿能给出至少一条 high severity 问题（不是全 low）。
- 定稿前必须用户确认；模拟发布生成 feedbackArticle。

## v0.9 反馈回流、评测与 Demo 演示

合并自旧 v0.12（反馈回流）、v0.13（评测与成本）、v0.14（Demo 演示）。三者一起收尾。

### 交付（v0.9）

- feedback-service 调 llm-service feedback-summary 真实 task。
- 评论摘要 → 二次种子 payload + memory_update_request（不自动写入 Memory）。
- Memory apply / reject 闭环（profile-service 已支持，前端串起来）。
- 评测指标采集：内容输入质量、种子质量、发芽质量、写作质量、成本（缓存命中 / quota / LLM 调用 / fallback 率）。
- LLM trace 离线分析脚本（按 task / provider 聚合）。
- 一键启动脚本（v0.4 已有 `scripts/run_all_services.sh`，本版本补 docker-compose 9 服务模板）。
- 演示数据预置 + 关键 API fallback + Demo 脚本。
- AI 辅助声明和知乎规范说明。

### 验收（v0.9）

- 反馈不直接写 Memory，只产 `memory_update_request`。
- 用户 apply 后 Memory 才更新；reject 只改请求状态。
- `output/llm-trace/*.jsonl` 能给出每条链路的 LLM 调用次数、缓存命中率、fallback 率。
- Demo 推荐链路（11 步）能在断网或知乎 API 失败时降级演示。
- 现场不依赖长时间生成（昂贵任务必须有 mock fallback）。
