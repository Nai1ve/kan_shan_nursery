## 2. Harness Engineering 总则

每个版本的实施都遵循以下约定，目的是让契约和实现分离、让"哪个服务用哪个能力"成为配置而非硬编码。

1. **接口契约稳定**：业务服务对 llm-service / zhihu-adapter 只感知 task type 和 endpoint，不感知 provider、prompt、模型 ID。
2. **配置驱动**：provider 注册、task 路由、persona 编排都放配置文件（`config/*.yaml` 或环境变量），代码只做注册表查询。
3. **降级显式**：失败必须显式 `fallback=true` 或 `error.code=...`，不能静默返回看似正常的数据。
4. **trace 必落盘**：每次 LLM 调用 / 知乎调用产 jsonl 一行（task / provider / persona / cache_hit / fallback / latency / token_estimate），方便后期评测。
5. **测试与边界绑定**：每个新版本的"不负责"清单必须有用例反向验证（例如 feedback-service 不能直接写 Memory，要有用例确认调用后 GET memory 不变）。
6. **文档同步**：本文件对应版本章节 + 服务 README + 路线图状态必须和代码同 commit。

## 3. 多 LLM 视角 Agent 框架（v0.5 落地）

### 3.1 设计目标

让"哪个 Agent 用哪个 LLM 视角"成为 llm-service 内部的路由问题，业务服务永远只调 `POST /llm/tasks/{task_type}`。

为什么要做：

- 不同任务对模型偏好不同（摘要 vs 反方 vs 草稿）。
- 圆桌审稿天然多视角，单 provider 单 prompt 容易同质化。
- 真实接入外部 provider 时，希望 0 改动业务代码。

### 3.2 架构

```text
business service
        │  POST /llm/tasks/{task_type}
        ▼
    llm-service
    ├─ validators            (request / output schema)
    ├─ cache                 (memory or redis)
    ├─ registry              (load config/providers.yaml)
    ├─ router                (task_type → provider_id 或 multi_persona 列表)
    ├─ providers/
    │    ├─ base.py          (Provider Protocol)
    │    ├─ mock.py          (现 mock_provider 升级)
    │    ├─ zhihu_direct.py  (现 zhihu_provider 升级)
    │    └─ openai_compat.py (stub，等密钥)
    ├─ prompts/v1/
    │    ├─ <task>.md        (主 prompt)
    │    └─ personas/<name>.md (persona prompt)
    └─ trace                 (output/llm-trace/*.jsonl)
```

### 3.3 Provider 抽象

```python
class Provider(Protocol):
    name: str  # "mock" / "zhihu_direct" / "openai_compat"

    def run(
        self,
        task_type: str,
        input_data: dict,
        prompt: str,
        persona: str | None = None,
    ) -> dict:
        """Returns parsed structured output. Raises ProviderError on failure."""
```

所有 provider 共享同一接口；router 拿 provider 后注入对应 prompt（含 persona 段）。

### 3.4 Task × Provider 映射表（v0.5 起作用）

| task_type | 默认 provider | fallback | 模式 | 备注 |
| --- | --- | --- | --- | --- |
| summarize-content | zhihu_direct | mock | single | 高频，cache 优先 |
| extract-controversies | zhihu_direct | mock | single | 高频 |
| generate-writing-angles | zhihu_direct | mock | single | |
| answer-seed-question | zhihu_direct | mock | single | v0.7 由 seed-service 调真实 |
| supplement-material | zhihu_direct | mock | single | v0.7 由 seed-service 调真实 |
| sprout-opportunities | zhihu_direct | mock | single | v0.8 由 sprout-service 调真实 |
| argument-blueprint | zhihu_direct | mock | single | v0.8 由 writing-service 调 |

> v0.5 仅 mock + zhihu_direct 真实可用。openai_compat 提供 stub 实现，启用需要 `OPENAI_COMPAT_BASE_URL` / `OPENAI_COMPAT_API_KEY`，否则 registry 不注册。

### 3.5 多 Persona（roundtable-review）

```yaml
roundtable-review:
  mode: multi_persona
  personas:
    - id: logic_reviewer
      provider: zhihu_direct
      persona_prompt: prompts/v1/personas/logic_reviewer.md
    - id: human_editor
      provider: zhihu_direct
      persona_prompt: prompts/v1/personas/human_editor.md
    - id: opponent_reader
      provider: zhihu_direct
      persona_prompt: prompts/v1/personas/opponent_reader.md
    - id: community_editor
      provider: zhihu_direct
      persona_prompt: prompts/v1/personas/community_editor.md
```

router 在 multi_persona 模式：

1. 对每个 persona 单独调 provider；并发由 `concurrent.futures.ThreadPoolExecutor` 串起。
2. 每条结果用 `roundtable-review` schema 校验。
3. 聚合到 `output.reviews[]`，每条带 `persona`、`provider` 标注。
4. 顶部按 `severity ∈ {high, medium, low}` 排序。
5. 任一 persona 失败：mock 该 persona 并标 `fallback=true`，不让全任务失败。

### 3.6 trace 输出

每次 `LlmService.run_task` 完成（无论命中缓存与否）写一行：

```json
{"ts":"2026-05-11T...","taskType":"roundtable-review","provider":"multi_persona",
 "persona":null,"cacheHit":false,"fallback":false,"latencyMs":182,
 "promptVersion":"v1","schemaVersion":"v1","inputHash":"abc...","mode":"multi_persona",
 "subCalls":[
   {"persona":"logic_reviewer","provider":"zhihu_direct","fallback":false,"latencyMs":120},
   {"persona":"human_editor","provider":"zhihu_direct","fallback":true,"latencyMs":20}
 ]}
```

文件按日期分片：`output/llm-trace/2026-05-11.jsonl`，目录由服务自动创建。

### 3.7 缓存键

保持现有：`llm:{task_type}:{input_hash}:{prompt_version}:{schema_version}`。
multi_persona 任务整体作为一条缓存（命中后跳过所有 persona），子 persona 不单独缓存——这避免缓存爆炸，也保证 reviewers 一致性。

### 3.8 失败语义

- 主 provider 抛 `ProviderError`：路由按 fallback 配置降级，response 设 `fallback=true`。
- multi_persona 内部任一 persona 抛错：仅该 persona 走 mock 替补；整体不算 fallback；trace 单独记。
- mock 也失败（schema 校验都过不了）：返回 HTTP 502，error.code=`LLM_OUTPUT_INVALID`。

## 11. 风险与对策

| 风险 | 对策 |
| --- | --- |
| 多 provider 让本地启动复杂 | 默认 `LLM_PROVIDER_MODE=mock`；OpenAI 兼容 env 未配则 registry 不注册 |
| 多 persona 让 roundtable 慢 | persona 并发执行；任一失败 mock 替补不阻塞整体；缓存命中跳过所有 persona |
| 业务服务被 llm-service 改动牵连 | v0.5 不动业务服务；契约（task_type + envelope）保持不变 |
| 文档和代码漂移 | 每次 commit 必须同步本文件、路线图、服务 README，CI 后续可加 lint |
| trace 文件膨胀 | 按日期切片；v0.9 加清理脚本（保留 7 天） |
