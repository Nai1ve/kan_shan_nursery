# 看山小苗圃-高风险核心任务细化说明

更新时间：2026-05-13

本文给其他 AI Coding 模型执行用，覆盖四个当前最容易跑偏的任务：

```text
1. OAuth 画像生成
2. 今日发芽算法
3. 写作苗圃状态机
4. LLM provider 路由修复
```

实现时优先保证 Demo 链路能跑通，不追求复杂工程治理。所有长期 Memory 更新必须让用户确认，不允许静默覆盖。

---

## 1. OAuth 画像生成

### 1.1 目标

用户完成注册和知乎 OAuth 关联后，系统要用两类信息生成画像：

```text
用户主动填写的信息
+ OAuth 授权获得的知乎信息
→ 临时画像
→ LLM 画像草稿
→ 用户确认
→ 写入长期 Memory
```

在 OAuth 数据抓取或 LLM 生成未完成前，系统必须能用用户主动填写的信息生成一份临时画像，让用户继续使用今日看什么和写作功能。

### 1.2 涉及服务

| 服务 | 职责 |
| --- | --- |
| `profile-service` | 画像任务入口、OAuth 数据汇总、Memory 草稿保存、用户确认写入 |
| `zhihu-adapter` | OAuth 用户信息、关注列表、粉丝列表、关注动态 |
| `llm-service` | `profile-memory-synthesis` 任务 |
| `api-gateway` | 代理 profile 画像任务和 Memory 更新请求 |
| `frontend` | 展示画像生成状态、临时画像、待确认 Memory 建议 |

### 1.3 最小数据流

```text
用户注册 / 登录
→ 填写兴趣、写作偏好、基础资料
→ profile-service 保存临时画像
→ 用户关联知乎账号
→ profile-service 拉取 access_token
→ zhihu-adapter 获取 /user、/user/followed、/user/followers、/user/moments
→ profile-service 清洗为 ProfileSignalBundle
→ llm-service profile-memory-synthesis
→ profile-service 生成 memory_update_requests
→ 前端展示“待确认画像增强”
→ 用户确认 / 编辑 / 拒绝
→ 确认后写入 globalMemory 和 interestMemories
```

### 1.4 建议接口

如果已有接口能复用，优先复用；没有则新增这些最小接口：

```text
POST /profiles/me/enrichment-jobs
GET  /profiles/me/enrichment-jobs/latest
GET  /memory/update-requests
POST /memory/update-requests/{request_id}/apply
POST /memory/update-requests/{request_id}/reject
```

`POST /profiles/me/enrichment-jobs` 请求：

```json
{
  "trigger": "oauth_bound",
  "includeSources": ["zhihu_user", "followed", "followers", "moments"],
  "manualSamples": []
}
```

响应：

```json
{
  "jobId": "profile-job-xxx",
  "status": "queued",
  "temporaryProfileReady": true
}
```

`GET /profiles/me/enrichment-jobs/latest` 响应：

```json
{
  "jobId": "profile-job-xxx",
  "status": "running|completed|failed|fallback",
  "temporaryProfile": {},
  "signalCounts": {
    "zhihu_user": 1,
    "followed": 20,
    "followers": 20,
    "moments": 50
  },
  "memoryUpdateRequestIds": ["memreq-xxx"],
  "errorMessage": ""
}
```

### 1.5 ProfileSignalBundle

进入 LLM 前，profile-service 必须先整理成稳定结构，不要把官方 raw 直接交给 LLM。

```ts
ProfileSignalBundle {
  userId: string;
  generatedAt: string;
  onboarding: {
    selectedInterestIds: string[];
    writingStyleAnswers: Record<string, unknown>;
    selfDescription?: string;
  };
  signals: ProfileSignalSourceItem[];
}

ProfileSignalSourceItem {
  evidenceId: string;
  sourceType:
    | "onboarding"
    | "zhihu_user"
    | "followed"
    | "followers"
    | "moments"
    | "manual_content";
  sourceId: string;
  title?: string;
  excerpt?: string;
  authorName?: string;
  headline?: string;
  actionText?: string;
  publishedAt?: string;
  confidenceHint: number;
}
```

清洗要求：

- 去掉 email、phone、access_token、refresh_token 等敏感字段。
- `excerpt` 控制在 160-240 字。
- 每条信号必须有 `evidenceId`。
- 关注列表和粉丝列表不能直接推断“用户相信什么”，只能作为“信息源偏好 / 潜在读者”弱信号。
- 用户主动填写的信息优先级最高。

### 1.6 LLM 输入输出

调用：

```text
POST /llm/tasks/profile-memory-synthesis
```

输入：

```json
{
  "taskType": "profile-memory-synthesis",
  "input": {
    "bundle": {},
    "existingMemory": {},
    "interestCatalog": []
  }
}
```

输出建议：

```ts
ProfileMemorySynthesisOutput {
  globalProfilePatch: {
    summary: string;
    writingStyle: string;
    riskNotes: string[];
  };
  interestMemoryPatches: Array<{
    interestId: string;
    confidence: number;
    interestName: string;
    preferredPerspective: string[];
    evidencePreference: string;
    writingReminder: string;
    evidenceRefs: string[];
  }>;
  rejectedInferences: string[];
}
```

### 1.7 写入规则

LLM 输出不能直接写长期 Memory。profile-service 应生成 `memory_update_requests`：

```ts
MemoryUpdateRequest {
  id: string;
  userId: string;
  scope: "global" | "interest";
  interestId?: string;
  proposedPatch: object;
  reason: string;
  evidenceRefs: string[];
  status: "pending" | "applied" | "rejected";
  createdAt: string;
}
```

用户操作：

| 动作 | 处理 |
| --- | --- |
| 确认 | merge 到长期 Memory，记录版本 |
| 编辑后确认 | merge 用户编辑后的 patch |
| 拒绝 | request 标记 `rejected`，不写 Memory |

### 1.8 最小验收

- 新用户不关联知乎，也能用主动填写信息生成临时画像。
- 关联知乎后能启动 enrichment job。
- OAuth 抓取失败时状态为 `fallback`，不影响使用。
- LLM 成功后生成待确认 Memory 更新，不直接覆盖。
- 用户确认后，`GET /profiles/me/interests` 能看到更新后的兴趣 Memory。

## 2. 今日发芽算法

### 2.1 目标

今日发芽只在用户手动点击时运行，回答：

```text
我过去沉淀的哪颗种子，今天因为热点、今日卡片或新材料，值得写？
```

不做后台自动触发，避免成本和用户感知问题。

### 2.2 涉及服务

| 服务 | 提供能力 |
| --- | --- |
| `sprout-service` | 发芽 run、机会计算、机会状态 |
| `seed-service` | 用户种子、材料、疑问、成熟度 |
| `content-service` | 今日看什么已提供卡片、热点缓存、系统内容池 |
| `profile-service` | globalMemory、interestMemory |
| `llm-service` | `sprout-opportunities`、换角度文案 |

### 2.3 输入数据

`POST /sprout/start` 请求：

```json
{
  "userId": "user-xxx",
  "interestId": "optional",
  "limit": 4
}
```

sprout-service 内部需要收集：

```ts
SproutInput {
  userId: string;
  interestId?: string;
  seeds: IdeaSeed[];
  hotCards: WorthReadingCard[];
  todayCards: WorthReadingCard[];
  userRecentActions: Array<{
    cardId: string;
    action: "agree" | "disagree" | "question" | "write";
    createdAt: string;
  }>;
  memory: {
    globalMemory: object;
    interestMemories: object[];
  };
}
```

如果跨服务调用来不及实现，P0 可由 api-gateway 或前端把 `seeds/todayCards/memory` 放进 payload，但最终口径应是服务端汇总。

### 2.4 规则筛选

先用规则筛选 Top 候选，再调用 LLM，不要把所有种子全部丢给 LLM。

候选种子必须满足：

```text
status != published
status != writing
maturityScore >= 45
wateringMaterials 至少 1 条
```

排除：

```text
7 天内 dismissed 过的同 seed + 同 trigger
已经有 active writing session 的 seed
没有任何观点内容的空种子
```

### 2.5 激活评分

P0 使用可解释公式即可：

```text
activationScore =
  seedMaturity * 0.25
+ topicRelatedness * 0.25
+ freshness * 0.20
+ newInfoGain * 0.15
+ memoryFit * 0.10
+ controversyPotential * 0.05
- penalties
```

字段计算建议：

| 因子 | 简化计算 |
| --- | --- |
| `seedMaturity` | `maturityScore / 100` |
| `topicRelatedness` | seed 标题/核心观点/材料与热点卡片标题的关键词重叠；有 embedding 时再换 |
| `freshness` | 热榜、今日卡片优先；超过 7 天降低 |
| `newInfoGain` | 今日卡片能补充 seed 缺少的 evidence/counterargument/open_question |
| `memoryFit` | seed.interestId 命中用户兴趣 Memory，或 Memory 里有对应偏好 |
| `controversyPotential` | seed.counterArguments 数量 + 今日卡片争议点数量 |

惩罚：

```text
证据材料为空: -0.25
只有摘要没有来源: -0.15
最近 dismissed: -0.30
同类主题已发布过: -0.20
```

### 2.6 LLM 机会生成

只把 Top 4-6 个候选传给 LLM。

调用：

```text
POST /llm/tasks/sprout-opportunities
```

输入：

```json
{
  "taskType": "sprout-opportunities",
  "input": {
    "candidates": [
      {
        "seed": {},
        "triggerCards": [],
        "scoreSignals": {}
      }
    ],
    "memory": {},
    "limit": 4
  }
}
```

输出：

```ts
SproutOpportunity {
  id: string;
  runId: string;
  userId: string;
  seedId: string;
  interestId: string;
  triggerType: "hot" | "today_card" | "following" | "search";
  triggerCardIds: string[];
  triggerTopic: string;
  activatedSeed: string;
  whyWorthWriting: string;
  suggestedTitle: string;
  suggestedAngle: string;
  suggestedMaterials: string;
  missingMaterials: string[];
  score: number;
  status: "active" | "supplemented" | "angle_changed" | "dismissed" | "writing";
  createdAt: string;
}
```

### 2.7 用户动作

| 接口 | 行为 |
| --- | --- |
| `POST /sprout/opportunities/{id}/supplement` | 返回 `seedMaterial`，由前端或 gateway 写入 seed-service |
| `POST /sprout/opportunities/{id}/switch-angle` | 根据用户要求重新生成 `suggestedTitle/suggestedAngle` |
| `POST /sprout/opportunities/{id}/dismiss` | 标记 dismissed，并记录 seedId + triggerCardIds |
| `POST /sprout/opportunities/{id}/start-writing` | 返回 writing handoff，前端再创建 writing session |

### 2.8 最小验收

- 前端点击“开始今日发芽”必须请求后端，不再用 local `setTimeout` 造机会。
- 机会必须能指出“哪颗种子 + 被什么热点/卡片激活”。
- 点击补充资料能把材料写入对应 seed。
- 换角度后标题和角度变化，并保留原 opportunity。
- 暂时不写后刷新不应马上再次出现同一机会。

## 3. 写作苗圃状态机

### 3.1 目标

写作苗圃不是“一键生成文章”，而是逐步细化：

```text
观点确认
→ 蓝图
→ 大纲与引用
→ 草稿
→ 圆桌讨论
→ 定稿
→ 发布/反馈
```

用户必须能修改和重新生成关键阶段内容。

### 3.2 WritingSession

建议后端 session 结构：

```ts
WritingSession {
  sessionId: string;
  userId: string;
  seedId: string;
  interestId: string;
  articleType: string;
  coreClaim: string;
  memoryOverride: object;
  status:
    | "claim_confirming"
    | "blueprint_ready"
    | "blueprint_confirmed"
    | "outline_ready"
    | "outline_confirmed"
    | "draft_ready"
    | "reviewing"
    | "finalized"
    | "published";
  blueprint?: WritingBlueprint;
  outline?: WritingOutline;
  draft?: WritingDraft;
  roundtable?: RoundtableState;
  finalDraft?: FinalDraft;
  createdAt: string;
  updatedAt: string;
}
```

### 3.3 状态转移

```text
create_session
  -> claim_confirming

confirm_claim
  claim_confirming -> claim_confirming

generate_blueprint
  claim_confirming -> blueprint_ready

patch_blueprint
  blueprint_ready -> blueprint_ready

regenerate_blueprint
  blueprint_ready -> blueprint_ready

confirm_blueprint
  blueprint_ready -> blueprint_confirmed

generate_outline
  blueprint_confirmed -> outline_ready

patch_outline
  outline_ready -> outline_ready

regenerate_outline_section
  outline_ready -> outline_ready

confirm_outline
  outline_ready -> outline_confirmed

generate_draft
  outline_confirmed -> draft_ready

start_roundtable
  draft_ready -> reviewing

roundtable_author_message / continue_roundtable
  reviewing -> reviewing

adopt_roundtable_suggestion
  reviewing -> reviewing

finalize
  reviewing|draft_ready -> finalized

publish_mock
  finalized -> published
```

不要允许跳过蓝图直接生成正式草稿；如果为了 Demo 降级，也必须在服务端自动补一个默认蓝图和大纲。

### 3.4 最小接口

已有接口可保留，但建议补齐或扩展：

```text
POST  /writing/sessions
GET   /writing/sessions/{session_id}
PATCH /writing/sessions/{session_id}
POST  /writing/sessions/{session_id}/confirm-claim
POST  /writing/sessions/{session_id}/blueprint
PATCH /writing/sessions/{session_id}/blueprint
POST  /writing/sessions/{session_id}/blueprint/regenerate
POST  /writing/sessions/{session_id}/blueprint/confirm
POST  /writing/sessions/{session_id}/outline
PATCH /writing/sessions/{session_id}/outline
POST  /writing/sessions/{session_id}/outline/sections/{section_id}/regenerate
POST  /writing/sessions/{session_id}/outline/confirm
POST  /writing/sessions/{session_id}/draft
POST  /writing/sessions/{session_id}/roundtable/start
POST  /writing/sessions/{session_id}/roundtable/messages
POST  /writing/sessions/{session_id}/roundtable/suggestions/{suggestion_id}/adopt
POST  /writing/sessions/{session_id}/finalize
POST  /writing/sessions/{session_id}/publish/mock
```

如果时间不足，P0 至少实现：

```text
create session
confirm claim
generate/regenerate blueprint
patch blueprint
generate outline
generate draft
roundtable with author message
finalize
publish mock
```

### 3.5 蓝图输出

```ts
WritingBlueprint {
  centralClaim: string;
  mainThread: string;
  argumentSteps: Array<{
    id: string;
    title: string;
    purpose: string;
    keyPoints: string[];
  }>;
  counterArguments: string[];
  responseStrategy: string;
  personalExperienceNeeded: string[];
  riskNotes: string[];
}
```

### 3.6 大纲与引用输出

```ts
WritingOutline {
  sections: Array<{
    id: string;
    title: string;
    purpose: string;
    keyPoints: string[];
    referencedMaterialIds: string[];
    referencedSourceIds: string[];
    missingMaterialHints: string[];
  }>;
}
```

大纲必须体现创新点：

```text
每一节都知道引用哪些材料，避免 AI 空写。
```

### 3.7 圆桌状态

```ts
RoundtableState {
  status: "not_started" | "active" | "closed";
  turns: Array<{
    id: string;
    role: "author" | "logic_reviewer" | "human_editor" | "opponent_reader" | "community_editor";
    content: string;
    createdAt: string;
  }>;
  suggestions: Array<{
    id: string;
    fromRole: string;
    content: string;
    severity: "high" | "medium" | "low";
    adopted: boolean;
  }>;
}
```

圆桌最小实现：

- 第一次开始时生成四个 Agent 的意见。
- 用户可以输入作者发言。
- 点击继续讨论时，把历史 turns + 作者发言传给 LLM。
- 用户可以采纳建议。
- 采纳建议只写入 session，不自动改用户长期 Memory。

### 3.8 发布与反馈 handoff

`publish_mock` 输出：

```json
{
  "session": {},
  "publishedArticle": {
    "articleId": "article-xxx",
    "title": "xxx",
    "interestId": "ai-coding",
    "linkedSeedId": "seed-xxx",
    "publishedAt": "2026-05-13T00:00:00Z",
    "publishMode": "mock"
  },
  "feedbackHandoff": {
    "articleId": "article-xxx",
    "linkedSeedId": "seed-xxx",
    "interestId": "ai-coding"
  }
}
```

前端或 gateway 拿到后再调用 feedback-service sync。

### 3.9 最小验收

- 用户能看到 Memory 注入，并能修改当前 session 的 Memory override。
- 蓝图可重新生成，也可手动编辑。
- 确认蓝图后能生成带引用材料的大纲。
- 圆桌允许作者发言并继续讨论。
- 定稿不自动真实发布，模拟发布后进入历史反馈。

## 4. LLM provider 路由修复

### 4.1 当前问题

当前代码和测试存在口径冲突：

- `providers.json` 默认把多数任务路由到 `openai_compat`。
- 测试仍期望 `provider_mode=zhihu` 时走 `zhihu_direct`。
- `Registry._build_providers` 会从 `config.yaml` 读取 openai_compat，因此“env 缺失不注册 openai_compat”的旧测试不再成立。
- 多 persona 圆桌在 provider 选择上也需要明确 fallback 口径。

### 4.2 推荐最终口径

`LLM_PROVIDER_MODE` / settings.provider_mode 只保留三个有效值：

```text
mock
zhihu
openai_compat
```

行为：

| mode | 行为 |
| --- | --- |
| `mock` | 所有任务强制使用 mock，不调用外部服务 |
| `zhihu` | 所有 single task 优先 `zhihu_direct`，失败 fallback mock；roundtable persona 也优先 `zhihu_direct` |
| `openai_compat` | 所有 single task 优先 `openai_compat`，失败 fallback mock；roundtable persona 优先 `openai_compat` |

`providers.json` 可保留 per_task，但在 P0 阶段 provider_mode 应优先于 per_task provider，避免不同环境下路由不可预测。

### 4.3 provider 注册规则

注册 provider 时：

```text
mock: 永远注册
zhihu_direct: 只要有 zhihu_adapter_url 就注册
openai_compat: config.yaml 或 env 中 base_url + api_key 都存在才注册
```

如果请求的 provider 没注册：

```text
不要启动失败
该 task fallback 到 mock
response.routeMeta.fallback = true
response.routeMeta.fallbackTo = "mock"
```

### 4.4 effective_provider 规则

建议实现：

```python
def effective_provider(route):
    if settings.provider_mode == "mock":
        return "mock"
    if settings.provider_mode == "zhihu":
        return "zhihu_direct"
    if settings.provider_mode == "openai_compat":
        return "openai_compat"
    return route.provider or "mock"
```

`effective_persona_provider` 同理：

```python
def effective_persona_provider(persona):
    if settings.provider_mode == "mock":
        return "mock"
    if settings.provider_mode == "zhihu":
        return "zhihu_direct"
    if settings.provider_mode == "openai_compat":
        return "openai_compat"
    return persona.provider
```

Router 里不要提前把未注册 provider 改成 mock，否则无法判断“该 persona 是否 fallback”。应让 `get_provider` 抛 `KeyError`，在当前 persona 的 try/except 里 fallback。

### 4.5 routeMeta 要求

single task 成功：

```json
{
  "mode": "single",
  "provider": "openai_compat",
  "fallback": false
}
```

single task fallback：

```json
{
  "mode": "single",
  "provider": "openai_compat",
  "fallback": true,
  "fallbackTo": "mock"
}
```

multi persona：

```json
{
  "mode": "multi_persona",
  "provider": "multi_persona",
  "fallback": true
}
```

`subCalls` 中每个 persona 单独标记：

```json
{
  "personaId": "human_editor",
  "provider": "openai_compat",
  "fallback": true,
  "latencyMs": 123
}
```

### 4.6 测试修复方向

根据最终口径更新测试，而不是让代码迎合旧假设。

需要覆盖：

```text
provider_mode=mock: 所有任务 provider=mock
provider_mode=zhihu: summarize-content routeMeta.provider=zhihu_direct
provider_mode=openai_compat 且配置存在: routeMeta.provider=openai_compat
provider_mode=openai_compat 但 provider 未注册: fallbackTo=mock
roundtable: 某一个 persona provider 抛错，只该 persona fallback
```

### 4.7 最小验收

```text
pytest services/llm-service/tests -q 通过
GET /llm/config/me 能显示当前 provider_mode
POST /llm/tasks/summarize-content 在 provider 失败时返回 fallback=true，不抛 500
POST /llm/tasks/roundtable-review 返回 subCalls，且每个 persona 有 provider/fallback
```

## 5. 给其他模型的执行顺序

建议分四个独立任务，不要并行改同一服务。

```text
任务 A: LLM provider 路由修复
范围: services/llm-service/**
验收: llm-service tests 全通过

任务 B: OAuth 画像生成最小闭环
范围: services/profile-service/**, prompts/profile-memory-synthesis.md
```
