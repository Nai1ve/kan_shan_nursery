# 看山小苗圃 - 历史反馈实现方案

## 1. 模块定位

历史反馈是看山小苗圃读写闭环的最后一环。

它不只是展示文章发布后的点赞、评论、收藏等数据，而是把发布后的读者反馈重新转化为后续创作资产。

核心目标：

```text
文章发布
→ 收集反馈
→ 理解读者反应
→ 生成二次创作方向
→ 回流观点种子 / Memory 更新建议
```

历史反馈需要回答三个问题：

```text
1. 这篇文章表现怎么样？
2. 读者认可、质疑、追问的是什么？
3. 这些反馈如何进入下一轮创作？
```

## 2. 产品原则

### 2.1 反馈不是单纯统计

历史反馈不应该只展示：

```text
点赞数
评论数
收藏数
阅读数
```

而应该进一步解释：

```text
哪些观点被认可
哪些表达被质疑
哪些内容缺少证据
哪些评论可以变成下一篇文章
```

### 2.2 不直接修改长期 Memory

历史反馈可以生成 Memory 更新建议，但不能直接写入长期 Memory。

所有重要画像更新必须经过用户确认：

```text
反馈分析
→ Memory 更新建议
→ 用户确认 / 编辑 / 拒绝
→ profile-service 写入 Memory
```

### 2.3 P0 只做手动刷新

当前距离比赛截止时间有限，P0 不做后台定时追踪。

历史反馈页提供手动刷新按钮：

```text
用户点击刷新反馈
→ 拉取最新评论和指标
→ 调用 LLM 分析
→ 更新反馈画像
```

P1 再考虑 2-3 周持续追踪和趋势分析。

## 3. 端到端流程

```text
写作苗圃定稿
↓
模拟发布 / 知乎圈子发布
↓
writing-service 通知 feedback-service 创建反馈文章
↓
用户进入历史反馈页
↓
点击刷新反馈
↓
feedback-service 拉取文章指标和评论
↓
llm-service 生成反馈分析
↓
用户选择：
  - 查看评论摘要
  - 生成二次文章
  - 转为观点种子
  - 生成 Memory 更新建议
```

## 4. 服务边界

### 4.1 writing-service

负责写作到发布的动作。

输入：

```ts
WritingSession
IdeaSeed
FinalDraft
PublishPayload
```

职责：

```text
1. 完成定稿。
2. 执行模拟发布或真实发布。
3. 发布成功后调用 feedback-service 创建反馈文章记录。
```

不负责：

```text
1. 评论分析。
2. Memory 更新。
3. 二次创作机会判断。
```

### 4.2 feedback-service

历史反馈主服务。

职责：

```text
1. 管理历史文章记录。
2. 管理反馈快照。
3. 拉取最新评论和指标。
4. 调用 LLM 生成反馈分析。
5. 生成二次创作建议。
6. 生成观点种子候选。
7. 生成 Memory 更新建议。
```

不负责：

```text
1. 真正发布文章。
2. 直接写入长期 Memory。
3. 直接创建正式观点种子。
```

### 4.3 zhihu-adapter

负责知乎侧数据适配。

P0 主要使用：

```text
GET /openapi/ring/detail
GET /openapi/comment/list
```

如果真实文章指标暂时无法通过 API 获取，P0 只依赖评论数据和圈子可返回的互动数。

不可把 mock 指标伪装成真实知乎数据。前端展示时必须标记指标来源：

```text
real        真实接口返回
estimated   系统根据真实数据估算
mock        Demo 模拟
unavailable 知乎当前未开放
```

### 4.4 llm-service

负责反馈分析。

使用任务：

```text
feedback-summary
```

路由规则：

```text
通用提取类任务使用平台 LLM。
历史反馈属于用户个人写作表现分析，应优先使用用户配置 LLM。
用户 LLM 失败后使用平台兜底，并由前端统一弹框提醒。
```

### 4.5 seed-service

负责把反馈候选转成正式观点种子。

输入：

```ts
FeedbackSeedCandidate
```

输出：

```ts
IdeaSeed
```

### 4.6 profile-service

负责 Memory 更新请求。

历史反馈只创建更新建议：

```text
POST /memory/update-requests
```

用户确认后才写入：

```text
POST /memory/update-requests/{request_id}/apply
```

### 5.4 知乎开放能力边界

历史反馈必须严格区分“知乎已经提供的能力”和“看山小苗圃自己生成的分析能力”。

#### 5.4.1 知乎当前可提供

按 `docs/知乎API.md` 当前能力，历史反馈可以真实接入：

```text
1. 圈子内容列表
   GET /openapi/ring/detail
   可获取圈子帖子列表，以及帖子级 comment_num、like_num、fav_num 等粗粒度互动字段。

2. 评论列表
   GET /openapi/comment/list
   可获取帖子或评论下的评论内容、回复关系、评论点赞数、发布时间。

3. 圈子发布
   POST /openapi/publish/pin
   发布成功后返回 content_token，可作为后续反馈追踪的稳定关联 ID。

4. 评论 / 回复 / 点赞
   POST /openapi/comment/create
   POST /openapi/reaction
   可支持后续维护讨论，但不属于被动反馈分析数据源。
```

因此 P0 真实可做：

```text
评论摘要
主要认可点
主要质疑点
读者追问
读者补充材料
评论中可转化的新种子
二次创作角度
圈子内容可用时展示点赞数 / 评论数 / 收藏数
```

#### 5.4.2 知乎当前没有提供

以下能力不能按真实接口承诺，需要隐藏、标记不可用、使用 mock，或由用户手动补充：

```text
1. 阅读量、曝光量、点击率、完读率。
2. 收藏率、点赞率、转化率。
   即使圈子返回 fav_num，也没有曝光或阅读分母，不能计算真实“率”。
3. 普通知乎回答 / 文章的发布与反馈追踪。
   当前开放的是圈子想法发布，不等价于用户正常知乎回答或文章。
4. 评论情绪、争议度、认可点、反方观点。
   这些需要 llm-service 基于评论内容分析生成。
5. 读者画像、粉丝画像、评论人群画像。
   关注和粉丝接口可以提供关系数据，但不会直接提供读者类型分析。
6. 自动反馈推送 / webhook。
   当前没有可用 webhook，需要用户手动刷新或后续自建定时任务。
7. 历史趋势曲线。
   知乎返回的是当前状态，趋势需要每次刷新时保存 FeedbackSnapshot。
```

前端展示规则：

```text
阅读量：知乎暂未开放
完读率：知乎暂未开放
收藏率：无法计算
评论分析：已接入
点赞 / 评论 / 收藏数：圈子内容可用时展示
```

#### 5.4.3 P0 真实追踪边界

P0 只承诺追踪两类内容：

```text
1. 通过看山小苗圃发布到知乎圈子的内容。
   必须保存 ringId、contentToken、contentType=pin。

2. Demo 模拟发布内容。
   必须标记 publishMode=mock，所有指标都显示为 Demo 模拟。
```

用户手动发布到普通知乎回答、文章、想法的内容，P0 不做自动追踪。后续可以支持用户手动录入链接和评论，但不能承诺通过当前 API 自动抓取完整反馈。

#### 5.4.4 LLM 生成能力边界

以下内容由看山小苗圃通过 LLM 和规则生成，不是知乎 API 原生字段：

```text
主要认可点
主要质疑点
开放问题
材料缺口
评论争议度
文章画像
二次创作建议
FeedbackSeedCandidate
MemoryUpdateCandidate
```

生成时必须保留来源评论 ID 或来源摘要，方便用户判断分析是否可信。

## 7. 反馈分析算法

P0 使用规则预处理 + LLM 总结，不做复杂机器学习。

### 7.1 评论预处理

输入：

```text
comments
article
seed
memory
```

处理：

```text
1. 过滤空评论、重复评论、明显无效评论。
2. 按 likeCount 和回复关系排序。
3. 截断过长评论，保留核心内容。
4. 选取 Top N 高价值评论进入 LLM。
```

高价值评论判断：

```text
1. 点赞较高。
2. 包含问号或追问。
3. 明确反对或补充。
4. 提供案例、数据、个人经验。
5. 与文章核心观点高度相关。
```

### 7.2 评论分类

评论分类枚举：

```text
agree        认同 / 补充支持
disagree     反对 / 质疑
question     追问 / 没看懂
evidence     提供新材料
experience   分享个人经验
noise        无效评论
```

### 7.3 反馈信号提取

LLM 需要提取：

```text
认可点：
读者接受了哪些观点或表达？

质疑点：
读者主要反对什么？

理解障碍：
哪些内容让读者追问或误解？

材料缺口：
文章缺少哪些证据、案例、数据或个人经验？

二次创作机会：
哪些评论可以发展成下一篇文章？
```

### 7.4 文章画像生成

输出文章画像：

```text
最强观点
最弱环节
吸引的读者类型
主要争议地图
表达风格反馈
下一篇优先改进点
```

### 7.5 二次创作建议

二次创作建议必须满足：

```text
1. 来自真实反馈，而不是凭空生成。
2. 能指出来源评论。
3. 有明确写作角度。
4. 能解释为什么值得继续写。
```

## 8. LLM 任务设计

### 8.1 feedback-summary

请求：

```ts
{
  taskType: "feedback-summary"
  input: {
    article: FeedbackArticle
    seed?: IdeaSeed
    memory?: {
      globalMemory?: object
      interestMemory?: object
    }
    metrics: FeedbackMetrics
    comments: FeedbackComment[]
  }
  promptVersion: "v1"
  schemaVersion: "v1"
}
```

响应：

```ts
{
  summary: string
  signals: ReaderSignal[]
  positiveFeedback: string[]
  negativeFeedback: string[]
  openQuestions: string[]
  counterArguments: string[]
  missingMaterials: string[]
  articlePortrait: ArticlePortrait
  secondArticleIdeas: SecondArticleIdea[]
  seedCandidates: FeedbackSeedCandidate[]
  memoryUpdateCandidates: MemoryUpdateCandidate[]
}
```

### 8.2 Provider 路由

历史反馈属于个人写作表现分析。

路由策略：

```text
优先使用用户配置 LLM。
如果用户 LLM 失败，使用平台免费额度兜底。
后端返回 notices。
前端统一弹框提示用户 LLM 连接失效。
```

## 9. feedback-service 接口设计

### 9.1 查询历史文章

```http
GET /feedback/articles?user_id={user_id}&interest_id={interest_id?}
```

返回：

```ts
{
  items: FeedbackArticle[]
}
```

### 9.2 查询文章详情

```http
GET /feedback/articles/{article_id}
```

返回：

```ts
{
  article: FeedbackArticle
  snapshots: FeedbackSnapshot[]
  latestAnalysis?: FeedbackAnalysis
}
```

### 9.3 从写作 session 创建反馈文章

```http
POST /feedback/articles/from-writing-session
```

请求：

```ts
{
  userId: string
  writingSessionId: string
  seedId?: string
  interestId: string
  title: string
  finalDraft: string
  coreClaim: string
  articleType: string
  publishMode: "mock" | "zhihu_ring"
  platformContentId?: string
  ringId?: string
  url?: string
  publishedAt: string
}
```

### 9.4 手动刷新反馈

```http
POST /feedback/articles/{article_id}/refresh
```

行为：

```text
1. 根据 publishMode 判断数据来源。
2. mock 发布读取 mock 指标和评论。
3. zhihu_ring 发布调用 zhihu-adapter 拉评论。
4. 创建 FeedbackSnapshot。
5. 更新 FeedbackArticle.latestMetrics。
```

### 9.5 生成反馈分析

```http
POST /feedback/articles/{article_id}/analyze
```

行为：

```text
1. 读取最新 snapshot。
2. 读取文章关联 seed。
3. 读取用户兴趣 Memory。
4. 调用 llm-service feedback-summary。
5. 保存 FeedbackAnalysis。
6. 更新 FeedbackArticle.status = analyzed。
```

### 9.6 反馈候选转观点种子

```http
POST /feedback/articles/{article_id}/seed-candidates/{candidate_id}/create
```

行为：

```text
1. 找到 seedCandidate。
2. 调用 seed-service 创建 IdeaSeed。
3. 来源类型标记为 feedback_comment。
4. 保留 articleId 和 sourceCommentIds。
```

### 9.7 反馈候选生成 Memory 更新请求

```http
POST /feedback/articles/{article_id}/memory-candidates/{candidate_id}/request
```

行为：

```text
1. 找到 memoryUpdateCandidate。
2. 调用 profile-service 创建 Memory update request。
3. 不直接写入 Memory。
```

## 11. 状态流转

### 11.1 FeedbackArticle 状态

```text
tracking
  初始状态，文章已发布，等待反馈刷新

analyzed
  已完成至少一次 LLM 反馈分析

closed
  用户不再追踪该文章
```

### 11.2 Snapshot 状态

Snapshot 本身不可变。

每次刷新都新建一份：

```text
snapshot_1
snapshot_2
snapshot_3
```

P0 前端只展示最新一份。

P1 再做趋势对比。

## 12. P0 实现优先级

### P0 必做

```text
1. writing-service 发布后创建 FeedbackArticle。
2. feedback-service 支持历史文章列表。
3. feedback-service 支持文章详情。
4. feedback-service 支持手动刷新反馈。
5. llm-service 支持 feedback-summary。
6. 前端历史反馈页接真实接口。
7. 支持反馈候选转观点种子。
```

### P1 再做

```text
1. 多次 snapshot 趋势对比。
2. 2-3 周自动追踪。
3. 评论热度排序优化。
4. Memory 更新建议批量确认。
5. 文章表现评分。
```

### P2 再做

```text
1. 长期写作风格演化。
2. 不同文章风格对比。
3. 选题成功率预测。
4. 发布最佳时机建议。
```

## 13. 模型任务拆分建议


```text
services/feedback-service/**
```


```text
services/writing-service/**
```

```text

2. 刷新反馈按钮。
```

```text
frontend/**
```

## 15. P0 简化约束

```text
1. 不做自动定时追踪。
2. 不做复杂趋势图。
3. 不直接更新 Memory。
4. 不依赖真实阅读量 API。
5. 不要求真实知乎文章发布，圈子发布或 mock 发布均可。
6. 不把 mock 阅读量、完读率、收藏率展示成真实知乎指标。
7. 普通知乎回答 / 文章暂不承诺自动追踪。
```

这样可以保证历史反馈在 demo 中形成完整闭环，同时不会拖慢主链路开发。
