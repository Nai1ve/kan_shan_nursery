## 1. 项目完成度与 Demo 说明

### 1.1 当前完成度概览

看山小苗圃当前已经不是纯前端原型，而是一个具备前后端骨架、知乎能力适配层、LLM 路由和核心状态机的读写创作工作台。

当前主链路：

```text
登录与用户画像
→ 今日看什么
→ 观点种子库
→ 今日发芽
→ 写作苗圃
→ 历史反馈
```

当前代码中已经预设或部分实现的服务：

| 模块 | 当前能力 | Demo 状态 |
| --- | --- | --- |
| 前端入口 | 登录、注册、知乎绑定、LLM 配置、用户画像入口 | 可演示，仍需做完整浏览器验收 |
| api-gateway | 前端统一入口，代理 profile/content/seed/llm/sprout/writing/feedback | 可作为主入口 |
| profile-service | 用户、知乎绑定、兴趣、Memory、LLM 配置、画像生成任务 | 用户入口 P0 可用，OAuth 画像生成仍需联调 |
| zhihu-adapter | 热榜、知乎搜索、全网搜索、直答、圈子、评论、关注流、用户信息 | Data Platform 和 OAuth 能力已进入 live 接入阶段 |
| content-service | 内容池、用户快照、卡片评分、刷新、来源详情、按需 enrich | 今日看什么主链路可演示 |
| seed-service | 种子创建、from-card、疑问、四格材料、Agent 补材料、合并 | 观点种子库主链路可演示 |
| llm-service | task router、mock/provider 路由、用户 LLM、平台兜底、prompt | P0 可用，需继续校准路由和失败提示 |
| sprout-service | 手动触发、机会状态、补材料、换角度、开始写作 | 接口已存在，算法联调仍是 P1 风险 |
| writing-service | session、观点确认、蓝图、大纲、草稿、圆桌、定稿、模拟发布 | 服务骨架较完整，前端接线仍需补强 |
| feedback-service | 文章记录、同步、评论摘要、二次种子、Memory 更新建议 | Demo 可 mock，真实反馈受知乎接口限制 |

### 1.2 推荐 Demo 主流程

Demo 应该重点展示“不是一键代写，而是从高质量输入到观点形成再到表达”的闭环。

推荐展示顺序：

```text
1. 新用户登录 / 注册
2. 配置兴趣领域、写作风格和用户 LLM
3. 关联知乎账号，展示后续可接入关注流和用户画像
4. 进入今日看什么，展示真实知乎来源生成的兴趣卡片
5. 展开卡片来源，展示原始内容、摘要、争议点、可写角度
6. 用户选择认同 / 反对 / 有疑问
7. 系统把卡片和用户反应沉淀为观点种子
8. 进入种子库，继续浇水：补证据、找反方、问 Agent
9. 手动触发今日发芽，寻找今天值得写的种子
10. 进入写作苗圃，完成观点确认、蓝图、大纲、草稿、圆桌审稿、定稿
11. 模拟发布，进入历史反馈
12. 反馈分析生成二次创作建议或 Memory 更新建议
```

### 1.3 Demo 中最应该强调的创新点

答辩时不建议把重点放在“AI 生成一篇文章”，而应该强调：

```text
1. 输入先于输出：
   先筛选高质量内容，再辅助表达。

2. 观点先于文章：
   用户的认同、反对、疑问会沉淀为观点种子，而不是立即生成正文。

3. Memory 可见可控：
   用户能看到兴趣 Memory 如何被注入写作流程，也能修改。

4. AI 不代理立场：
   AI 负责摘要、反方、证据、结构和表达，不替用户决定观点。

5. 反馈回流：
   发布后的评论和反馈会变成新种子、二次创作方向和 Memory 更新建议。
```

### 1.4 Demo 降级策略

比赛现场如果真实接口不稳定，按以下顺序降级：

| 风险 | 降级方式 |
| --- | --- |
| 知乎 OAuth 不稳定 | 使用已绑定用户或展示关注流能力说明，主流程继续走热榜/搜索 |
| 用户自配 LLM 失败 | 使用平台兜底 LLM，并在前端弹框提示用户配置失效 |
| 直答额度不足 | 使用 mock provider 或缓存结果继续演示状态流转 |
| 圈子发布不可用 | 使用模拟发布，不影响历史反馈闭环展示 |
| 历史真实指标缺失 | 明确展示“知乎暂未开放”，只演示评论摘要和二次创作 |

## 3. 项目边界与当前限制

### 3.1 产品边界

看山小苗圃不是：

```text
1. 普通热榜搬运工具。
2. 一键代写文章工具。
3. 自动替用户发布内容的工具。
4. 静默修改用户画像的推荐系统。
```

看山小苗圃是：

```text
1. 高质量输入筛选器。
2. 观点种子沉淀系统。
3. 写作结构化协作工具。
4. 用户可控的 Memory 和反馈回流系统。
```

### 3.2 知乎能力边界

当前不能承诺的能力：

```text
1. 普通知乎回答 / 文章的自动发布和完整反馈追踪。
2. 阅读量、曝光量、完读率、点击率等真实数据。
3. 收藏率、点赞率、转化率等需要分母的真实比例。
4. 评论情绪、读者画像、争议度等结构化结论。
5. webhook 或平台主动推送反馈。
```

这些能力需要由系统自建、LLM 分析、用户手动补充，或在知乎后续开放接口后再接入。

### 3.3 技术限制

当前主要技术限制：

```text
1. 多服务已具备骨架，但部分前端流程还没有完全接真实服务。
2. 今日发芽和写作苗圃的服务接口较完整，但端到端联调不足。
3. 历史反馈当前更适合 Demo mock，真实评论追踪依赖圈子发布和评论接口。
4. LLM provider 路由需要稳定处理用户 LLM 失败、平台兜底和前端弹框提醒。
5. 知乎 API 有额度限制，必须依赖 Redis 短缓存和按需 enrich。
6. OAuth 依赖外部回调域名，本地和线上环境需要不同配置。
```

### 3.4 Demo 限制表达

答辩中可以这样说明边界：

```text
当前 Demo 已经完成读写闭环的核心状态流转。
真实知乎输入主要覆盖热榜、搜索、全网搜索和关注流。
发布和历史反馈在 P0 中优先支持知乎圈子和模拟发布。
普通回答/文章的完整数据追踪受当前开放 API 限制，不作为 P0 承诺。
```

## 4. 效果评估指标设计

评估不只看“生成内容是否流畅”，而要看系统是否真正帮助用户完成：

```text
看到好内容
形成好观点
写出好表达
收到反馈后继续改进
```

### 4.1 今日看什么指标

| 指标 | 含义 | 计算方式 |
| --- | --- | --- |
| 卡片有效打开率 | 用户是否愿意展开阅读 | 展开来源 / 卡片曝光 |
| 种子转化率 | 卡片是否激发观点 | 由卡片生成种子数 / 卡片曝光 |
| 疑问触发率 | 卡片是否激发思考 | 有疑问次数 / 卡片曝光 |
| 观点反应率 | 用户是否有立场 | 认同或反对次数 / 卡片曝光 |
| 内容多样性 | 是否避免同质化 | 同批卡片来源类型、主题、作者去重率 |
| 刷新满意度 | 是否能快速看到新东西 | 刷新后有效操作数 / 刷新次数 |
| LLM 等待影响 | 卡片是否被 LLM 阻塞 | 首屏返回耗时、按需 enrich 耗时 |

### 4.2 观点种子指标

| 指标 | 含义 | 计算方式 |
| --- | --- | --- |
| 种子成熟度 | 是否具备写作条件 | 基于证据、反方、个人经验、开放问题计算 |
| 浇水完成度 | 材料是否均衡 | 四格材料中非空且被采纳的类型数 |
| 疑问解决率 | Agent 是否帮助用户澄清 | resolved questions / total questions |
| 种子写作转化率 | 种子是否进入输出 | 开始写作种子数 / 总种子数 |
| 相似种子合并率 | 是否减少重复沉淀 | 合并次数 / 新增种子数 |

### 4.3 今日发芽指标

| 指标 | 含义 | 计算方式 |
| --- | --- | --- |
| 发芽命中率 | 推荐机会是否有用 | 用户选择开始写作 / 发芽机会曝光 |
| 触发解释采纳率 | 用户是否认可“为什么今天写” | 用户保留机会 / 机会曝光 |
| 材料补充成功率 | 发芽是否补齐写作缺口 | 补充材料被采纳数 / 推荐材料数 |
| 重复推荐率 | 是否重复打扰用户 | 重复 seed/opportunity 曝光比例 |
| 热点相关性 | 是否真和今天有关 | 热榜/今日卡片与 seed 语义匹配得分 |

### 4.4 写作苗圃指标

| 指标 | 含义 | 计算方式 |
| --- | --- | --- |
| 蓝图确认率 | 观点结构是否可用 | confirmed blueprint / generated blueprint |
| 蓝图重生成率 | 初版是否偏离用户 | regenerate blueprint / generated blueprint |
| 大纲编辑率 | 用户是否参与结构决策 | edited outline / generated outline |
| 引用覆盖率 | 大纲是否关联材料 | 带引用段落数 / 总段落数 |
| 圆桌建议采纳率 | 审稿是否有价值 | adopted suggestions / total suggestions |
| 定稿完成率 | 是否完成写作闭环 | finalized sessions / writing sessions |
| 人类参与度 | 是否避免 AI 代写 | 用户编辑字数、确认次数、作者发言次数 |

### 4.5 历史反馈指标

| 指标 | 含义 | 计算方式 |
| --- | --- | --- |
| 评论分析覆盖率 | 反馈是否被理解 | 已分析评论数 / 总评论数 |
| 二次种子转化率 | 反馈是否回流资产 | 由反馈创建种子数 / 反馈文章数 |
| Memory 建议采纳率 | 画像更新是否可信 | applied memory requests / generated requests |
| 反方补强率 | 是否帮助下一篇更扎实 | 由反馈补充的 counterargument 数 |
| 反馈闭环完成率 | 发布后是否继续生长 | 有分析、有新种子或有 Memory 建议的文章数 / 发布文章数 |

### 4.6 系统运行指标

| 指标 | 目标 |
| --- | --- |
| 内容缓存命中率 | 降低知乎 API 调用和等待 |
| LLM 缓存命中率 | 降低成本和额度消耗 |
| 用户 LLM 失败率 | 用于提示用户检查配置 |
| 平台兜底触发率 | 衡量用户 LLM 可用性和平台成本 |
| 接口 P95 延迟 | 保证 Demo 流畅 |
| 知乎 API quota 消耗 | 避免比赛现场额度耗尽 |
| 端到端错误率 | 确保主链路稳定 |

## 5. 后续开发计划

### 5.1 截止前 P0：保证 Demo 闭环

目标：让用户能完整走通“看见 → 种下 → 发芽 → 写作 → 反馈”的主流程。

任务：

```text
1. 修复所有阻断前端主流程的接口错误。
2. 确认今日看什么能稳定返回真实卡片。
3. 验证关注流精选通过 OAuth token 拉取并进入内容池。
4. 确保卡片操作全部落到 seed-service。
5. 今日发芽改为手动触发，不在初始页自动请求。
6. 写作苗圃接入 writing-service 的蓝图、大纲、草稿和圆桌接口。
7. 模拟发布后写入 feedback-service。
8. 所有用户 LLM 失败都由前端弹框提示，后端只返回 notices。
9. `npm run build` 和关键后端单测通过。
```

### 5.2 赛后 P1：提升真实体验

目标：减少 mock，提升推荐质量和写作协作质量。

任务：

```text
1. content-service 引入动态 QueryPlan，而不是只依赖固定兴趣词。
2. 今日看什么加入 MMR 多样性重排，减少同质卡片。
3. 关注流、搜索、热榜、全网搜索做多来源聚合和去重。
4. 发芽算法接入真实 seed/content/profile/llm 全链路。
5. 写作苗圃支持用户逐段编辑、引用材料管理和圆桌多轮辩论。
6. 历史反馈接入知乎圈子评论列表，生成真实评论摘要。
7. Memory 更新建议全部走用户确认流。
8. 增加系统埋点和效果指标看板。
```

### 5.3 赛后 P2：做成可长期使用的创作工作台

目标：从 Demo 系统升级为稳定产品。

任务：

```text
1. 支持 2-3 周历史反馈追踪和趋势分析。
2. 支持用户手动导入普通知乎文章 / 回答链接并补充反馈。
3. 建立长期写作风格演化报告。
4. 对不同兴趣分类建立独立 Memory 和表达策略。
5. 引入更细的 LLM 成本控制、模型选择和质量评估。
6. 完善生产部署、日志、监控、告警和数据备份。
7. 做多用户隔离、权限校验和数据安全审查。
8. 根据真实用户行为持续优化卡片评分和发芽算法。
```

### 5.4 不建议短期投入的方向

这些方向展示价值不高或风险较高，当前不建议抢做：

```text
1. 自动后台高频发芽推送。
2. 自动发布到普通知乎回答 / 文章。
3. 用不可靠方式抓取知乎未开放数据。
4. 复杂富文本编辑器。
5. 大规模多用户推荐训练。
6. 无用户确认的自动 Memory 写入。
```

短期最重要的是把“读写闭环”讲清楚、跑稳定、让用户感知到每一步都由自己的观点驱动。
