# 看山小苗圃-今日看什么卡片获取与推流设计

> 2026-05-13 流程校准：24 小时冲刺阶段以 `docs/看山小苗圃-核心流程校准方案.md` 为准。本文档保留为后续细化参考。

本文档定义 `content-service` 如何从“单纯热榜获取”升级为“高质量阅读输入生成器”。

当前问题：

```text
主要依赖 hot_list
→ 内容固定
→ 刷新后变化少
→ 兴趣小类不够个性化
→ 很难真正实现“看到好内容”
```

目标状态：

```text
用户 Memory
+ 知乎热榜
+ 知乎搜索
+ 全网搜索
+ 关注流
+ LLM 聚合与筛选
→ 每个分类生成少量值得看的多来源卡片
```

---

## 1. 服务定位

### 1.1 content-service 负责

- 根据用户画像生成 Query Plan。
- 通过 `zhihu-adapter` 获取内容候选。
- 聚合多个来源为 `WorthReadingCard`。
- 调用 `llm-service` 生成摘要、争议点、可写角度。
- 管理分类候选池、刷新、去重和多样性排序。

### 1.2 content-service 不负责

- 不直接调用知乎官方 API，必须通过 `zhihu-adapter`。
- 不直接调用模型供应商，必须通过 `llm-service`。
- 不创建观点种子。
- 不计算种子成熟度。
- 不发布内容。

---

## 2. 内容分类

前端分类与业务语义：

| 类型 | 示例 | 是否长期兴趣 | 说明 |
| --- | --- | --- | --- |
| 兴趣小类 | AI Coding、RAG、后端工程 | 是 | 来自用户画像和 Memory |
| 关注流 | 关注作者动态 | 否 | 与兴趣小类平级，单独 tab |
| 偶遇输入 | 相邻但不完全重合的主题 | 否 | 用于打破信息茧房 |

规则：

```text
关注流和偶遇输入是输入来源分类，不写入长期兴趣主类。
兴趣小类由 profile-service 管理。
```

## 3. 可用知乎能力

### 3.1 hot_list

接口：

```text
GET /api/v1/content/hot_list
Limit <= 30
额度：100 / day / user
TTL：15 分钟
```

用途：

- 公共热点输入。
- 今日发芽时效信号。
- 为兴趣小类生成“热点扩展查询”。

不应作为唯一来源。

### 3.2 zhihu_search

接口：

```text
GET /api/v1/content/zhihu_search
Query: string
Count <= 10
额度：1000 / day / user
TTL：1 小时
```

用途：

- 兴趣小类主来源。
- 站内优质讨论、问答、文章。
- 观点种子证据和反方材料。

核心字段：

```text
ContentID
Title
ContentText
Url
AuthorName
EditTime
CommentCount
VoteUpCount
AuthorityLevel
RankingScore
CommentInfoList
SearchHashId
```

### 3.3 global_search

接口：

```text
GET /api/v1/content/global_search
Query: string
Count <= 20
额度：1000 / day / user
TTL：1 小时
```

用途：

- 外部证据补充。
- 偶遇输入。
- 新信息增益。
- 反方材料和跨站讨论。

注意：

```text
ContentText 可能包含 <em> 高亮。
adapter 需保留 rawExcerptHtml，并生成去标签 summary。
```

### 3.4 following feed

接口：

```text
GET /openapi/feed/following
GET /openapi/user/following
GET /openapi/user/followers
```

用途：

- 关注流 tab。
- 用户关系网络下的高质量输入。
- 画像增强。

OAuth 未开放时：

```text
返回空状态或 mock。
不阻塞兴趣小类和偶遇输入。
```

### 3.5 direct_answer / LLM

接口：

```text
POST /v1/chat/completions
通过 zhihu-adapter 或 llm-service provider 路由
额度：100 / day / user
TTL：6 小时
```

用途：

- 内容摘要。
- 争议点提取。
- 可写角度生成。
- Query 扩展和主题归类。

注意：

```text
不要对所有候选逐条调用 LLM。
先规则筛选，后 TopK enrichment。
```

## 4. 总体数据流

```text
GET /api/v1/content/bootstrap
↓
1. profile-service 获取 ProfileData + Memory
↓
2. content-service 构建 CategoryQueryPlan
↓
3. zhihu-adapter 读取 Redis / 回源知乎 API
↓
4. 标准化为 ZhihuContentItem
↓
5. content-service 候选池去重、评分、聚类
↓
6. 聚合成 WorthReadingCard
↓
7. llm-service enrich TopK 卡片
↓
8. 写入 Redis 候选池
↓
9. 返回前端
```

## 5. Query Plan

### 5.1 输入

```ts
QueryPlanInput {
  userId: string
  categoryId: string
  categoryName: string
  categoryKind: "interest" | "following" | "serendipity"
  globalMemory: GlobalMemory
  interestMemory?: MemorySummary
  recentSeeds: IdeaSeed[]
  shownCardIds: string[]
  hotListDigest: ZhihuContentItem[]
}
```

### 5.2 输出

```ts
CategoryQueryPlan {
  categoryId: string
  mode: "interest" | "following" | "serendipity"
  queries: QuerySpec[]
  diversityPolicy: DiversityPolicy
  maxCandidateItems: number
  maxCards: number
}

QuerySpec {
  query: string
  source: "hot_list" | "zhihu_search" | "global_search" | "following"
  intent: "core" | "hot_expand" | "counter" | "evidence" | "adjacent" | "fresh"
  count: number
  priority: number
}
```

### 5.3 兴趣小类 Query 生成

来源：

```text
category.name
interestMemory.interestName
interestMemory.preferredPerspective
interestMemory.evidencePreference
recentSeeds.coreClaim
hotList titles that match category
```

示例：AI Coding

```json
{
  "categoryId": "ai-coding",
  "queries": [
    {"query": "AI Coding 程序员成长", "source": "zhihu_search", "intent": "core", "count": 10},
    {"query": "AI 编程工具 工程交付", "source": "zhihu_search", "intent": "evidence", "count": 10},
    {"query": "AI Coding 会不会削弱基本功", "source": "zhihu_search", "intent": "counter", "count": 10},
    {"query": "AI coding developer workflow", "source": "global_search", "intent": "adjacent", "count": 10}
  ]
}
```

### 5.4 热榜扩展查询

热榜不能只展示为卡片，而应该作为 query expansion：

```text
hot title 与兴趣小类匹配
→ 抽取 2-3 个关键词
→ 拼接兴趣关键词
→ 发起 zhihu_search/global_search
```

例：

```text
热榜标题：某 AI 编程工具发布新功能
兴趣：AI Coding / 程序员成长
扩展 Query：
- "AI 编程工具 程序员 工作流"
- "AI Coding 初级程序员 成长"
- "AI 工具 工程判断"
```

### 5.5 偶遇输入 Query

目标不是随机，而是“相邻探索”。

来源：

```text
用户兴趣集合
全局 Memory 内容偏好
热榜中未命中兴趣但存在公共价值的话题
global_search adjacent topics
```

策略：

```text
70% 相邻兴趣
20% 公共热点
10% 完全陌生但有高质量解释价值的话题
```

偶遇输入必须给出解释：

```text
“为什么推荐给你”
“它和你已有兴趣的连接点”
```

### 5.6 关注流 Query

关注流不走搜索 query，而走 OAuth feed：

```text
following_feed
→ 关注作者/动态
→ 内容密度评分
→ 与用户兴趣/种子匹配
→ 生成关注流精选卡片
```

OAuth 不可用时：

```text
返回 mock 或空态：
“知乎授权暂不可用，关联后可展示关注流精选。”
```

## 6. 候选获取

### 6.1 调用顺序

每个兴趣分类：

```text
1. 读取 hot_list cache
2. 执行 zhihu_search core/counter/evidence
3. 执行 global_search adjacent/evidence
4. 合并 candidates
5. 去重
```

### 6.2 Redis 缓存

使用 zhihu-adapter 已定义缓存：

```text
zhihu:hot_list:{limit}
zhihu:zhihu_search:{query_hash}:{count}
zhihu:global_search:{query_hash}:{count}
zhihu:user_moments:{user_id}
```

content-service 自己维护候选池缓存：

```text
content:candidate_pool:{user_id}:{category_id}:{date}:{plan_hash}
content:cards:{user_id}:{category_id}:{date}:{plan_hash}
content:shown:{user_id}:{category_id}:{date}
```

TTL：

```text
candidate_pool: 6 小时
cards: 6 小时
shown: 当天有效
```

### 6.3 额度控制

建议 P0 每次完整刷新预算：

```text
hot_list: 1 次
每个兴趣小类 zhihu_search: 2-3 次
每个兴趣小类 global_search: 0-1 次
偶遇输入 global_search: 2 次
LLM enrich: 每分类最多 5 张卡
```

规则：

```text
缓存命中不扣额度。
缓存 miss 前检查 quota。
额度不足时降级为已有候选池 + mock enrichment。
```

## 8. 候选评分

每个 `ZhihuContentItem` 先计算 item 分：

```text
itemScore =
  categoryRelevance * 0.30
+ authorityScore * 0.18
+ engagementScore * 0.14
+ recencyScore * 0.12
+ controversyPotential * 0.12
+ noveltyScore * 0.08
+ writingPotential * 0.06
- lowQualityPenalty
```

### 8.1 categoryRelevance

```text
标题/摘要与 category query 的 token overlap
+ interestMemory.preferredPerspective 命中
+ recentSeeds 相关
+ hot_expand 命中
```

P0：

```text
token overlap + RankingScore
```

P1：

```text
LLM 判断 fitScore，只对 Top 20 候选执行。
```

### 8.2 authorityScore

来源：

```text
AuthorityLevel
AuthorBadgeText
RankingScore
```

映射：

```text
AuthorityLevel high → 90
medium → 70
low → 50
missing → 55
```

### 8.3 engagementScore

```text
log1p(VoteUpCount) * 0.6 + log1p(CommentCount) * 0.4
```

归一化到 0-100。

### 8.4 recencyScore

```text
24h 内 100
3d 内 80
7d 内 60
30d 内 35
更早 15
无时间 50
```

### 8.5 controversyPotential

来源：

```text
评论数
精选评论
标题是否包含“是否/会不会/为什么/争议”
LLM extract-controversies 结果数量
```

P0 先用规则：

```text
CommentCount 高 + 问句标题 + selectedComments 非空
```

### 8.6 noveltyScore

避免一直刷到相同内容。

```text
sourceId 未展示 +25
title 与今日已展示卡片相似度低 +25
author 未重复 +15
sourceType 多样 +15
兴趣相邻但不完全相同 +20
```

### 8.7 lowQualityPenalty

```text
summary 过短 -10
标题营销味过强 -10
无作者且无来源信息 -8
纯情绪无信息密度 -15
高风险话题缺权威来源 -15
```

## 9. 多来源聚合成卡片

卡片不是单条内容，而是一个“阅读议题”。

```text
2-3 个相关 ZhihuContentItem
→ 1 张 WorthReadingCard
```

### 9.1 聚类规则

同一 cluster 条件：

```text
title token Jaccard >= 0.45
OR shared keywords >= 2
OR same hot_expand topic
OR searchHashId 相同且标题相近
```

每个 cluster 最多 3 个来源：

```text
primary: 分数最高或最能代表主题
support: 高权威/高赞/案例
counter: 反方或不同角度
```

### 9.2 ContentSource 生成

```ts
ContentSource {
  sourceId: string
  sourceType: string
  sourceUrl?: string
  title: string
  author?: string
  publishedAt?: string
  authorityMeta?: string
  rawExcerpt: string
  fullContent?: string
  contribution: string
}
```

`contribution` 生成规则：

```text
primary → 提供核心话题和主要讨论背景
support → 提供补充论据、数据或案例
counter → 提供反方视角或争议点
global_search → 提供外部证据或跨站讨论
following → 提供关注作者视角
```

### 9.3 WorthReadingCard

```ts
WorthReadingCard {
  id: string
  categoryId: string
  tags: string[]
  title: string
  recommendationReason: string
  contentSummary: string
  controversies: string[]
  writingAngles: string[]
  originalSources: ContentSource[]
  relevanceScore: number
  authorityScore: number
  popularityScore: number
  controversyScore: number
  createdAt: string
}
```

## 10. LLM enrichment

### 10.1 调用策略

不对所有候选调用 LLM。

```text
每分类候选 30-60 条
规则选 Top 10 clusters
LLM enrich Top 5 cards
其余卡片保留基础摘要
```

### 10.2 任务链

```text
summarize-content
→ extract-controversies
→ generate-writing-angles
```

输入必须包含：

```text
card title
originalSources
interestMemory
globalMemory.contentPreference
globalMemory.riskReminder
```

### 10.3 enrichment 输出落点

```text
summary → card.contentSummary
keyPoints → card.recommendationReason
controversies → card.controversies
angles → card.writingAngles
```

### 10.4 LLM 失败降级

```text
contentSummary = primary.summary
controversies = rule_extract_questions(title/comments)
writingAngles = rule_generate_angles(categoryName, title)
recommendationReason = "该内容与 {categoryName} 相关，且具备较高讨论度。"
```

## 11. 卡片排序与多样性

### 11.1 cardScore

```text
cardScore =
  avg(itemScore) * 0.45
+ max(authorityScore) * 0.15
+ max(engagementScore) * 0.10
+ controversyScore * 0.10
+ sourceDiversityScore * 0.10
+ memoryFitScore * 0.10
```

### 11.2 MMR 重排

避免同一分类下都长得一样：

```text
finalScore = cardScore * 0.75 - similarityToSelected * 0.25
```

约束：

```text
同作者最多 1 张
同 hot topic 最多 2 张
纯 hot_list 来源最多 1 张
每批至少 1 张反方/争议卡
每批至少 1 张可写角度明确的卡
```

## 12. 刷新机制

用户点击“刷新输入”时不能简单重新展示同一批热榜。

### 12.1 状态

```ts
CategoryRefreshState {
  categoryId: string
  refreshCount: number
  refreshedAt: string
  visibleCardIds: string[]
  cursor: number
}
```

### 12.2 刷新流程

```text
1. 读取 content:cards candidate pool。
2. 排除 today shownCardIds。
3. 若 unshown >= pageSize，按 MMR 取下一批。
4. 若 unshown 不足：
   4.1 重新生成 Query Plan，增加 hot_expand / adjacent query。
   4.2 读取 Redis/回源 zhihu-adapter。
   4.3 增量合并候选池。
5. 返回新的 visibleCardIds、refreshCount、refreshedAt。
```

### 12.3 刷新不影响

- 种子库。
- 写作 session。
- 用户画像。
- 历史反馈。

### 12.4 刷新策略

| refreshCount | 策略 |
| --- | --- |
| 0 | 核心高相关卡 |
| 1 | 增加反方/争议卡 |
| 2 | 增加全网 evidence |
| 3+ | 增加偶遇/相邻探索 |

## 15. 实现优先级

### P0：从热榜固定流升级到多来源候选池

- Query Plan v1。
- 每兴趣小类 2 个 zhihu_search query。
- 偶遇输入使用 hot_list + global_search。
- 规则评分、去重、聚类。
- 每分类 Top 5 LLM enrichment。
- 刷新时从 candidate pool 取下一批。

### P1：个性化和发芽联动

- 利用 recent seeds 生成 query。
- 利用 question/material gaps 生成 evidence/counter query。
- 关注流 OAuth 接入。
- 记录 shownCardIds 和 user reactions。
- 今日发芽读取 content cards 作为 trigger。

### P2：高级探索

- LLM 生成 query expansion。
- 更精细的主题聚类。
- 多天趋势追踪。
- 基于反馈优化推荐权重。

## 16. 验收用例

1. 不同兴趣分类生成不同 Query Plan。
2. 同分类至少包含 zhihu_search 来源，不只 hot_list。
3. 偶遇输入能使用 global_search。
4. 热榜变化会影响 hot_expand query，但不会让所有卡片都来自热榜。
5. 每张卡至少有 1 个来源，优先 2-3 个来源。
6. 来源详情包含 `fullContent`，前端可卡内滚动展示。
7. 刷新同一分类时，visibleCardIds 变化。
8. 已展示卡片不会立即重复出现。
9. LLM 失败时仍能返回降级卡片。
10. Redis 命中时不增加知乎 API quota。
11. 额度耗尽时返回明确降级状态。
12. 关注流 OAuth 未配置时，关注流 tab 展示空态，不影响其他分类。
