# 看山小苗圃-当前前端接口文档

本文档整理当前 `frontend` 已经存在的所有前端接口、数据结构、mock API、页面操作契约，以及后续迁移到 `api-gateway` 时应落到的后端接口。它描述的是**当前前端真实依赖**，不是理想化后端设计。

参考代码：

- `frontend/lib/types.ts`
- `frontend/lib/api-client.ts`
- `frontend/lib/mock-data.ts`
- `frontend/app/api/mock/*/route.ts`
- `frontend/components/kanshan-app.tsx`

约定：

- 当前前端是 Next.js mock 闭环。
- 初始化数据来自 5 个 `GET /api/mock/*` 接口。
- 所有按钮产生的业务变更目前都在 React state + `localStorage` 中完成，没有发送 HTTP mutation。
- 后端接入时，前端应迁移为只访问 `api-gateway`，并从当前本地操作函数拆出正式接口。

## 1. 当前前端运行模型

### 1.1 初始化数据流

```text
KanshanApp mounted
↓
Promise.all([
  GET /api/mock/profile,
  GET /api/mock/content,
  GET /api/mock/seeds,
  GET /api/mock/sprout,
  GET /api/mock/feedback
])
↓
组装 DemoState initialState
↓
readStoredState(initialState)
↓
React state + localStorage 持久化
```

本地持久化 key：

```text
kanshan:nursery:demo-state:v1
```

`readStoredState` 的合并规则：

- `profile` 优先使用 localStorage 中的用户修改。
- `categories` 和 `cards` 始终使用 mock API 的最新结果，不使用 localStorage 覆盖。
- `seeds` 如果 localStorage 中存在，则使用本地版本，并执行 `ensureUniqueMaterialIds` 修复材料 id。
- `sproutOpportunities` 和 `feedbackArticles` 如果 localStorage 中存在，则使用本地版本。
- `reactions`、`expandedCardIds`、`expandedSourceIds`、`categoryRefreshState`、`sproutStarted` 从 localStorage 恢复。
- `writingSession` 通过展开 `stored` 保留，但没有在返回对象中显式兜底校验。

### 1.2 当前 HTTP API 客户端

文件：`frontend/lib/api-client.ts`

当前只提供 5 个 GET 方法：

```ts
fetchProfile(): Promise<ProfileData>
fetchContent(): Promise<{ categories: InputCategory[]; cards: WorthReadingCard[] }>
fetchSeeds(): Promise<IdeaSeed[]>
fetchSproutOpportunities(): Promise<SproutOpportunity[]>
fetchFeedbackArticles(): Promise<FeedbackArticle[]>
```

统一请求函数：

```ts
getJson<T>(path: string): Promise<T>
```

当前行为：

- 使用 `fetch(path, { cache: "no-store" })`。
- 非 2xx 抛出 `Mock API failed: {status} {path}`。
- 不处理统一响应 envelope。
- 不处理业务错误码。
- 不携带鉴权 token。

后端迁移注意：

- `api-gateway` 已规划统一响应 `{ request_id, data }`，前端 `api-client` 需要增加 unwrap 逻辑。
- 当前 mock 接口返回裸数据，和未来 gateway envelope 不一致。

## 2. 当前 Mock API 清单

### 2.1 GET /api/mock/profile

路由文件：`frontend/app/api/mock/profile/route.ts`

返回：`ProfileData`

数据来源：`frontend/lib/mock-data.ts` 中的 `profile`。

用途：

- 首屏用户身份信息。
- 个人画像页。
- 写作苗圃中的兴趣 Memory 注入。
- 今日看什么的兴趣说明和推荐策略展示。

未来映射：

```text
GET /api/v1/profile/me
```

路由文件：`frontend/app/api/mock/content/route.ts`

返回：

```ts

  categories: InputCategory[]
  cards: WorthReadingCard[]

```

用途：

- 今日看什么全部分类入口。

用途：

## 3. 前端核心 DTO

本节按 `frontend/lib/types.ts` 当前定义整理。后端 DTO 应尽量与这些字段兼容，或由 `api-client` 做清晰适配。

### 3.1 TabId

```ts
type TabId =
  | "today"
  | "seeds"
  | "sprout"
  | "write"
  | "history"
  | "profile"
  | "onboarding"
```

用途：主导航与进入状态。

### 3.2 InputCategory

```ts
interface InputCategory {
  id: string
  name: string
  meta: string
  kind: "interest" | "following" | "serendipity"
  active?: boolean
}
```

当前分类：

- `agent`: Agent 工程化
- `ai-coding`: AI Coding
- `rag`: RAG / 检索
- `backend`: 后端工程
- `growth`: 程序员成长
- `finance-risk`: 金融风控
- `medical-ai`: 医学 AI
- `product-design`: 产品设计
- `content-creation`: 内容创作
- `following`: 关注流精选
- `serendipity`: 偶遇输入

业务规则：

- `interest` 是用户兴趣小类。
- `following` 和 `serendipity` 与兴趣小类平级展示，但不是兴趣 Memory 的主分类。
- `profile.interestMemories` 只覆盖兴趣小类，不覆盖关注流和偶遇输入。

### 3.3 ContentSource

```ts
interface ContentSource {
  sourceId: string
  sourceType: string
  sourceUrl?: string
  title: string
  author?: string
  publishedAt?: string
  authorityMeta?: string
  meta: string[]
  rawExcerpt: string
  fullContent: string
  contribution: string
}
```

含义：内容卡片的一条原始来源。一个 `WorthReadingCard` 可以聚合多条来源。

当前展示规则：

- 不跳转外链。
- 点击来源后在当前卡片内展开 `InlineSourcePanel`。
- 同一张卡片一次只展开一个来源。
- 再次点击同一来源会收起。
- 来源面板展示 `sourceType`、作者、发布时间、权威/热度、`sourceId`、原文要点、完整内容、对当前卡片贡献。

真实 API 接入时的来源映射：

- 知乎热榜：`sourceType = hot_list`
- 知乎搜索：`sourceType = zhihu_search`
- 全网搜索：`sourceType = global_search`
- 关注流：`sourceType = following`
- 圈子帖子/评论：`sourceType = ring/comment`
- 直答结果：仅作为材料或摘要，不应伪装成原文来源。

### 3.4 WorthReadingCard

```ts
interface WorthReadingCard {
  id: string
  categoryId: string
  tags: { label: string; tone: Tone }[]
  title: string
  recommendationReason: string
  contentSummary: string
  controversies: string[]
  writingAngles: string[]
  originalSources: ContentSource[]
  relevanceScore: number
  authorityScore?: number
  popularityScore?: number
  controversyScore?: number
  createdAt: string
  featured?: boolean
}
```

含义：今日看什么页面的“值得看卡片”。

业务规则：

- 每个分类至少展示 2-3 张卡片。
- 当前分类第一张卡片默认完整展开。
- 非首卡默认小卡片，点击卡片主体或“展开卡片”后展示完整摘要与来源。
- `originalSources` 必须保留完整原始内容展示能力，不应只提供 URL。
- 用户可从卡片触发：总结、认同、反对、有疑问、加入种子库、基于它写一篇。

### 3.5 IdeaSeed

```ts
interface IdeaSeed {
  id: string
  interestId: string
  title: string
  interestName: string
  source: string
  sourceTitle: string
  sourceSummary: string
  sourceUrl?: string
  sourceType: string
  userReaction: SeedReaction
  userNote: string
  coreClaim: string
  possibleAngles: string[]
  counterArguments: string[]
  requiredMaterials: string[]
  wateringMaterials: WateringMaterial[]
  questions: SeedQuestion[]
  status: SeedStatus
  maturityScore: number
  activationScore?: number
  createdFromCardId?: string
  createdAt: string
  updatedAt: string
}
```

`SeedReaction`：

```ts
"agree" | "disagree" | "question" | "supplement" | "want_to_write" | "manual"
```

`SeedStatus`：

```ts
"dormant" | "water_needed" | "sproutable" | "high_timeliness" | "writing" | "published" | "expired"
```

业务规则：

- 从内容卡片生成种子时，`createdFromCardId` 作为幂等键。
- 同一卡片多次认同/反对/提问，不重复创建种子，而是更新同一颗种子的 `userReaction`、`userNote` 和材料。
- 手动新建种子 `sourceType = manual`。
- 写作中种子 `status = writing`。
- 模拟发布后种子 `status = published`、`maturityScore = 96`。

### 3.6 WateringMaterial

```ts
interface WateringMaterial {
  id: string
  type: "evidence" | "counterargument" | "personal_experience" | "open_question"
  title: string
  content: string
  sourceLabel?: string
  adopted: boolean
  createdAt: string
}
```

四格材料板：

- `evidence`: 事实证据，来源、数据、案例、摘要。
- `counterargument`: 反方质疑，反对意见、漏洞、边界。
- `personal_experience`: 个人经验，项目经历、观察、踩坑。
- `open_question`: 待解决问题，尚未回答的疑问。

业务规则：

- `adopted = true` 的材料计入成熟度核心类型数。
- `open_question` 默认不采纳。
- 待解决问题可点击“问 Agent”，生成一条 `SeedQuestion`，同时把问题转为已采纳并新增事实证据材料。
- `evidence` 和 `counterargument` 支持 Agent 补充。

### 3.7 SeedQuestion

```ts
interface SeedQuestion {
  id: string
  question: string
  agentAnswer: string
  citedSourceIds: string[]
  status: "answered" | "resolved" | "needs_material"
  createdAt: string
}
```

当前能力：

- 点击“有疑问”打开弹窗。
- 用户输入问题后，mock Agent 生成回答。
- 回答写入 `SeedQuestion`。
- 问题同时写入 `open_question` 材料。
- 回答同时写入 `evidence` 或 `counterargument` 材料。
- 弹窗内支持多轮追问。
- 标记“已解决”后当前追问流程结束。
- 标记“仍需补资料”后自动填入下一轮追问建议，允许继续回答并记录。

当前结构缺口：

- 当前 `SeedQuestion` 没有 `threadId`、`parentQuestionId`、`turnIndex`。
- 多轮追问在 UI 上是 thread，但落库结构仍是平铺问题列表。
- 后端实现应扩展为线程结构，否则无法稳定还原多轮上下文。

建议后端增强字段：

```ts
threadId?: string
parentQuestionId?: string
turnIndex?: number
sourceCardId?: string
materialIds?: string[]
```

### 3.8 SproutOpportunity

```ts
interface SproutOpportunity {
  id: string
  seedId: string
  interestId: string
  score: number
  tags: { label: string; tone: Tone }[]
  activatedSeed: string
  triggerTopic: string
  whyWorthWriting: string
  suggestedTitle: string
  suggestedAngle: string
  suggestedMaterials: string
  status?: "new" | "supplemented" | "angle_changed" | "dismissed" | "writing"
}
```

业务规则：

- 今日发芽由用户主动点击触发。
- 机会来源于历史种子、今日热点、搜索结果、关注流和兴趣 Memory 的匹配。
- “补充资料”会把 `suggestedMaterials` 写回对应种子的材料板。
- “换个角度”会更新 `suggestedTitle` 和 `suggestedAngle`。
- “暂时不写”会把 `status` 改为 `dismissed`，页面隐藏。
- “开始写作”会创建写作 session。

```ts
interface WritingSession {
```

## 4. 当前页面操作接口

本节把 `KanshanApp` 中的本地函数按未来后端接口理解整理。

### 4.1 登录与导航

#### enterApp(mode)

```ts
enterApp(mode: "zhihu" | "onboarding" | "demo"): void
```

行为：

- `zhihu`: 跳转到 `${NEXT_PUBLIC_ZHIHU_ADAPTER_URL || "http://127.0.0.1:8070"}/zhihu/oauth/authorize?redirect=true`
- `onboarding`: 进入画像采集页。
- `demo`: 进入今日看什么。

当前缺口：

- `zhihu-adapter` 需要实现前端期望的 `/zhihu/oauth/authorize?redirect=true`。
- OAuth 回调后需要把用户状态写回 profile 或 session。

#### goTab(tab)

```ts
goTab(tab: TabId): void
```

行为：切换主页面，写入 `DemoState.activeTab`。

#### resetDemoState()

```ts
resetDemoState(): void
```

行为：删除 `localStorage[kanshan:nursery:demo-state:v1]` 并 reload。

后端接入后：仍应保留为前端演示/调试功能，不调用后端删除真实数据。

### 4.2 今日看什么

#### selectCategory(categoryId)

```ts
selectCategory(categoryId: string): void
```

行为：切换当前分类，写入 `selectedCategoryId`。

未来接口：

```text
GET /api/v1/content/cards?category_id={categoryId}
```

#### refreshSelectedCategory()

```ts
refreshSelectedCategory(): void
```

当前输入：

- `selectedCategory`
- 当前分类下所有 `WorthReadingCard`
- `categoryRefreshState[selectedCategory]`

当前计算：

```text
nextCount = previous.refreshCount + 1
baseIds = previous.visibleCardIds 或当前分类 card ids
rotated = rotateIds(baseIds, 1)
categoryRefreshState[selectedCategory] = {
  refreshCount: nextCount,
  refreshedAt: now(),
  visibleCardIds: rotated
}
expandedSourceIds = {}
```

未来接口：

```text
POST /api/v1/content/categories/{categoryId}/refresh
```

建议请求：

```json
{
  "profile_id": "me",
  "refresh_mode": "rotate_or_refetch",
  "exclude_card_ids": ["card_xxx"],
  "preserve_user_state": true
}
```

建议响应：

```json
{
  "category_refresh_state": {
    "refresh_count": 2,
    "refreshed_at": "2026-05-11T10:00:00+08:00",
    "visible_card_ids": ["card_b", "card_c", "card_a"]
  },
  "cards": []
}
```

规则：刷新只影响当前分类输入流，不影响种子、写作、画像和历史反馈。

#### toggleSource(cardId, sourceId)

```ts
toggleSource(cardId: string, sourceId: string): void
```

行为：更新 `expandedSourceIds[cardId]`。

规则：

- 同一卡片一次只展开一个来源。
- 点击已展开来源时设置为空字符串，表示收起。

未来接口：

- 如果 `fullContent` 已在卡片返回中提供，则不需要额外接口。
- 如果完整原文需要按需加载，则使用：

```text
GET /api/v1/content/cards/{cardId}/sources/{sourceId}
```

#### onToggleSummary(cardId)

```ts
onToggleSummary(cardId: string): void
```

行为：把 `cardId` 加入或移出 `expandedCardIds`。

规则：

- 首卡 `featured` 始终完整展开。
- 非首卡点击卡片主体、按 Enter/Space、点击“展开卡片”均触发该接口。

## 5. 当前本地算法接口

### 5.1 从卡片生成种子：buildSeedFromCard

输入：

```ts
card: WorthReadingCard
categories: InputCategory[]
reaction: SeedReaction
note?: string
id: string
```

输出：`IdeaSeed`

映射规则：

```text
interestId      = card.categoryId
interestName    = category.name
source          = card.tags.label join(" / ")
sourceTitle     = card.title
sourceSummary   = card.contentSummary
sourceUrl       = card.originalSources[0].sourceUrl
sourceType      = card.originalSources[0].sourceType
userReaction    = reaction
userNote        = note 或 reactionLabel(reaction)
coreClaim       = card.writingAngles[0] 或 card.title
possibleAngles  = card.writingAngles
counterArguments = card.controversies
requiredMaterials = [补充个人经验, 补充反方回应, 补充可引用来源]
createdFromCardId = card.id
```

初始材料：

- `evidence`: 第一条来源 `rawExcerpt`，`adopted = true`
- `counterargument`: 第一条争议，`adopted = reaction === disagree`

然后执行 `recalcSeed`。

### 5.2 成熟度计算：recalcSeed

当前公式：

```text
adoptedTypes = adopted wateringMaterials 的 type 去重
resolvedQuestions = status == resolved 的问题数量
score = min(96, 32 + adoptedTypes.size * 14 + questions.length * 3 + resolvedQuestions * 4)
```

状态规则：

```text
如果原状态是 published 或 writing，则保持不变
否则 score >= 70 → sproutable
否则 → water_needed
```

后端实现要求：

- 所有新增/修改/删除材料后必须重新计算。
- 所有新增/更新问题后必须重新计算。
- 不能让前端自己提交 `maturityScore` 作为可信值。

### 5.3 Agent 问答 mock：buildAgentAnswer

输入：

```ts
card: WorthReadingCard
question: string
turnIndex: number
```

当前逻辑：

- 第 0 轮：根据所有来源类型、第一条写作角度、第一条争议生成初步回答。
- 奇数轮：从 `card.originalSources[turnIndex % sources.length]` 补证据。
- 偶数非 0 轮：补反方边界。

后端应替换为：

```text
llm-service /answer-seed-question
```

输入上下文必须包含：

- 用户问题
- 关联卡片标题、摘要、争议点、写作角度
- 原始来源 fullContent / rawExcerpt
- 当前 seed 的 coreClaim、materials、questions thread
- 用户兴趣 Memory

### 5.4 今日发芽机会：buildSproutOpportunity

当前公式：

```text
score = min(94, seed.maturityScore + 10 + index * 3)
```

当前输出：

- `id = sprout-{seed.id}`
- `triggerTopic = {interestName} 今日出现新讨论：{sourceTitle}`
- `suggestedTitle = seed.possibleAngles[0]`
- `suggestedAngle = seed.possibleAngles[1] 或 seed.coreClaim`
- `suggestedMaterials = seed.requiredMaterials[0] 或默认建议`

后端正式算法见 `docs/看山小苗圃-算法与模块详细设计.md`，但前端至少需要上述字段齐全。

### 5.5 写作蓝图：buildWritingBlueprint

输入：

```ts
seed: IdeaSeed
memory: MemorySummary
```

当前逻辑：

- 只使用 `adopted = true` 的材料。
- 按材料类型拆为 evidence、counterargument、personal_experience。
- 用 `memory.preferredPerspective[0]` 作为文章视角。
- 生成 4 个分论点：
  1. 界定讨论范围
  2. 用已采纳材料证明为什么成立
  3. 回应反方和边界条件
  4. 补入个人经验

后端写作服务应把这个规则升级为结构化 LLM 输出，但仍保持四段式可解释骨架。

### 5.6 内容分类刷新：cardsForCategory

输入：

```ts
cards: WorthReadingCard[]
categoryId: string
refreshState?: CategoryRefreshState
```

当前逻辑：

- 先过滤 `card.categoryId === categoryId`。
- 如果有 `visibleCardIds`，按该顺序排序。
- 如果 `refreshCount > 0`：
  - `relevanceScore = min(99, relevanceScore + refreshCount + index)`
  - tags 移除旧 `刷新` 标签，加入 `刷新 {refreshCount}`。

未来 content-service 实现时，刷新可以从知乎 API/缓存重新取数，但必须保留“不破坏用户状态”的语义。

| 生成二次文章 | local seed create | `POST /api/v1/feedback/articles/{articleId}/second-seed` | feedback-service + seed-service |

## 7. 后端实现必须满足的前端不变量

### 7.1 内容卡片不变量

- `WorthReadingCard.originalSources` 不能为空。
- 每条 `ContentSource.fullContent` 必须可展示；如果无法一次性返回，必须支持按 source 拉取详情。
- `card.id` 必须稳定，否则 localStorage 中的 reactions、expanded 状态、createdFromCardId 会失效。
- `categoryId` 必须能匹配 `InputCategory.id`。

### 7.2 种子不变量

- 从同一 `card.id` 生成种子必须幂等。
- `maturityScore` 由后端计算，不信任前端提交。
- `wateringMaterials.id` 必须全局或 seed 内唯一。
- `questions.id` 必须唯一。
- `status = writing/published` 不应被普通材料更新回滚成 `water_needed/sproutable`。

### 7.3 疑问线程不变量

- “让 Agent 回答并记录”可以多次触发。
- “已解决”代表当前线程结束。
- “仍需补资料”代表继续追问时应携带历史上下文。
- 每轮回答都要能回写材料板。
- 回答引用的 `citedSourceIds` 必须能在来源列表中找到，或者标明是外部补充来源。

### 7.4 写作 Memory 不变量

- 写作 session 必须展示使用了哪个兴趣 Memory。
- 用户修改本次写作 Memory 不应覆盖全局画像。
- `memoryOverride.interestId` 应与 `session.interestId` 一致。
- 重置 Memory 应恢复 `profile.interestMemories[interestId]`。

### 7.5 历史反馈不变量

- 历史反馈页不直接更新 Memory。
- `memoryAction` 只是候选建议或摘要说明。
- 二次文章应进入种子库，而不是直接进入写作。

## 8. 当前已知前端接口风险

### 8.1 selectSeed 递归风险

当前 `selectSeed(seedId)` 内部调用了自身：

```ts
function selectSeed(seedId: string) {
  selectSeed(seedId)
  updateData(...)
}
```

这会在点击种子或某些选中种子操作时造成递归调用风险。本文档只记录问题，不在本轮修改代码。后续修复应改为：

```ts
setSelectedSeedId(seedId)
updateData(...)
```

### 8.2 Mock API 与 Gateway envelope 不一致

当前 mock 返回裸数据；后端服务文档规划的是：

```json
{ "request_id": "...", "data": {} }
```

迁移时应先改 `api-client`，统一 unwrap。

### 8.3 mutation 全部是本地状态

当前所有按钮操作都没有 HTTP 请求。后端开发时不要只实现 5 个 GET，否则页面交互无法落库。至少要补齐：

- seed from-card
- question answer/mark
- material CRUD
- writing session patch
- sprout opportunity update
- feedback second-seed
- profile save

### 8.4 SeedQuestion 缺少线程字段

UI 已支持多轮追问，但 DTO 只有平铺问题列表。正式后端需要补 `threadId` 和 `turnIndex`，否则“仍需补材料后继续追问”的上下文无法稳定保存。

### 8.5 OAuth 前端入口依赖 adapter 路由

“关联知乎账号”会跳转：

```text
{NEXT_PUBLIC_ZHIHU_ADAPTER_URL}/zhihu/oauth/authorize?redirect=true
```

后端需要实现该路由，或者前端改为 gateway 的 OAuth 入口。

### 8.6 写作步骤只有本地生成

当前初稿、蓝图、圆桌审稿、定稿均是本地模板生成。正式写作服务需要补：

- 结构化蓝图生成
- 初稿生成
- 圆桌审稿 Agent
- 定稿生成
- AI 辅助创作声明提示

### 8.7 历史反馈没有真实知乎评论数据

当前 `syncFeedback` 只是本地 metrics 增量。正式反馈服务需要通过：

- `zhihu-adapter` 拉评论列表
- `llm-service` 汇总支持、反方、补充材料
- 写入反馈文章和候选二次种子

## 11. 最小验收清单

迁移到后端接口后，浏览器应逐项验证：

1. 打开页面能加载 profile/content/seeds/sprout/feedback。
2. 切换全部兴趣小类、关注流、偶遇输入都有卡片。
3. 每张卡片来源可展开完整内容。
4. 刷新当前分类后，卡片顺序或标签变化，种子和写作状态不丢。
5. 点击认同/反对后，种子库出现或更新对应种子。
6. 点击有疑问，多轮回答能记录到问题列表和材料板。
7. 标记已解决后，当前追问流程结束。
8. 标记仍需补资料后，可继续追问。

10. Agent 补证据/找反方能写入材料并更新成熟度。
11. 合并相似种子后，来源、材料、疑问不丢。
12. 今日发芽能生成机会，补充资料能回写种子材料。
13. 开始写作能注入对应兴趣 Memory，且用户可修改本次 Memory。
14. 写作流程按钮能改变 session 状态。
15. 模拟发布后历史反馈新增文章，种子变为已发布。
16. 历史反馈能生成二次文章种子。
17. 刷新页面后关键状态可恢复，或后端可重新拉取。
