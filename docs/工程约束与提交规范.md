# 工程约束与提交规范

## 1. 总原则

这个项目由一个主开发者维护，并借助多个 AI coding 工具并行开发。

因此工程约束要服务于三件事：

1. 可监控：每个模块边界清楚，改动可定位。
2. 可回滚：每个小版本都有 Git 提交。
3. 可交接：每个服务都有 README、接口契约和 mock 数据。

## 2. 目录边界

不同任务只允许修改自己的目录。

示例：

```text
profile-service 任务只能修改：
- services/profile-service/**
- packages/shared-schemas/profile/**
- docs 中明确要求同步的部分
```

除非任务明确要求，不要跨服务修改。

## 3. 服务约束

所有后端服务遵守：

- FastAPI 为默认框架。
- 每个服务必须有 `/health`。
- P0 必须支持 mock 数据。
- 响应结构必须可序列化为 JSON。
- 错误响应使用统一格式。
- 不直接读写其他服务拥有的数据表。

统一错误格式：

```json
{
  "request_id": "req_demo",
  "error": {
    "code": "ERROR_CODE",
    "message": "用户可读错误",
    "detail": "调试信息"
  }
}
```

## 4. Memory 约束

P0 阶段 Memory 在 `profile-service/memory` 内维护。

约束：

- Memory 对用户可见、可编辑。
- Memory 更新不能静默覆盖。
- 历史反馈只能生成更新建议。
- 用户确认后才能写入 Memory。
- 写作苗圃必须展示本次注入的兴趣分类 Memory。

## 5. LLM 约束

所有 LLM 调用必须经过 `llm-service`。

要求：

- Prompt 模板集中管理。
- 输出优先 JSON schema。
- 结果需要缓存。
- 失败时提供 mock / fallback。
- 不在业务服务里散落模型调用。

## 6. 知乎接口约束

所有知乎开放接口必须经过 `zhihu-adapter`。

要求：

- 支持 `mock | cached | live`。
- 默认不开 live。
- 所有 live 结果写缓存。
- token 不进入前端。
- 额度敏感接口必须限流。

## 7. Git 提交规范

建议提交格式：

```text
docs: update product rules
scaffold: add service skeletons
feat(profile): add memory schema
feat(content): add today cards mock
fix(writing): correct roundtable state
```

每次提交尽量只包含一个目标。

不要把 `.DS_Store`、`.env`、日志、缓存和构建产物提交进仓库。

## 8. 小版本定义

一个小版本应该具备：

- 能解释为什么改。
- 能独立运行或独立验证。
- 能通过 Git 一步回滚。

不建议把“目录创建、接口实现、UI 重构、Prompt 修改”混在同一个提交里。
