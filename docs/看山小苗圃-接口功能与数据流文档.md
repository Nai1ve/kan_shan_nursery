# 看山小苗圃-接口功能与数据流文档

本文档用于把当前前端 mock 逻辑、知乎开放 API、Redis 短缓存和后端服务边界整理成可实现契约。目标是让不同 AI Coding 模型可以按模块实现，不需要自行判断数据从哪里来、如何计算、写到哪里去。

参考资料：

- `docs/知乎API.md`
- `docs/看山小苗圃-产品文档.md`
- `docs/看山小苗圃-技术文档.md`
- `docs/看山小苗圃-服务拆分与开发协作.md`

## 1. 总体原则

### 1.1 调用边界

```text
web-frontend
↓
api-gateway
↓
profile-service / content-service / seed-service / sprout-service / writing-service / feedback-service
↓
zhihu-adapter / llm-service
↓
Redis short cache
↓
Zhihu API / Zhihu Data Platform / Zhida Agent
```

规则：

- 前端只调用 `api-gateway`。
- 业务服务不能直接调用知乎接口，必须通过 `zhihu-adapter`。
- 业务服务不能直接调用直答 Agent，必须通过 `llm-service`。
- 前端不能接触 `app_secret`、`access_secret`、OAuth `access_token`、Redis。
- P0 只使用 Redis 短缓存，不设计 Postgres 缓存表，不设计独立 `cache-service`。

### 1.2 响应与错误

内部服务统一响应建议：

```json
{
  "request_id": "req_xxx",
  "data": {}
}
```

统一错误：

```json
{
  "request_id": "req_xxx",
  "error": {
    "code": "ZHIHU_QUOTA_EXCEEDED",
    "message": "知乎热榜今日额度已用完",
    "detail": "hot_list limit: 100/day"
  }
}
```

`zhihu-adapter` 负责把知乎不同响应格式归一化：

- 社区 API 多数返回 `status/msg/data`。
- `comment/create` 示例返回 `code/msg/data`。
- 数据开放平台返回 `Code/Message/Data`。
- 直答 API 返回 OpenAI-compatible `chat.completion` 结构或 `error`。

## 2. Zhihu Adapter 客户端拆分

`zhihu-adapter` 物理上可以是一个服务，但代码上必须拆成三类 client，避免鉴权和限流逻辑混在一起。

### 2.1 Community Client

用途：

- 圈子详情
- 发布想法
- 评论列表
- 创建评论
- 删除评论
- 点赞 / 取消点赞
- 故事列表与详情

Base URL：

```text
https://openapi.zhihu.com
```

鉴权：

```text
sign_string = app_key:{app_key}|ts:{timestamp}|logid:{log_id}|extra_info:{extra_info}
X-Sign = Base64(HMAC-SHA256(app_secret, sign_string))
```

请求头：

```text
X-App-Key: 用户 token
X-Timestamp: 秒级 Unix 时间戳
X-Log-Id: 请求日志 ID
X-Sign: 签名
X-Extra-Info: 可为空
Content-Type: application/json，仅 POST 需要
```

限制：

- 社区 API 全局 10 QPS。
- 写操作必须由用户显式触发，不允许 Agent 批量自动发布、评论或点赞。
- `comment/create` 每小时每个想法下最多 20 条。

### 2.2 OAuth Client

用途：

- 用户信息
- 粉丝列表
- 关注列表
- 关注动态

Base URL：

```text
https://openapi.zhihu.com
```

鉴权：

```text
Authorization: Bearer {access_token}
```

OAuth 流程：

```text
GET /authorize?redirect_uri={redirect_uri}&app_id={app_id}&response_type=code
POST /access_token
GET /user
GET /user/followers
GET /user/followed
GET /user/moments
```

### 2.3 Data Platform Client

用途：

- 知乎热榜
- 知乎搜索
- 全网搜索
- 直答 Agent

Base URL：

```text
https://developer.zhihu.com
```

鉴权：

```text
Authorization: Bearer {access_secret}
X-Request-Timestamp: 秒级 Unix 时间戳
```

## 3. Redis 短缓存

### 3.1 读取规则

```text
1. 生成 cache key。
2. 查询 Redis。
3. 命中：直接返回，cache.hit=true，不扣额度。
4. 未命中：检查 quota key。
5. 额度不足：返回明确错误，不强制回源。
6. 额度允许：调用真实 API。
7. 回源成功：写 Redis，增加额度计数。
8. 回源失败：不扣额度，返回错误；P0 演示环境可返回 fallback mock，但必须标记 fallback=true。
```

### 3.2 Cache Key

```text
zhihu:hot_list:{limit}
zhihu:zhihu_search:{query_hash}:{count}
zhihu:global_search:{query_hash}:{count}
zhihu:direct_answer:{model}:{messages_hash}:stream_false
zhihu:ring_detail:{ring_id}:{page_num}:{page_size}
zhihu:comment_list:{content_type}:{content_token}
zhihu:story_list
zhihu:story_detail:{work_id}
zhihu:user_moments:{user_id}
zhihu:user_followed:{user_id}:{page}:{per_page}
zhihu:user_followers:{user_id}:{page}:{per_page}
quota:{endpoint}:{user_id}:{yyyyMMdd}
```

`query_hash` 和 `messages_hash` 使用规范化 JSON 后的 SHA-256，避免中文、空格和参数顺序导致重复缓存。

### 3.3 TTL

| 数据 | TTL |
| --- | --- |
| 热榜 | 15 分钟 |
| 知乎搜索 | 1 小时 |
| 全网搜索 | 1 小时 |
| 直答 Agent | 6 小时 |
| 圈子详情 | 10 分钟 |
| 评论列表 | 10 分钟 |
| 故事列表 / 故事详情 | 6 小时 |
| 关注动态 | 10 分钟 |
| 关注 / 粉丝列表 | 30 分钟 |
| 今日发芽结果 | 当天有效 |

### 3.4 默认额度

按比赛描述配置，最终以平台分配为准：

| endpoint | 默认额度 |
| --- | --- |
| `hot_list` | 100 / day / user |
| `zhihu_search` | 1000 / day / user |
| `global_search` | 1000 / day / user |
| `direct_answer` | 100 / day / user |

社区 API 没有日额度说明，但有 10 QPS 和社区治理限制；写操作必须额外做业务节流。

## 4. 标准内部 DTO

业务服务只依赖标准 DTO，不依赖官方 raw 字段。

```ts
interface ZhihuContentItem {
  sourceType:
    | "hot_list"
    | "zhihu_search"
    | "global_search"
    | "following"
    | "ring"
    | "comment"
    | "story"
    | "direct_answer";

  sourceId: string;
  contentType?: string;
  title: string;
  url?: string;
  author?: string;
  authorAvatar?: string;
  authorBadgeText?: string;
  publishedAt?: string;

  summary: string;
  rawExcerptHtml?: string;
  fullContent?: string;
  thumbnailUrl?: string;
  tags?: string[];

  heatScore?: number;
  relevanceScore?: number;
  rankingScore?: number;
  authorityLevel?: string;
  likeCount?: number;
  commentCount?: number;
  favoriteCount?: number;
  shareCount?: number;

  selectedComments?: string[];
  searchHashId?: string;
  usageNotice?: string;
  raw: unknown;
}
```

`publishedAt` 在内部统一转换成 ISO 字符串。官方 Unix 时间戳保留在 `raw` 中。

## 5. Data Platform API 映射

### 5.1 热榜

官方接口：

```text
GET https://developer.zhihu.com/api/v1/content/hot_list
```

请求参数：

```ts
{
  Limit?: number; // 默认 30，最大 30；<=0 或 >30 时服务端回退为 30
}
```

响应：

```text
Data.Total
Data.Items[]
```

字段映射：

| 官方字段 | 内部字段 |
| --- | --- |
| `Item.Title` | `title` |
| `Item.Url` | `url` |
| `Item.ThumbnailUrl` | `thumbnailUrl` |
| `Item.Summary` | `summary` |

派生字段：

```text
sourceId = sha256(Item.Url)
sourceType = hot_list
contentType = 根据 Url 推断 question / article
heatScore = max(1, 100 - index * 2)
```

用途：

- 今日看什么的热点输入。
- 今日发芽的时效触发信号。
- 写作苗圃里的热点背景材料。

### 5.4 直答 Agent

官方接口：

```text
POST https://developer.zhihu.com/v1/chat/completions
```

请求：

```ts
{
  model: "zhida-fast-1p5" | "zhida-thinking-1p5" | "zhida-agent";
  messages: {
    role: string;
    content: string;
  }[];
  stream?: boolean;
}
```

P0 默认：

```text
stream = false
```

非流式响应映射：

| 官方字段 | 内部字段 |
| --- | --- |
| `id` | `taskId` |
| `model` | `model` |
| `choices[0].message.content` | `content` |
| `choices[0].message.reasoning_content` | `reasoningContent` |
| `choices[0].finish_reason` | `finishReason` |

错误映射：

| 官方错误 | 内部错误 |
| --- | --- |
| `missing_required_parameter` | `INVALID_LLM_REQUEST` |
| `model_not_found` | `LLM_MODEL_NOT_AVAILABLE` |
| stream 中途 `finish_reason=error` | `LLM_STREAM_INTERRUPTED` |

用途：

- 有疑问：回答用户问题并回写浇水材料。
- 继续浇水：Agent 补证据 / 找反方。
- 内容卡片：摘要、争议点、可写角度。
- 今日发芽：生成机会解释和写作角度。
- 写作苗圃：论证蓝图、草稿、圆桌审稿。

## 9. 今日看什么数据流

### 9.1 兴趣小类

输入：

```text
兴趣 Memory
全局 Memory
知乎热榜
知乎搜索
全网搜索
可选关注动态
```

流程：

```text
1. profile-service 返回用户兴趣 Memory。
2. content-service 生成 Query Plan。
3. content-service 请求 zhihu-adapter。
4. zhihu-adapter 先查 Redis，miss 后回源。
5. content-service 聚合 2-3 个 ZhihuContentItem 为 WorthReadingCard。
6. llm-service 用直答 Agent 生成摘要、争议点、可写角度。
7. content-service 返回当前分类内容卡片。
```

Query Plan 示例：

```ts
{
  interestId: "ai-coding",
  keywords: ["AI Coding", "程序员成长", "研发工作流"],
  sources: ["hot_list", "zhihu_search", "global_search"],
  count: 10
}
```

### 9.2 关注流

输入：

```text
OAuth /user/moments
OAuth /user/followed
OAuth /user/followers
兴趣 Memory
```

规则：

- 关注流作为独立分类，与兴趣小类平级。
- 关注流重点看观点密度、作者关系、能否补充已有种子。
- 不把关注流内容混入兴趣小类列表。

### 9.3 偶遇输入

输入：

```text
热榜
全网搜索
故事 API
与用户兴趣相邻但不完全重合的主题
```

规则：

- 目标是打破信息茧房，不是随机推荐。
- `novelty` 权重高于兴趣小类，但仍需和用户长期方向存在可解释关联。
- 故事内容必须展示使用限制和署名要求。

### 9.4 卡片评分

```text
relevanceScore =
兴趣匹配 35%
+ authorityLevel / rankingScore 20%
+ VoteUpCount / CommentCount 15%
+ 争议点数量 15%
+ EditTime / 热榜排序 10%
+ 可写角度质量 5%
```

落点：

```text
content_cards
content_sources
Redis: zhihu:* / llm:*
前端今日看什么页面
```

## 11. 观点种子与浇水

### 11.1 从卡片生成种子

输入：

```text
WorthReadingCard
用户 reaction
用户 note
```

规则：

```text
title = card.writingAngles[0] || card.title
coreClaim = card.writingAngles[0] || card.title
sourceSummary = card.contentSummary
possibleAngles = card.writingAngles
counterArguments = card.controversies
requiredMaterials = ["补充个人经验", "补充反方回应", "补充可引用来源"]
```

初始材料：

```text
evidence:
  card.originalSources[0].summary 或 fullContent 摘录
  adopted = true

counterargument:
  card.controversies[0]
  reaction=disagree 时 adopted = true
```

落点：

```text
idea_seeds
seed_sources
seed_materials
```

### 11.2 有疑问

输入：

```text
用户 question
seed
card.originalSources
历史 question thread
```

流程：

```text
1. seed-service 创建或找到 card 对应 seed。
2. seed-service 写入 SeedQuestion。
3. llm-service 调 direct_answer，P0 用 zhida-thinking-1p5。
4. 回答写入 evidence 或 counterargument material。
5. 原问题写入 open_question material。
6. 重新计算 maturityScore。
```

状态：

```text
answered：Agent 已回答，等待用户判断
resolved：用户标记已解决，该问题线程结束
needs_material：仍需补材料，下次追问带 parentQuestionId
```

### 11.3 成熟度

```text
maturityScore =
```

## 15. AI Coding 任务拆分

### 15.1 zhihu-adapter

必须实现：

- Community Client HMAC 签名。
- OAuth Client Bearer token 调用。
- Data Platform Client Bearer access_secret 调用。
- Redis cache wrapper。
- quota counter。
- 官方字段到 `ZhihuContentItem` 映射。

验收：

- 热榜、知乎搜索、全网搜索、直答、圈子详情、故事详情、关注动态都有 mock/live 两种 provider。
- 相同请求第二次命中 Redis。
- token 和 secret 不出现在前端。

### 15.2 content-service

必须实现：

- Query Plan 生成。
- 多来源聚合为 WorthReadingCard。
- 卡片评分。
- 当前分类刷新。
- 来源全文展示数据。

验收：

- 兴趣小类、关注流、偶遇输入都能返回卡片。
- 每张卡片至少 1 个来源，重点卡片支持 2-3 个来源。

### 15.3 seed-service

必须实现：

- 从卡片创建或更新 seed。
- question thread。
- watering materials。
- maturityScore。
- merge seeds。

验收：

- 认同、反对、有疑问、加入种子库、基于它写一篇都能落到 seed 状态。

### 15.4 sprout-service

必须实现：

- 用户主动触发发芽。
- activationScore。
- Redis 当日缓存。
- 补充资料、换角度、暂时不写、开始写作。

### 15.5 writing-service

必须实现：

- WritingSession。
- Memory 注入和 `memoryOverride`。
- 蓝图、草稿、圆桌审稿、定稿、模拟发布。

### 15.6 feedback-service

必须实现：

- 发布文章反馈列表。
- 评论摘要。
- 二次文章种子。
- Memory 更新建议，不直接覆盖 Memory。

## 16. 验收用例

- 热榜：`Limit > 30` 时按 30 处理；无 ID 时用 URL hash 生成 `sourceId`。
- 知乎搜索：`Count > 10` 时截断；`RankingScore` 映射为 `relevanceScore`。
- 全网搜索：`Count > 20` 时截断；`ContentText` 去除 `<em>` 后生成 `summary`。
- 直答：`stream=false` 返回 `content` 和 `reasoningContent`。
- Redis：相同 Query 第二次命中缓存，不增加 quota。
- 额度：hot/search/global/direct 达到当日限制后禁止回源。
- 内容卡片：同一兴趣小类能从热榜、知乎搜索、全网搜索聚合成多来源卡片。
- 有疑问：直答结果能写入 `SeedQuestion` 和浇水材料。
- 发芽：热榜缓存变化能影响 `activationScore`。
- 写作：Memory 注入可见且可编辑，不覆盖 profile Memory。
- 故事：故事内容进入偶遇输入时必须带署名和活动限制。
