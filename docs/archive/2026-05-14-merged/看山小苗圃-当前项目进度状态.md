### content-service

目录：

```text
services/content-service/
```

当前状态（2026-05-13 第二轮更新）：

核心能力：

- 已有 content bootstrap、cards、card detail、source detail、category refresh、summarize、enrich 接口。
- 已实现 UserProfileSnapshot 机制：后台同步用户画像到 PostgreSQL + Redis，请求时本地选卡。
- 系统级内容池与用户无关，用户请求时通过 snapshot 中的 interest_ids 和 interestMemory 现场评分选卡。
- 已展示卡片跟踪从全局 Redis set 改为 per-user 的 snapshot.shown_card_ids + DB 表。
- api-gateway 已透传 user_id 到 content-service。

关注流：

- zhihu-adapter 的 following-feed / user-followed / user-followers 接口已完整实现（mock + live）。
- profile-service 新增 `GET /internal/auth/zhihu-token?user_id=xxx` 内部接口，暴露 access_token 给 content-service。
- content-service 后台预取关注流数据，转为 WorthReadingCard 并缓存到系统内容池。
- 关注流分类不再返回空列表，meta 不再显示"需知乎关联"。

LLM 富化：

- enricher 输入格式已修复，匹配 llm-service prompt 期望的结构化 JSON（sources 数组、userProfile 对象）。
- 三个 LLM 任务链：summarize-content → extract-controversies → generate-writing-angles。
- 支持按需富化：`POST /content/cards/{card_id}/enrich`，前端展开卡片时触发。

渐进加载（抖音模式）：

- bootstrap 只对第 1 张卡片执行 LLM 完整富化，返回 `enriched: true`。
- 第 2、3 张卡片返回基础数据（title、sources），标记 `enriched: false`。
- 前端展开未富化卡片时，调用 enrich 端点，富化完成后更新卡片状态。
- 富化结果写回 Redis 缓存，同一张卡片只富化一次。

已实现的核心流程：

```text
后台: profile-service → UserProfileSnapshot → PostgreSQL + Redis
后台: zhihu-adapter hot_list + search + following_feed → 系统内容池 Redis 缓存
请求: GET /content → 读 snapshot → 过滤内容池 → 评分 → Top 3 → 第1张 LLM 富化 → 返回
前端: 展开第2/3张卡片 → POST /enrich → 按需 LLM 富化 → 更新缓存 → 渲染
刷新: POST /content/categories/{id}/refresh {user_id, exclude_ids} → 排除已展示 → 下一批 3 张
```

下一步重点：

- P1: 快照同步频率从 24h 提升到 15min 独立循环。
- P1: 引入动态 QueryPlan 替代 category_queries 默认查询。
- P2: 偶遇输入增加 global_search、MMR 多样性重排、多来源 Jaccard 聚类。
