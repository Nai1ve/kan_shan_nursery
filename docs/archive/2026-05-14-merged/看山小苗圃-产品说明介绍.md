# 看山小苗圃-产品说明介绍

**项目定位**：看山小苗圃是一个面向知乎生态的**读写一体创作 Agent**，帮助用户在信息爆炸中**筛选高质量输入**，**沉淀个人观点**，并在 AI 协助下完成**更有主体性的内容表达**。

## 一、 痛点是什么？

AI 正在降低内容生产的门槛，但这并不意味着高质量内容会自然变多。相反，在之前就存在的信息过载和生成式 AI 普及之后，创作者和阅读者同时面临一个更复杂的问题：我们每天看到的内容更多了，但真正值得阅读、值得思考、值得进一步表达的内容并没有变得更容易获得。

我们认为当前主要存在三个痛点。

## 1.1 用户不知道看什么：信息爆炸与信息茧房同时存在

今天的用户每天都会面对大量信息：热榜、关注流、推荐流、搜索结果、评论区讨论、公众号文章、短内容、长回答等等。信息供给变得极其充足，但我们的注意力并没有随之增加。

更重要的是，推荐算法会持续强化用户已经表现出的兴趣。用户越关注某一类内容，系统越倾向于推送同类内容。久而久之，用户会被困在自己的信息茧房里，看似每天都在接收新内容，实际上接触到的观点、问题和表达方式越来越相似。

**AI 时代进一步加剧了这个问题。**

大量 AI 生成内容开始进入信息流，其中部分内容结构完整、措辞流畅、逻辑看似正确，但并没有新的观点、经验和判断。它们可能是**正确的废话**：看完之后没有让读者获得新的理解，也无法激发进一步思考。

这导致用户在输入端面临三个问题：

```
内容很多，但不知道哪些值得看；
内容看起来都对，但不知道哪些有观点增量；
长期停留在熟悉兴趣内，很难主动接触兴趣外的知识。
```

因此，第一个核心痛点是：

> **用户不缺少信息，而是缺少高质量、有启发、能激发观点的“好“输入。**
> 

---

## 1.2 用户看了很多，但没有形成自己的观点

在内容平台中，用户经常会点赞、收藏、转发或浏览大量内容，但这些行为并不等于真正完成了思考。

很多时候，用户只是短暂地产生了一个反应：

```
我认同这个判断；
我不同意这个观点；
这个地方好像有问题；
这个内容和我的经历有关；
这个话题以后可以详细阅读一下/可以写一篇文章。
```

但这些反应往往很快消失。

它们没有被记录成观点，也没有被持续追踪，更不会在未来遇到新热点、新讨论、新材料时被重新激活。这些好的东西都随着时间变成了收藏夹里的一个个数字。

这也是很多创作者常见的状态：

```
收藏很多，但真正写的时候想不起来；
看过很多，但很难说清楚自己的判断；
明明有表达欲，但不知道从哪个点开始写。
```

所以第二个痛点是：

> **用户的阅读行为没有被有效转化为观点资产。**
> 

---

## 1.3 用户有想法，但不知道如何写成一篇好文章

我们不否定 AI 辅助写作的价值。

事实上，对于很多用户来说，他们不是没有想法，而是受限于表达能力、结构能力和论证能力，无法把一个模糊想法写成一篇**高质量的文章**，让自己进入一个输出→收到好的反馈→表达欲被增强→继续输出的反馈闭环中。

AI 在这里确实有价值：它可以帮助用户整理思路、组织结构、补充论据、提出反方、优化表达。

但问题在于，如果 AI 只是从一个标题直接生成一篇文章，最终很容易得到一篇“结构正确但缺少主体性”的内容。它可能有完整段落，有看似合理的论证，也有漂亮的总结，但缺少用户自己的判断、经验和真实问题意识。

这样的内容即使没有明显错误，也很难真正打动读者。

我们认为，高质量 AI 辅助创作不应该是：

```
用户给一个题目
↓
AI 直接生成一篇文章
```

而应该是：

```
用户看到内容
↓
产生反应
↓
沉淀观点
↓
与新信息碰撞
↓
确认自己的立场
↓
经过结构化论证和反方审视
↓
最终形成可修改、可发布的文章
```

因此，第三个痛点是：

> **AI 可以帮助用户写作，但真正有价值的内容必须被人类驾驭，必须经过观点碰撞、个人经验和结构化论证。**
>

# 2. 我们如何构建一个模型来尝试解决这个问题？

针对以上问题，我们没有把产品设计成一个简单的 AI 写作工具，而是尝试构建一个从输入到输出的创作闭环。

我们的核心判断是：

> **输入和输出是相互关联的。没有高质量输入，很难产生高质量观点；没有观点沉淀，AI 写作就容易变成空泛生成；没有反馈回流，创作者也很难持续成长。**
> 

因此，我们设计了一个“读写生长模型”。

它可以概括为：

```
看到好内容
↓
形成好观点
↓
写出好文章
↓
根据反馈继续生长
```

在产品中，这个模型被拆成六个阶段：

```
看见
↓
种下
↓
发芽
↓
写作
↓
审稿
↓
回流
```

---

## 2.1 看见：帮助用户筛选高质量输入

第一步是解决“不知道看什么”的问题。

系统会结合知乎开放能力，从多个来源获取内容：

```
知乎热榜
知乎搜索
全网搜索
知乎关注流
知乎圈子内容
评论区讨论
```

但我们不会简单地把这些内容全部展示给用户，而是结合用户的兴趣分类和创作画像进行筛选。

例如，用户关注 AI Coding、Agent 工程化、RAG、程序员成长等不同兴趣小类，那么系统会按兴趣小类组织内容。用户点击某个兴趣小类，只看到当前分类下最值得看的 3～5 张内容卡片。

每张卡片不只是摘要，而是围绕“是否能激发观点”进行组织：

```
标题
推荐理由
内容摘要
主要争议
可写角度
用户操作
```

这样做的目标不是让用户刷更多内容，而是帮用户更快判断：

```
这条内容为什么值得看？
它和我有什么关系？
它有哪些争议？
我能不能基于它写出自己的观点？
```

同时，我们引入“偶遇输入”机制，保留一部分用户兴趣之外、但可能产生跨界启发的内容。

例如，当系统发现用户关注 AI Coding，同时具备金融软件或风控相关背景时，可以把“AI 编程工具进入企业协作场景”和“金融风控中的风险暴露”进行关联，生成一个远端启发：

```
Agent 产品的护城河，不在单次任务完成率，而在持续上下文、风险控制和工作流闭环。
```

这部分设计的目的，是避免推荐系统只强化用户已有兴趣，而是为用户保留一定比例的非线性输入。

---

## 2.2 种下：把阅读反应沉淀为观点种子

当用户看到一张内容卡片时，系统不会要求用户立刻写文章，而是提供轻量操作：

```
总结一下
我认同
我反对
有疑问
加入种子库
基于它写一篇
```

这些操作看起来很轻，但背后会帮助系统识别用户的真实反应。

例如，用户点击“我认同”，系统会进一步提炼：

```
用户认同的是什么？
用户可能保留什么判断？
这个观点未来是否可以写成文章？
还需要补充哪些材料？
```

最终，这条内容会被沉淀成一颗“观点种子”。

观点种子不是原文收藏，而是一个可继续生长的创作单元，包含：

```
内容来源
原始摘要
用户反应
核心观点
可写方向
反方观点
需要补充的材料
当前状态
```

例如：

```
观点种子：
AI 编程工具的护城河可能不在代码生成。

我的反应：
我认同代码生成会越来越普遍，但不认为这就是最终壁垒。

核心观点：
单纯代码生成工具的护城河很浅，真正的壁垒可能在上下文积累、工作流入口、团队协作和企业数据闭环。

可写方向：
1. AI 编程工具为什么会快速同质化？
2. 企业真正需要的是代码，还是可控的工程交付？
3. Agent 产品的护城河到底在哪里？
```

通过这种方式，用户零散的阅读反应会被转化为长期可复用的观点资产。

---

## 2.3 发芽：让旧观点和新热点发生碰撞

很多时候，一个观点不是立刻适合写。

它可能需要等待新的事件、新的讨论、新的材料出现。

因此，我们设计了“今日发芽”机制。

它会把用户历史观点种子与当天的新热点、搜索结果、关注流内容进行匹配，判断哪些旧想法在今天重新具备写作价值。

但考虑到这个过程涉及搜索、语义匹配、LLM 判断和 Token 消耗，我们不会默认在后台高频运行，而是由用户主动触发。

交互上，我们会让“小刘看山”作为引导角色出现：

```
今天要不要给你的种子浇浇水？
我可以帮你看看有没有值得发芽的想法。
```

用户点击后，系统才开始计算并展示发芽机会。

例如：

```
被激活的种子：
单纯代码生成工具的护城河很浅。

触发热点：
某 AI 编程工具宣布被收购 / 发布企业版 / 引发讨论。

为什么值得写：
这个热点验证了你之前的判断：当代码生成能力逐渐同质化，产品竞争重点会转向工作流入口、上下文管理和企业协作数据。

建议标题：
AI 编程工具的护城河，可能不在“会写代码”。
```

这个机制解决的是一个很实际的问题：

> **用户以前有过好想法，但经常错过了最适合表达的时机。**
>

## 2.4 写作：从观点确认到文章生成

当用户决定开始写作后，系统不会直接生成最终文章，而是进入一个分阶段的写作流程：

```
选择观点种子
↓
确认核心观点
↓
选择文章类型
↓
生成论证蓝图
↓
生成表达初稿
↓
圆桌审稿会
↓
定稿草案
↓
发布与反馈
```

其中最关键的是前两步：确认观点和生成论证蓝图。

因为很多 AI 写作内容的问题，不是语言不流畅，而是用户自己的立场没有被确认，文章的论证结构没有真正成立。

所以系统会先向用户确认：

```
我理解你想表达的是：

核心观点：
单纯代码生成能力很难成为 AI 编程工具的长期护城河。真正有价值的是上下文积累、工程工作流入口、团队协作数据和企业级可控交付。

文章基调：
不是否定 AI 编程工具的价值，而是讨论它的长期竞争壁垒在哪里。

你想让读者带走的判断：
AI 编程产品的竞争，将从“谁更会写代码”转向“谁更懂工程现场”。
```

用户确认后，系统再生成论证蓝图，而不是直接生成正文。

蓝图会包含：

```
中心观点
背景
分论点
需要说明的内容
反方观点
回应方向
建议加入的个人经验
```

这一步的作用是确保文章不是简单堆砌内容，而是围绕一个明确观点展开论证。

---

## 2.5 审稿：用圆桌会议替代单一润色

在初稿生成后，系统不会只做普通润色，而是引入“圆桌审稿会”。

圆桌中包含多个不同角色的 Agent：

| Agent | 作用 |
| --- | --- |
| 逻辑审稿 Agent | 检查论证跳跃、未证明前提、概念偷换 |
| 人味编辑 Agent | 检查是否缺少个人经验、真实细节和主体性 |
| 反方读者 Agent | 从反对者视角攻击文章漏洞 |
| 社区传播 Agent | 检查标题、开头、传播性和知乎语境 |

这样做是为了避免文章只变得“更顺”，却没有变得“更有价值”。

例如，逻辑审稿 Agent 可能会指出：

```
你从“代码生成能力会同质化”直接推到“工程上下文更重要”，中间缺少论证。需要解释为什么工程上下文难以被模型直接学习。
```

人味编辑 Agent 可能会指出：

```
文章目前像行业分析，缺少你的真实项目经验。建议加入你过去做复杂后端系统时遇到的状态追踪、历史兼容或客户需求变化。
```

最终，系统会根据圆桌意见整理出定稿草案，但不会直接发布，而是明确要求用户修改后再发布。

这样可以保留人的主体性，也避免纯 AI 生成内容进入社区。

---

## 2.6 回流：把历史反馈变成下一轮创作资产

文章发布后，系统会监控历史反馈，包括：

```
阅读完成率
收藏率
评论数
争议点
高赞评论
反方观点
读者补充材料
```

这些反馈不会只作为数据展示，而会回流到三个地方：

```
用户画像
观点种子库
写作风险提醒
```

例如，如果一篇文章被评论认为“框架很好，但缺少真实案例”，系统会把这个反馈写入用户在对应兴趣分类下的写作提醒：

```
在 Agent 工程化分类下，写作时需要优先补充真实失败案例。
```

这样，产品不是只帮助用户完成一次写作，而是帮助用户持续改进自己的输入、观点和表达方式。
