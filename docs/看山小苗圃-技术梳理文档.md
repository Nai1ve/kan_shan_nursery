## 1. 系统架构设计

### 1.1 总体架构

看山小苗圃采用 **Monorepo + 微服务** 架构，9 个 FastAPI 后端服务 + 1 个 Next.js 前端，通过 API Gateway 统一入口，服务间通过 HTTP JSON 通信。

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Frontend (Next.js 16 :3000)                  │
│   React 19 · TypeScript · 纯 CSS · 双模式(mock/gateway)            │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ HTTP (x-session-id)
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      API Gateway (:8000)                            │
│              纯反向代理 · 会话解析 · 请求路由 · 无业务逻辑            │
└──┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬────────────────┘
   │      │      │      │      │      │      │      │
   ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼
┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐┌──────────┐
│Profile││Content││ Seed ││Sprout││Writing││Feedback││ Zhihu ││   LLM   │
│:8010  ││:8020  ││:8030 ││:8040 ││:8050  ││:8060   ││Adapter││ Service │
│       ││       ││      ││      ││       ││        ││:8070  ││  :8080  │
└──┬────┘└──┬───┘└──────┘└──┬───┘└──────┘└──┬─────┘└──────┘└─────────┘
   │        │               │               │
   ▼        ▼               ▼               ▼
┌─────────────────────────────────────────────────────────────────────┐
│  PostgreSQL (pgvector:pg16 :5432)    │    Redis (7-alpine :6379)    │
│  6 schemas · JSON data column        │    缓存 · 分布式锁 · 配额    │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 服务职责矩阵

| 服务 | 端口 | 核心职责 | 外部依赖 |
|------|------|---------|---------|
| **api-gateway** | 8000 | 纯反向代理，x-session-id → user_id 解析 | profile-service |
| **profile-service** | 8010 | 用户管理、OAuth、画像、Memory、Enrichment | zhihu-adapter, llm-service |
| **content-service** | 8020 | 内容发现、卡片评分、渐进式富化 | zhihu-adapter, profile-service, llm-service |
| **seed-service** | 8030 | 创意种子 CRUD、材料管理、种子合并 | llm-service |
| **sprout-service** | 8040 | 发芽匹配算法、机会识别 | llm-service, seed-service, content-service, profile-service |
| **writing-service** | 8050 | 8 步写作状态机、大纲/初稿/圆桌审稿 | llm-service |

| **llm-service** | 8080 | 12 种 LLM 任务路由、配额、缓存、Trace | OpenAI 兼容 API |

### 1.3 数据库架构

PostgreSQL 按业务域分 6 个 Schema：

```
profile (8 表)                content (2 表)
├── users                     ├── user_profile_snapshots
├── sessions                  └── user_shown_cards
├── zhihu_bindings
├── profile_data              seed (ORM 动态建表)
├── profile_versions          └── seeds (JSON data column)
├── memory_update_requests
├── writing_styles            writing (ORM)
├── llm_configs               └── sessions
└── enrichment_jobs
                              sprout (ORM)
                              ├── opportunities
                              └── runs

                              feedback (ORM)
                              └── articles
```

**设计特点**：ORM 层的表（seed/writing/sprout/feedback）采用 `{data: TEXT JSON}` 列模式，通过 JSON 序列化实现灵活的 Schema 演进，避免频繁 DDL 迁移。

### 1.4 配置体系

三层配置优先级：`默认值 < services/config.yaml < 环境变量`

```python
KanshanConfig
├── provider_mode: "mock" | "live"
├── storage_backend: "memory" | "postgres"
├── database_url / cors_origins
├── zhihu (community / oauth / data_platform / quota)
├── cache (backend: memory | redis, redis_url)
```

### 1.5 双存储后端

每个服务支持 `memory`（进程内 dict）和 `postgres`（SQLAlchemy）两种存储后端，通过 `STORAGE_BACKEND` 环境变量切换。开发时用 memory 实现零依赖启动，生产用 postgres 持久化。

## 2. 算法设计

### 2.1 内容评分算法（7 维度加权）

`scorer.py` 实现 0-100 分的内容卡片评分：

```
Score = InterestFit × 0.35 + SourceQuality × 0.20 + Freshness × 0.15
      + Controversy × 0.10 + WritingPotential × 0.10
      + MemoryFit × 0.05 + DiversityGain × 0.05
```

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| **兴趣匹配** | 35% | `min(100, relevanceScore or 50)` |
| **信源质量** | 20% | `authority × 0.7 + min(100, sourceCount × 30) × 0.3` |
| **时效性** | 15% | `min(100, 50 + likes // 100)` |
| **争议性** | 10% | `min(100, controversy + len(controversies) × 20)` |
| **写作潜力** | 10% | `min(100, len(angles) × 25)` |
| **记忆匹配** | 5% | 默认 50，每命中一个用户偏好视角 +20 |
| **多样性增益** | 5% | `min(100, len(uniqueSourceTypes) × 40)` |

`select_top_cards` 函数：评分 → 降序排列 → 按 sourceId 去重 → 返回 Top N。

### 2.2 多源聚合算法

`transformer.py` 将 2-3 条知乎原始条目聚合为一张多信源卡片：

```
输入: [item_primary, item_supplementary_1, item_supplementary_2]
  │
  ├─ item_primary → 主论点（primarySource）
  ├─ item_supplementary → 补充论据 / 行业案例
  │
  ├─ 聚合评分: 平均相关性、最大点赞/评论数、最高权威度
  ├─ 按分类生成默认推荐理由、争议点、写作角度
  │
输出: WorthReadingCard { originalSources: [3], aggregated scores }
```

### 2.3 TikTok 式渐进富化

内容卡片采用「先展示、后富化」策略，降低首屏加载延迟：

```
请求 Bootstrap
  → 返回卡片 (enriched: false)     ← 零 LLM 调用，毫秒级响应
  → 前端展示基础信息（标题、摘要、来源）

用户浏览卡片
  → POST /content/cards/{id}/enrich  ← 触发 3 次 LLM 调用
     ├─ Step 1: summarize-content    (1200 字 × 3 源)
     ├─ Step 2: extract-controversies
     └─ Step 3: generate-writing-angles (注入用户兴趣记忆)
  → 写回缓存，标记 enriched: true
```

### 2.4 内容调度与缓存

`scheduler.py` 实现三层缓存 + 后台调度：

```
Redis (主)  →  PostgreSQL (持久)  →  内存 dict (降级)
  TTL 24h       UPSERT 持久化       进程内 fallback
```

- **后台线程**：启动后 3 秒首次拉取，之后每 24 小时刷新一次
- **拉取逻辑**：热榜 → 12 个兴趣分类 × 2 查询 × 2 来源 → 聚合 → 缓存 → 预富化 Top 3
- **关注流**：按需加载（依赖 OAuth Token），10 分钟短缓存，分布式锁防并发

### 2.5 用户快照同步

`UserProfileSnapshot` 将用户画像解耦为请求时快照，避免实时 HTTP 调用：

```
Profile Service  ──HTTP──>  Content Service
   (user data)                (snapshot_repo)
        │                         │
        ▼                         ▼
   MD5 hash 比对 ──变更──> 更新 Redis/PG 快照
   MD5 hash 比对 ──未变──> 跳过
```

## 3. Context 设计：圆桌审稿会

### 3.1 架构概览

圆桌审稿是看山最具特色的设计——4 个 AI 角色从不同视角并行审稿，最终聚合为结构化建议。

```
                    ┌─────────────────────┐
                    │   Writing Service   │
                    │  (roundtable-review) │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │    LLM Service      │
                    │  (multi_persona)    │
                    └─────────┬───────────┘
                              │ ThreadPoolExecutor (并发)
              ┌───────┬───────┼───────┬───────┐
              ▼       ▼       ▼       ▼       │
         ┌────────┐┌────────┐┌────────┐┌────────┐
         │ Logic  ││ Human  ││Opponent││Community│
         │Reviewer││ Editor ││ Reader ││ Editor │
         └────────┘└────────┘└────────┘└────────┘
              │       │       │       │
              └───────┴───────┴───────┘
                        │
                  ┌─────▼─────┐
                  │  Aggregator│
                  │ severity 排序 │
                  │ 去重 & 聚合  │
                  └───────────┘
```

### 3.2 四角色定义

每个角色有独立的 Prompt 文件（`prompts/v1/personas/*.md`），聚焦单一审查维度：

| 角色 | ID | 审查焦点 | 典型问题 |
|------|-----|---------|---------|
| **逻辑审稿** | logic_reviewer | 论证完整性、因果关系、数据引用 | 论证跳跃、未证明前提、概念偷换 |
| **人味编辑** | human_editor | 个人经验、反模板化、真实细节 | AI 套话、排比过多、缺少具体场景 |
| **反方读者** | opponent_reader | 反例、样本偏差、结论外推 | 未承认反例、论据漏洞、过度外推 |
| **社区编辑** | community_editor | 知乎语境、标题吸引力、规范合规 | 标题不吸引、开头缺代入感、AI 声明风险 |

### 3.3 执行流程

```python
# router.py - _run_multi_persona
1. 加载基础 Prompt + 各角色 Persona Prompt
2. ThreadPoolExecutor 并发提交 4 个任务
3. 每个角色独立执行：
   ├─ 优先用用户自定义 LLM
   ├─ 失败则降级到平台 LLM
   └─ 再失败则降级到 MockProvider
4. 聚合结果：
   ├─ reviews[] 按 severity 排序 (high > medium > low)
   ├─ 每条 review 标注 _persona 和 _provider
   ├─ subCalls[] 记录每个角色的延迟和降级状态
   └─ 整体缓存（一次缓存 = 4 个角色结果）
```

### 3.4 降级隔离

**关键设计**：每个角色的降级是独立的。一个角色失败不影响其他角色：

```
logic_reviewer  → user_provider ✅  → 正常结果
human_editor    → user_provider ❌ → platform ✅  → 正常结果 + 降级通知
opponent_reader → user_provider ❌ → platform ❌ → mock ✅  → 兜底结果
community_editor → user_provider ✅  → 正常结果
```

### 3.5 输出结构

```json
{
  "reviews": [
    {
      "role": "logic_reviewer",
      "summary": "50字以内整体评价",
      "problems": [{ "type": "logic", "description": "...", "location": "...", "severity": "high" }],
      "suggestions": [{ "type": "...", "content": "...", "example": "..." }],
      "strengths": ["文章优点"],
      "overallScore": 78,
      "_persona": "logic_reviewer",
      "_provider": "openai_compat"
    }
  ],
  "aggregatedIssues": [{ "issue": "...", "mentionedBy": ["logic_reviewer", "opponent_reader"], "priority": "high" }],
  "recommendedActions": ["..."]
}
```

## 4. 用户画像与 Memory 管理

### 4.1 Memory 双层结构

```
ProfileData
├── globalMemory (全局记忆)
│   ├── longTermBackground    -- 长期背景
│   ├── contentPreference     -- 内容偏好
│   ├── writingStyle          -- 写作风格
│   ├── recommendationStrategy -- 推荐策略
│   └── riskReminder          -- 风险提醒
│
└── interestMemories[] (兴趣记忆，按 12 个分类)
    ├── interestId / interestName
    ├── knowledgeLevel         -- 入门/中级/进阶
    ├── preferredPerspective   -- 偏好视角列表
    ├── evidencePreference     -- 证据偏好
    ├── writingReminder        -- 写作提醒
    └── feedbackSummary        -- 反馈摘要
```

### 4.2 Memory 生命周期

```
注册 → 兴趣选择 → 写作风格问卷
         │
         ▼
   Enrichment Job (queued)
         │
         ├─ 收集 OAuth 信号（关注列表、粉丝、动态）
         ├─ 构建 ProfileSignalBundle
         ├─ 调用 LLM profile-memory-synthesis
         │     ├─ 成功 → diff 现有记忆 → 生成 MemoryUpdateRequest
         │     └─ 失败 → 规则兜底 → 基于 onboarding 生成请求
         │
         ▼
   MemoryUpdateRequest (pending)
         │
         ├─ 用户审批 → apply → 写入 globalMemory / interestMemory
         └─ 用户拒绝 → reject → 标记 rejected
```

### 4.3 行为分析算法

`MemoryAlgorithm` 实现三层时间窗口的行为分析：

| 窗口 | 周期 | 分析内容 |
|------|------|---------|
| 短期 | 7 天 | 最近种子交互（agree/disagree/question 模式） |
| 中期 | 30 天 | 模式识别（知识水平变化、视角偏好迁移） |
| 长期 | 全量 | 稳定画像特征 |

**自动建议阈值**：
- 3+ 次 agree → 高置信度 `preferredPerspective` 建议
- 3+ 次 disagree → 中置信度对立信号
- **所有建议都需用户确认**（`should_auto_apply()` 始终返回 `False`）

### 4.4 Memory 注入机制

写作和内容推荐时，Memory 被注入 LLM Prompt：

```python
# memory_service.py - build_injection_summary
"【兴趣记忆：数码科技】
 知识水平：中级
 偏好视角：AI工具, 编程实践
 证据偏好：个人经验 + 案例
 写作提醒：不要只讲参数和发布会，需要补充实际使用场景

 【全局记忆】
 写作风格：清晰、克制；允许有观点锋芒
 风险提醒：容易写成逻辑完整但缺少个人经历"
```

## 5. 今日发芽算法

### 5.1 算法目标

将用户的「种子库」与「今日热点」进行匹配，识别值得立即写作的发芽机会。

### 5.2 评分模型

`scorer.py` 实现 6 因子加权评分（0-100）：

```
ActivationScore = seedMaturity × 0.25
               + topicRelatedness × 0.25
               + freshness × 0.20
               + newInfoGain × 0.15
               + memoryFit × 0.10
               + controversyPotential × 0.05
               + penalties
```

| 因子 | 权重 | 计算方式 |
|------|------|---------|
| **种子成熟度** | 25% | `maturityScore / 100`（由种子材料充分度决定） |
| **话题关联度** | 25% | 种子标题/主张与热点卡片标题的 **Bigram Jaccard 相似度** |
| **时效性** | 20% | 基于卡片时间戳：≤1天=1.0, ≤3天=0.9, ≤7天=0.7, >7天线性衰减 |
| **新信息增益** | 15% | 今日卡片能否补充种子缺失材料（Bigram 匹配 `requiredMaterials`） |
| **记忆匹配** | 10% | 种子 `interestId` 命中用户 `interestMemories` → 1.0，部分命中 → 0.7 |
| **争议潜力** | 5% | `min(1, (counterArguments + controversies) / 5)` |

### 5.3 惩罚因子

| 惩罚项 | 值 | 触发条件 |
|--------|-----|---------|
| 无材料 | -0.25 | `wateringMaterials` 为空 |
| 无来源 | -0.15 | 材料中无 `sourceLabel` / `sourceUrl` |
| 近期已跳过 | -0.30 | 种子在 `dismissed_pairs` 中 |
| 话题已发布 | -0.20 | 种子标题在 `published_topics` 中 |

### 5.4 关联度计算：Bigram Jaccard

```python
# 种子文本 = title + coreClaim + sourceTitle + 材料标题
seed_bigrams = extract_bigrams(seed_text)  # 中文二元组

# 对每张热点/今日卡片计算 Jaccard
similarity = |seed_bigrams ∩ card_bigrams| / |seed_bigrams ∪ card_bigrams|

# 取最高相似度的卡片作为触发卡片
best_trigger_cards, trigger_type = argmax over all cards
```

### 5.5 LLM 增强层

确定性评分完成后，候选结果送入 LLM `sprout-opportunities` 任务做语义增强：

```
确定性评分 (scorer.py)          LLM 增强 (sprout-opportunities)
├── 6 因子分数                   ├── whyWorthWriting: 为什么现在写
├── 最佳触发卡片                 ├── suggestedTitle: 建议标题
├── 惩罚因子                     ├── suggestedAngle: 写作角度
└── total (0-100)               ├── missingMaterials: 缺失材料
                                ├── maturityBreakdown: 4 维成熟度
                                └── suggestedAction: write_now/supplement/discuss/wait
```

## 6. 成本与稳定性设计

### 6.1 LLM 成本控制

#### 6.1.1 每日配额系统

```python
# 配额键格式：按用户 × 任务类型 × 日期
quota:{task_type}:{user_id}:{YYYYMMDD}

# 只在平台 Provider 实际调用时扣减
# 用户自定义 LLM 不消耗平台配额
```

12 种任务类型各自独立配额，配置在 `config.llm.quota` 中。

#### 6.1.2 缓存优先

```
缓存键 = llm:{task_type}:{input_hash}:{prompt_version}:{schema_version}:{provider_cache_key}

输入 Hash = SHA-256(JSON(input_data))[:16]

TTL = 6 小时（可配置）
```

相同输入 + 相同 Prompt 版本 + 相同 Provider = 缓存命中，跳过所有 LLM 调用和配额扣减。

#### 6.1.3 Provider 降级链

```
用户自定义 LLM ──失败──> 平台 LLM ──失败──> MockProvider (零成本兜底)
     │                      │                    │
     │ 不扣平台配额          │ 扣平台配额          │ 不扣配额
     │                      │                    │
     └─ 附加降级通知 ────────┘                    └─ 结构化 mock 响应
```

#### 6.1.4 高成本操作需用户主动触发

| 操作 | 触发方式 | 成本 |
|------|---------|------|
| 内容富化（3 次 LLM） | 用户浏览卡片时按需触发 | 3 次 LLM 调用 |
| 圆桌审稿（4 个 Persona） | 用户在写作流程中主动点击 | 4 次 LLM 调用 |
| 发芽匹配 | 用户主动点击「今日发芽」 | 1 次 LLM 调用 + 确定性评分 |
| Profile Enrichment | 注册后自动触发一次 | 1 次 LLM 调用 |
| 种子材料补充 | 用户主动点击「Agent 补充」 | 1 次 LLM 调用 |

### 6.2 稳定性设计

#### 6.2.1 三级降级

```
Level 1: 正常路径    (OpenAI 兼容 API / 用户自定义 LLM)
Level 2: Provider 降级 (zhihu_direct → openai_compat → mock)
Level 3: 缓存降级     (Redis → PostgreSQL → 内存 dict)
```

#### 6.2.2 分布式锁

关注流加载使用 Redis `SET NX` 分布式锁（30 秒 TTL），防止多实例并发刷新：

```python
lock_key = f"following:lock:{user_id}:{token_hash}"
redis.set(lock_key, "1", nx=True, ex=30)
```

#### 6.2.3 Trace 日志

每次 LLM 调用记录 JSONL Trace：

```json
{
```

## 7. 技术亮点总结

### 7.1 圆桌审稿：多视角并行审查

4 个 AI Persona 从逻辑、人味、反方、社区四个维度并发审查，每个角色独立降级，聚合后按严重程度排序。这是对传统单一 LLM Review 的根本性改进——单一视角容易遗漏盲区，多视角交叉验证显著提升审查质量。

### 7.2 TikTok 式渐进富化

首屏零 LLM 调用（毫秒级响应），用户浏览时按需富化。3 次 LLM 调用（摘要 → 争议提取 → 写作角度）串行执行但异步返回，用户体验为「内容逐渐丰富」而非「等待加载」。

### 7.3 确定性评分 + LLM 增强的双层架构

内容评分和发芽匹配都采用「确定性算法打底 + LLM 语义增强」的双层设计：

```
确定性层（scorer.py）          LLM 层（prompt）
├── 可解释、可调试              ├── 语义理解
├── 零成本、低延迟              ├── 创意生成
├── 保证基本排序正确            ├── 个性化建议
└── 作为 LLM 输入的结构化信号   └── 输出自然语言解释
```

### 7.4 Memory 双层架构 + 用户审批

全局记忆（写作风格、风险提醒）+ 兴趣记忆（按 12 个分类独立维护），所有 AI 生成的记忆更新都以 `MemoryUpdateRequest` 形式进入 pending 状态，用户必须主动审批。

### 7.5 用户自定义 LLM + 平台兜底

用户可配置自己的 OpenAI 兼容 LLM（自有 API Key），系统优先使用用户 Provider，失败时自动降级到平台免费额度。用户 Provider 不消耗平台配额。

### 7.6 公共内容池 + 用户个性化筛选

内容拉取（知乎热榜 + 分类搜索）与用户画像解耦——后台线程维护公共内容池，请求时通过用户快照做个性化筛选和重排序。同一份缓存服务所有用户。

### 7.7 零外部 HTTP 依赖

所有服务间通信使用 Python 标准库 `urllib.request`，无 `httpx`/`requests` 依赖（profile-service 除外，它用 httpx 做 OAuth）。减少依赖冲突，简化部署。

### 7.8 JSONL 结构化日志

双 Sink 日志（控制台人类可读 + JSONL 文件机器可读），每个服务独立日志文件，支持按日期轮转：

```
output/logs/content-service-2026-05-14.jsonl
output/logs/llm-trace-2026-05-14.jsonl
```

## 8. 安全设计

### 8.1 Memory 安全

#### 8.1.1 用户完全控制

```
AI 生成建议 ──> MemoryUpdateRequest (pending)
                     │
            ┌────────┴────────┐
            ▼                 ▼
     用户审批 (apply)    用户拒绝 (reject)
            │                 │
            ▼                 ▼
     写入 Memory          标记 rejected
```

**核心原则**：`should_auto_apply()` 始终返回 `False`。没有任何 AI 建议会自动写入用户记忆。

#### 8.1.2 记忆变更可追溯

```sql
profile.memory_update_requests
├── interest_id      -- 关联的兴趣分类
├── target_field     -- 目标字段
├── suggested_value  -- 建议值
├── reason           -- 变更原因（可解释）
├── status           -- pending / approved / rejected
└── created_at       -- 创建时间
```

每条记忆变更都有明确的 `reason` 字段，用户可查看 AI 为什么提出这个建议。

#### 8.1.3 版本快照

```sql
profile.profile_versions
├── target    -- 变更目标
├── snapshot  -- 变更前的完整快照 (JSON)
├── reason    -- 变更原因
└── created_at
```

每次记忆变更都保存变更前的完整快照，支持回溯。

### 8.2 内容安全：AI 不自动发布，用户始终拥有最终决定权

#### 8.2.1 写作流程的 8 步状态机

```
claim → blueprint → outline → draft → roundtable → finalize → publish
  │        │          │        │         │            │          │
  ▼        ▼          ▼        ▼         ▼            ▼          ▼
用户确认  用户确认    用户确认  用户生成   用户审查     用户编辑    用户点击
  必须     必须        必须     主动触发   主动触发     必须       必须
```

**每一步都需要用户主动确认或触发**：

| 步骤 | AI 角色 | 用户角色 |
|------|---------|---------|
| Claim 确认 | 无 | 用户确认/修改核心主张 |
| Blueprint | 生成论证蓝图 | 用户审查、补丁、重新生成 |
| Outline | 生成大纲 | 用户审查、编辑、确认 |
| Draft | 生成初稿 | 用户审查、编辑 |
| 圆桌审稿 | 4 角色并行审查 | 用户逐条审查建议，选择采纳 |
| Finalize | 无 | 用户做最终编辑 |
| Publish | 无 | **用户主动点击发布按钮** |

#### 8.2.2 AI 贡献透明化

初稿生成时记录 `aiContributionLog`：

```json
{
  "aiDisclosureSuggestion": "本文在结构组织和部分论述上使用了AI辅助",
  "aiContributionLog": ["大纲结构", "部分论述段落"],
  "sourceList": [{ "title": "...", "url": "...", "usage": "..." }]
}
```

AI 辅助声明建议按参与度分级：
- 低参与 (<20%)：可不声明
- 中参与 (20-50%)：建议声明
- 高参与 (>50%)：必须声明

**系统只建议，不替用户决定是否声明。**

#### 8.2.3 个人经历不可编造

Draft Prompt 明确约束：
> "保留用户原话：用户提供的 personalExperience 必须原样保留"
> "不编造经历：AI不创造用户的个人经历"

Human Editor 角色专门检查：
> "不允许编造作者的个人经历。当文章缺细节时，应该提示作者补一段具体场景，而不是替作者写。"

#### 8.2.4 发布为独立操作

```python
# writing-service 状态机
"publish" 是一个需要用户显式触发的终态操作
AI 在任何步骤都不能自动将内容推向发布状态
```

### 8.3 Prompt 防护

#### 8.3.1 结构化输入输出

所有 12 种 LLM 任务都定义了严格的 JSON Schema：

```python
# validators.py
TASK_REQUIRED_KEYS = {
    "summarize-content": ["summary", "keyPoints", "sourceIds", "nextAction"],
    "draft": ["title", "body", "aiDisclosureSuggestion"],
    "roundtable-review": ["reviews"],
    # ... 每种任务都有明确的必需字段
}
```

输出验证失败 → 触发 Provider 降级（不会返回损坏数据给用户）。

#### 8.3.2 Prompt 注入防护

- **Prompt 版本化**：所有 Prompt 存储在 `prompts/v1/` 目录，通过 `prompt_version` 参数管理，变更可追溯
- **输入边界**：Prompt 明确约束输入格式（JSON Schema），减少注入面
- **角色约束**：每个 Prompt 都包含明确的约束条件（"不替作者决定立场"、"不编造用户历史"）
- **输出校验**：`validate_output()` 检查必需字段，异常输出触发降级而非直接返回

#### 8.3.3 多层 Prompt 隔离

```
系统 Prompt（固定）
  + 任务 Prompt（prompts/v1/{task}.md）
    + Persona Prompt（仅圆桌审稿，prompts/v1/personas/*.md）
      + 用户输入（结构化 JSON）
```

每层都有明确职责边界，Persona Prompt 不能覆盖任务 Prompt 的约束。

### 8.4 成本控制：高成本操作需用户主动触发

#### 8.4.1 成本分级

| 操作 | LLM 调用次数 | 触发方式 | 成本级别 |
|------|-------------|---------|---------|
| 内容富化 | 3 次/卡片 | 用户浏览时按需 | 低 |
| 种子问答 | 1 次 | 用户主动提问 | 低 |
| 材料补充 | 1 次 | 用户主动触发 | 低 |
| 发芽匹配 | 1 次 | 用户主动触发 | 中 |
| 论证蓝图 | 1 次 | 用户在写作流程中触发 | 中 |
| 大纲生成 | 1 次 | 用户在写作流程中触发 | 中 |
| 初稿生成 | 1 次 | 用户在写作流程中触发 | 高 |
| **圆桌审稿** | **4 次** | **用户主动触发** | **最高** |
| Profile 富化 | 1 次 | 注册后自动一次 | 中 |

#### 8.4.2 配额隔离

```python
# 只有平台 Provider 调用才扣配额
if provider_used == "platform":
    increment_quota(task_type, user_id)

# 用户自定义 LLM → 不扣平台配额
# MockProvider → 不扣配额
```

#### 8.4.3 超限处理

```json
// HTTP 429 响应
{
  "error": "quota_exceeded",
  "task_type": "roundtable-review",
  "used": 5,
  "limit": 5,
  "remaining": 0,
  "message": "今日圆桌审稿配额已用完"
}
```

用户可选择：
1. 等待次日配额重置
2. 配置自己的 LLM Provider（不受平台配额限制）

#### 8.4.4 圆桌审稿的成本优化

圆桌审稿是系统中成本最高的操作（4 次并发 LLM 调用），采取了专门的成本优化：

1. **缓存**：4 个角色的结果作为整体缓存，相同输入 6 小时内不重复调用
2. **按需触发**：只有用户在写作流程中主动点击「开始审稿」才触发
3. **独立降级**：单个角色失败不影响其他角色，失败角色使用零成本 Mock
4. **用户 LLM 优先**：配置了自定义 LLM 的用户优先使用自己的额度
