# 看山小苗圃-核心功能完成情况与模型任务分解

更新时间：2026-05-14

本文用于当前冲刺阶段的开发调度。口径以“一个完整的读写创作工作台”为准：

```text
登录与用户画像
→ 今日看什么
→ 观点种子库
→ 今日发芽
→ 写作苗圃
→ 历史反馈
```

判断标准不是“接口是否存在”，而是用户能否从页面完成闭环，并且关键状态能否进入后端服务。

---

## 1. 当前核心功能完成情况

### 1.1 总体结论

当前项目已经具备完整服务目录、主要接口骨架和前端主界面，最接近可演示的是：

```text
登录入口
→ 今日看什么真实内容卡片
→ 认同/反对/有疑问沉淀观点种子
→ 种子库继续浇水（已接入后端 Agent 服务）
```

**最新进展（2026-05-14）：**
- 种子库已完成 Agent 服务接入，包括：
  - "今日看什么"中的疑问功能（问 Agent）已调用后端 `addSeedQuestion` API
  - 种子库中的"Agent补证据"和"Agent找反方"已调用后端 `agentSupplementSeed` API
  - 种子库中的"待解决问题 - 问Agent"已调用后端 `addSeedQuestion` API
- 种子库存储后端已禁用 fixtures 预加载，只返回数据库中已有的种子
- LLM 服务已修复参数传递问题，`OpenAICompatProvider` 现在会将 `input_data` 注入到 prompt 中

仍然没有完全后端化的是：

```text
今日发芽
写作苗圃
历史反馈回流
OAuth 用户数据汇总画像生成
```

这几部分在前端或服务内都有骨架，但主流程仍偏 mock / local state，需要继续联调。

## 4. 当前真实链路判断

### 4.1 今日看什么

当前链路：

```text
zhihu-adapter
  hot-list / zhihu-search / global-search / following-feed
→ content-service scheduler 构建系统内容池
→ Redis / memory cache
→ content-service 按用户 snapshot / interestMemory 评分选卡
→ api-gateway
→ frontend 今日看什么单卡浏览
```

已完成：

- 内容不再只是热榜。
- 兴趣小类可从真实搜索结果生成卡片。
- 卡片支持原始来源、完整内容、摘要、争议点、可写角度。
- 缓存构建时会对 Top 卡片调用 LLM enrich。
- 前端改成单卡浏览，使用“换一条 + 后台预取”。

未完成 / 待验收：

- 关注流精选必须用真实 OAuth token 验收。
- `profile-service /internal/auth/zhihu-token` 与 content-service 的实际 user_id 是否一致，需要端到端确认。
- 当前只 enrich 每类 Top 3；后续如果用户刷得多，后面的卡片仍可能只有基础字段。
- 还没有接入知乎圈子作为内容来源。

### 4.2 观点种子库

当前链路：

```text
前端卡片操作
→ 本地状态立即更新
→ gateway / seed-service 持久化
```

已完成：

- `seed-service` 有核心状态机。
- `from-card` 能保存卡片快照、reaction、来源证据、反方材料。
- 疑问、材料、Agent 补证据 / 找反方、合并种子接口都存在。
- 成熟度公式可用。

未完成 / 待修：

- 前端 `answerQuestion` 仍使用本地 `buildAgentAnswer`，应改为调用 `POST /api/v1/seeds/{seed_id}/questions`。
- 前端 `markQuestion`、`addMaterial`、`updateMaterial`、`deleteMaterial`、`agentSupplement` 应以后端返回 seed 为准。
- `from-card` 查重目前按 `createdFromCardId`，要确认是否需要 user_id 维度，避免多用户互相覆盖。

### 4.3 今日发芽

当前链路：

```text
前端 local setTimeout
→ buildSproutOpportunity
→ 本地 sproutOpportunities
```

后端 `sprout-service` 虽有接口，但还不是主链路。

应改成：

```text
frontend POST /api/v1/sprout/start
→ sprout-service 读取 seed-service / content-service / profile-service
→ llm-service sprout-opportunities
→ 返回机会
→ 用户补资料 / 换角度 / 暂不写 / 开始写作
```

这是 P1 的核心创新点，必须在 Demo 前补到“看得出不是 mock”的程度。

### 4.4 写作苗圃

当前链路：

```text
前端 local WritingSession
→ 用户手动切步骤
→ local draft / roundtable / publish
```

后端 `writing-service` 有可用骨架，但前端没有真正接入主流程。

应改成：

```text
seed
→ POST /api/v1/writing/sessions
→ confirm claim
→ generate / regenerate blueprint
→ user edits blueprint
→ generate outline with citations
→ generate draft
→ roundtable discussion
→ finalize
→ publish mock
→ feedback-service sync
```

24 小时内不建议追求复杂编辑器，优先把“逐步细化”和“圆桌可讨论”做出来。

### 4.5 历史反馈

当前链路：

```text
前端 publishWriting
→ 本地 feedbackArticles 增加一条
```

后端 `feedback-service` 有能力，但没有成为前端真实发布后的下游。

应改成：

```text
writing-service publish mock
→ feedback-service sync article
→ comments-summary
→ second-seed payload
→ memory-update-request
→ profile-service 用户确认后写入 Memory
```

Demo 阶段可以保持 mock 指标，但必须让“反馈重新变成资产”的动作通过服务完成。

## 5. 模型分工原则

当前只建议分成四类角色：

```text
Codex：高风险核心链路、状态机、算法、跨服务联调、最终 review
GLM：边界明确的接口补齐、前端 API 接线、CRUD、测试桩修复
低风险模型：文档、curl 脚本、fixture、README、静态 UI 文案
人工：密钥、OAuth 授权、真实服务启动、浏览器验收、产品取舍
```

不要把同一个服务的同一批文件同时交给多个模型改。每个任务必须限定写入范围。

## 11. 当前最小 Demo 路径

如果时间非常紧，最小可讲 Demo 应锁定这条路径：

```text
1. 用户登录并完成基础兴趣画像
2. 关联知乎账号
3. 今日看什么展示真实兴趣卡片 / 关注流卡片

6. 继续浇水，补证据或反方材料
7. 手动今日发芽，从旧种子匹配今日热点
8. 进入写作苗圃，生成蓝图、大纲、圆桌建议、定稿草案
9. 模拟发布，进入历史反馈，生成二次种子或 Memory 更新建议
```

如果只能保 6 步，则优先保到“继续浇水”。如果还能保 9 步，再补发芽和写作。
