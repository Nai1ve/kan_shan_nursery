# 开发路线图

本文档描述看山小苗圃从原型到可演示产品的阶段路线。当前版本不应只停在“知乎能力接入”，后续还必须覆盖 Agent 编排、推流算法、发芽算法、写作苗圃、反馈回流、评测调优和部署演示。

---

## v0.1 文档与原型

目标：把产品方向讲清楚。

交付：

- 产品文档。
- 技术文档。
- 服务拆分文档。
- HTML 原型。
- 核心口号和主链路：看到好内容，形成好观点，写出好表达。

验收：

- 能讲清楚“不是 AI 代写，而是读写闭环”。
- 页面原型能展示今日看什么、种子库、今日发芽、写作苗圃、历史反馈。

---

## v0.2 工程骨架

目标：建立可分工的 monorepo。

交付：

- monorepo 目录。
- Git 管理。
- `.gitignore`。
- 服务 README。
- 工程约束文档。
- AI coding 任务模板。
- Docker Compose 占位。

验收：

- 每个服务的边界、允许修改范围、禁止修改范围清楚。
- 其他 AI Coding 工具能按 README 接任务。

---

## v0.3 服务健康检查

目标：让所有服务具备最小可运行形态。

交付：

- 每个服务提供 `/health`。
- `api-gateway` 能展示下游 readiness。
- 统一端口约定。
- 基础 requirements。

验收：

- 本地能启动关键服务。
- 服务未实现时返回 `SERVICE_NOT_READY`，而不是假成功。

---

## v0.4 P0 Mock 闭环

目标：不依赖真实知乎 API，跑通完整演示闭环。

交付：

- 演示模式登录。
- 读取用户画像和 Memory。
- 今日看什么 mock 卡片。
- 原始来源卡内展示。
- 加入种子库。
- 有疑问 / Agent mock 回答 / 浇水材料。
- 今日发芽 mock。
- 写作苗圃 8 步流程。
- 历史反馈 mock。

验收：

- 刷新页面后 mock 状态不丢失。
- 所有可见按钮有状态变化。
- 能演示一条端到端主链路。

---

## v0.5 核心 Agent 基座

目标：建立 LLM 任务统一入口，而不是各服务散写 prompt。

交付：

- `llm-service`。
- prompt 模板目录。
- mock provider。
- 直答 Agent wrapper。
- LLM 缓存。
- 输出 schema 校验。
- fallback。

任务类型：

- 内容摘要。
- 争议点提取。
- 可写角度生成。
- 疑问回答。
- 浇水材料补充。
- 发芽机会生成。
- 论证蓝图。
- 初稿。
- 圆桌审稿。
- 反馈总结。

验收：

- 每个 task 有结构化 mock 输出。
- 相同输入可缓存。
- provider 失败可 fallback。

---

## v0.6 知乎能力接入

目标：把知乎开放能力统一接入 `zhihu-adapter`，业务服务不直接依赖官方 raw 字段。

交付：

- 热榜。
- 知乎搜索。
- 全网搜索。
- 关注流。
- 圈子内容和评论。
- 故事内容。
- 直答 Agent。
- Redis 短缓存。
- quota 保护。
- `ZhihuContentItem` 标准 DTO。

验收：

- 缓存命中不扣额度。
- Count / Limit 超限时 adapter 截断。
- 全网搜索 `<em>` 保留 rawHtml，同时生成纯文本 summary。
- 故事内容展示署名和活动限制。

---

## v0.7 统一网关与本地联调

目标：前端逐步从 localStorage mock 迁移到 gateway。

交付：

- `api-gateway` 统一 envelope。
- request id 生成与透传。
- profile / seed / llm / zhihu-adapter 代理。
- 未完成服务返回 `SERVICE_NOT_READY`。
- 本地启动脚本或说明。

验收：

- 前端只需配置 gateway 地址。
- 下游错误统一返回。
- 已完成服务能经 gateway 调通。

---

## v0.8 今日看什么推流算法

目标：把“今日看什么”从静态 mock 升级为规则版创作输入流。

交付：

- Query Plan 生成。
- 兴趣小类候选召回。
- 关注流精选。
- 偶遇输入。
- 多来源卡片聚合。
- 卡片评分。
- 多样性去重。
- 当前分类刷新。
- 来源详情展示数据。

算法验收：

- 每个兴趣分类至少 2-3 张卡片。
- 关注流和偶遇输入平级展示。
- 重点卡片有 2-3 个来源。
- 刷新当前分类不影响种子、写作、画像、反馈。
- 偶遇输入必须能解释远端关联，不是随机内容。

---

## v0.9 观点种子与浇水 Agent 强化

目标：让“看完有想法”变成可成长的观点资产。

交付：

- 从卡片创建/更新种子。
- reaction 幂等更新。
- 疑问线程多轮追问。
- Agent 回答写入材料板。
- 四格浇水材料 CRUD。
- Agent 补证据 / 找反方。
- 成熟度计算。
- 相似种子合并。

算法验收：

- 同一 cardId 不重复创建种子。
- `resolved` 结束问题线程。
- `needs_material` 保留待解决问题。
- 每补充关键材料成熟度合理变化。
- 个人经验只能引导用户补充，不能由 AI 编造。

---

## v0.10 今日发芽算法

目标：把历史种子和今日新信息碰撞，生成值得写的机会。

交付：

- 用户主动触发发芽。
- 候选种子筛选。
- 热榜 / 搜索 / 关注流新信息召回。
- seed-topic 匹配。
- newInfoGain 计算。
- activationScore。
- 发芽机会卡片。
- 补资料 / 换角度 / 暂时不写 / 开始写作动作。

算法验收：

- 发芽不是自动高频任务。
- 发芽机会必须解释：哪颗种子、什么触发、为什么现在值得写。
- 分数低于阈值不展示。
- 热点相关但证据不足时，应建议继续浇水而不是直接写。

---

## v0.11 写作苗圃与圆桌审稿

目标：把成熟观点转成可编辑文章，而不是一步代写。

交付：

- WritingSession。
- Memory 注入和前端可见展示。
- memoryOverride 可编辑。
- 核心观点确认。
- 文章类型和语气。
- 论证蓝图。
- 初稿。
- 圆桌审稿。
- 定稿草案。
- AI 辅助声明建议。

Agent：

- 写作教练 Agent。
- 逻辑审稿 Agent。
- 人味编辑 Agent。
- 反方读者 Agent。
- 社区传播 Agent。

验收：

- 创建 session 时根据 seed.interestId 注入兴趣 Memory。
- 用户可修改 memoryOverride，且不覆盖长期 Memory。
- 圆桌审稿能指出逻辑、人味、反方和社区表达问题。
- 定稿前必须由用户确认。

---

## v0.12 历史反馈与 Memory 回流

目标：把发布后的互动数据变成下一轮创作资产。

交付：

- 历史反馈文章列表。
- 评论摘要。
- 互动指标展示。
- 二次文章建议。
- 二次种子 payload。
- Memory 更新建议。

验收：

- feedback-service 不直接写 Memory。
- 只创建 `memory_update_requests`。
- 用户 apply 后才更新长期 Memory。
- 评论中的反方观点能进入种子材料或二次文章建议。

---

## v0.13 Agent 质量评估与成本控制

目标：让系统可调优、可解释、可控成本。

交付：

- 内容输入质量指标。
- 种子质量指标。
- 发芽质量指标。
- 写作质量指标。
- LLM 调用成本统计。
- Redis 命中率统计。
- quota 使用预警。
- fallback 率统计。

验收：

- 能知道每次发芽用了多少缓存、多少搜索、多少 LLM。
- 能知道哪类卡片最容易转成种子。
- 能知道圆桌审稿建议是否被采纳。
- 能在接近知乎额度上限时停止回源。

---

## v0.14 Demo 演示链路与部署打磨

目标：比赛现场稳定演示。

交付：

- 一键启动说明。
- 演示数据预置。
- 关键 API fallback。
- Demo 脚本。
- 录屏备用。
- AI 辅助声明和知乎规范说明。

推荐演示链路：

```text
个人画像 Memory
→ 今日看什么 AI Coding 卡片
→ 展开原始来源
→ 有疑问并写入浇水材料
→ 加入种子库
→ 今日发芽
→ 写作苗圃 Memory 注入
→ 论证蓝图
→ 圆桌审稿
→ 定稿草案
→ 历史反馈生成二次种子和 Memory 更新建议
```

验收：

- 断网或知乎 API 失败时仍能 mock 演示。
- 现场不依赖长时间生成。
- 每一步都能解释产品价值。
