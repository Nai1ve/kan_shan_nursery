# 看山小苗圃技术设计文档

## 一、系统定位

看山小苗圃是一个围绕知乎生态构建的读写一体创作 Agent。

系统核心不是直接代写文章，而是维护一个完整的创作闭环：

```
输入筛选
↓
观点种子沉淀
↓
长期 Memory
↓
热点触发
↓
写作流程编排
↓
圆桌审稿
↓
定稿草案
↓
发布反馈回流
```

---

## 二、核心模块

系统可以拆成 7 个模块：

| 模块 | 说明 |
| --- | --- |
| 用户与认证模块 | 登录、知乎账号关联、本地用户管理 |
| 用户画像 Memory 模块 | 管理全局画像、兴趣分类画像、Memory 更新请求和版本记录 |
| 输入筛选模块 | 热榜、搜索、关注流、圈子内容筛选 |
| 观点种子模块 | 生成、保存、管理、合并观点种子 |
| 今日发芽模块 | 用户触发后计算种子与热点的匹配 |
| 写作苗圃模块 | 多步骤写作流程和文章生成 |
| 历史反馈模块 | 监控文章表现，提取反馈并回流 Memory |

---

## 3. 个人画像页

### 重要设计

用户画像不应该只有一份全局画像。

正确结构应该是：

```

+
InterestScopedProfile[]
```

也就是说：

```

  userId:string;
  globalMemory:GlobalMemory;
  interestProfiles:InterestProfile[];
}
```

### 全局画像

```
interfaceGlobalMemory {
  nickname:string;
  role:string;
  longTermBackground:string;
  generalWritingStyle:string;
  generalAvoidances:string[];
  globalContentPreference:string[];
}
```

### 兴趣分类画像

```

  interestId:string;
  interestName:string;

  knowledgeLevel:'beginner'|'intermediate'|'advanced'|'expert';
  currentFocus:string[];
  preferredPerspective:string[];
  writingStyleOverride:string;
  evidencePreference:'experience'|'data'|'paper'|'case'|'balanced';
  riskReminders:string[];

```

### 示例

```

  "interestName":"AI Coding",
  "knowledgeLevel":"intermediate",
  "currentFocus": ["AI 编程工具","研发工作流","工程上下文"],
  "preferredPerspective": ["工程交付","复杂系统","程序员成长"],
  "writingStyleOverride":"可以更偏工程师复盘，加入真实项目经验",
  "evidencePreference":"case",

"不要只讲行业趋势",
"需要补充具体工程案例",
"需要回应长上下文模型的反方观点"
  ]
}
```

### Memory 维护模块归属

P0 阶段，Memory 维护保留在 `profile-service` 中，但必须作为独立子模块实现。

推荐结构：

```
profile-service
├── profile
│   ├── global profile
│   └── interest profile
└── memory
    ├── global memory
    ├── interest scoped memory
    ├── memory update requests
    ├── memory versions
    └── memory injection summary
```

Memory 维护职责：

- 维护全局 Memory；
- 维护兴趣分类 Memory；
- 接收历史反馈模块生成的 Memory 更新建议；
- 让用户确认、拒绝或修改 Memory 更新；
- 保留 Memory 版本，支持后续回滚；
- 为今日看什么、今日发芽和写作苗圃提供可注入的 Memory 摘要。

P0 不单独拆 `memory-service` 的原因：

- 用户画像和 Memory 强绑定；
- 当前只有单人开发，减少跨服务联调成本；
- 写作苗圃只需要读取可展示的 Memory 摘要，不需要复杂 Memory 检索服务。

未来如果 Memory 需要向量检索、跨服务高频读写、审计回滚或跨产品复用，再独立拆成 `memory-service`。

# 六、观点种子模块

## 1. 观点种子定义

观点种子是用户看到内容后的思想沉淀。

它不是原文收藏，也不是 AI 总结，而是：

```
内容来源
+
用户反应
+
初步观点
+
可写方向
+
需要补充材料
```

---

## 2. 数据结构

```
interfaceIdeaSeed {
  seedId:string;
  userId:string;
  interestId:string;

  title:string;
  sourceType:'hot_list'|'zhihu_search'|'global_search'|'following'|'ring'|'manual';
  sourceTitle:string;
  sourceSummary:string;
  sourceUrl?:string;

  userReaction:'agree'|'disagree'|'question'|'supplement'|'want_to_write';
  userNote:string;

  coreClaim:string;
  possibleAngles:string[];
  counterArguments:string[];
  requiredMaterials:string[];

  status:'dormant'|'water_needed'|'sproutable'|'high_timeliness'|'writing'|'published'|'expired';

  maturityScore:number;
  activationScore?:number;

  createdAt:string;
  updatedAt:string;
}
```

---

## 3. 种子状态

| 状态 | 含义 |
| --- | --- |
| dormant | 休眠中，有想法但暂时不适合写 |
| water_needed | 需要补资料、补案例、补反方 |
| sproutable | 已经具备写作条件 |
| high_timeliness | 当前热点强相关，建议尽快写 |
| writing | 已进入写作流程 |
| published | 已发布 |
| expired | 时效已过，但可沉淀为长期观点 |

# 七、今日发芽模块

## 1. 模块定位

今日发芽负责计算：

```
哪些历史观点种子
在今天的新热点、新讨论、新资料下
重新具备写作价值
```

---

## 2. 重要成本设计

今日发芽涉及复杂计算和 Token 消耗，因此不应自动执行。

### 正确交互

```
今日发芽页面初始状态
↓
展示小刘看山角色
↓
小刘看山提示：
“今天要不要给你的种子浇浇水？我可以帮你看看有没有值得发芽的想法。”
↓
用户点击「开始今日发芽」
↓
后端开始执行发芽计算
↓
返回发芽机会卡片
```

---

## 3. 发芽计算输入

```
interfaceSproutRequest {
  userId:string;
  interestId?:string;
  seedIds?:string[];
  timeRangeHours:number;
  includeHotList:boolean;
  includeZhihuSearch:boolean;
  includeGlobalSearch:boolean;
  includeFollowingFeed:boolean;
}
```

---

## 4. 发芽计算流程

```
1. 读取用户当前兴趣分类画像
2. 读取该兴趣分类下的观点种子
3. 拉取最近 N 小时热榜 / 搜索 / 关注流内容
4. 对内容进行摘要和主题提取
5. 对观点种子和新内容进行语义匹配
6. 计算 Activation Score
7. 生成 Top N 发芽机会
8. 生成写作建议标题、角度、补充材料
```

---

## 5. Activation Score

```
Activation Score =
热点相关性
+ 新信息增益
+ 用户兴趣匹配度
+ 种子成熟度
+ 时效性
+ 写作潜力
- 过度重复惩罚
- 证据不足惩罚
```

---

## 6. 发芽机会数据结构

```

    relevance:number;

```

# 八、写作苗圃模块

## 1. 写作流程

```
1. 选择观点种子
2. 确认核心观点
3. 选择文章类型
```

# 九、圆桌审稿模块

## 1. 设计目标

圆桌审稿把逻辑压测和人味检查统一为多 Agent 评审。

## 2. Agent 角色

```
typeReviewAgentRole=
|'logic_reviewer'
|'human_editor'
|'opponent_reader'
|'community_editor';
```

| Agent | 检查内容 |
| --- | --- |
| logic_reviewer | 论证跳跃、未证明前提、概念偷换 |
| human_editor | 个人经历、真实细节、AI 味、表达主体性 |
| opponent_reader | 反方视角、攻击点、读者质疑 |
| community_editor | 标题、开头、传播性、知乎语境 |

---

## 3. 审稿结果结构

```
interfaceAgentReview {
  role:ReviewAgentRole;
  summary:string;
  problems:string[];
  suggestions:string[];
  severity:'low'|'medium'|'high';
}
```

# 十、历史反馈模块

## 1. 模块定位

历史反馈模块负责把发布后的互动数据重新转化为创作资产。

它不是单纯看数据，而是要提取：

```
读者喜欢什么
读者反对什么
评论补充了什么
哪些内容可以成为下一颗种子
哪些写作风险要写入 Memory
```

---

## 2. 数据结构

```
interfaceArticleFeedback {
  articleId:string;
  userId:string;
  interestId:string;

  title:string;
  publishUrl:string;
  publishedAt:string;

  viewCount:number;
  likeCount:number;
  favoriteCount:number;
  commentCount:number;
  completionRate?:number;

  positiveFeedback:string[];
  negativeFeedback:string[];
  counterArguments:string[];
  supplementaryMaterials:string[];

  generatedSeeds:IdeaSeed[];

}
```

---

## 3. 反馈回流

反馈回流到三个地方：

```
1. 用户兴趣画像
2. 观点种子库
3. 写作风险提醒
```

例如：

```
读者反馈：
文章偏框架化，缺少真实失败案例。

回流 Memory：
在 Agent 工程化分类下，写作时需要优先补充真实失败案例。
```

当前原型展示了多个兴趣小类内容，但最终产品中应该改为：

```
兴趣小类、关注流、偶遇输入作为平级切换入口
点击某一分类
只展示该分类下的内容
```

不要在一个页面塞满所有分类。

---

## 2. 今日发芽：用户主动触发

今日发芽涉及复杂计算和 Token 消耗，不应自动运行。

正确交互是：

```
小刘看山提示
↓
用户点击开始发芽
↓
系统计算
↓
展示发芽机会
```

---

## 3. 用户画像：按兴趣分类绑定

用户画像不应只有全局画像。

需要支持：

```
全局画像
+
兴趣分类画像
```

同一个用户在不同兴趣下可能有不同：

- 知识水平；
- 表达风格；
- 推荐偏好；
- 证据偏好；
- 视角需求；
- 写作风险提醒。

---

## 4. 圆桌审稿：替代单独逻辑压测和人味检查

逻辑压测和人味检查不作为两个割裂步骤，而是在“圆桌审稿会”中由多个 Agent 共同评判。

---

## 5. 定稿草案：用户修改后发布

系统生成的是最终草稿，不是自动发布稿。

必须提示：

```
请用户修改后再发布。
```

这能保留用户主体性，也能避免纯 AI 内容感。
