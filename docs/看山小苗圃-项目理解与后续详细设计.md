# 看山小苗圃-项目理解与后续详细设计

本文档用于沉淀当前阶段对“看山小苗圃”的统一理解，并补齐现有文档中尚未展开的 Agent 设计、推流算法、发芽算法、写作编排和反馈回流设计。

本文档不是接口实现说明，也不是代码任务清单。后续开发时，它应作为产品和架构总纲，`产品文档`、`技术文档`、`接口功能与数据流文档`、`服务拆分与开发协作` 可以继续作为局部文档。

---

## 1. 项目定位

看山小苗圃不是普通 AI 写作工具，也不是单纯内容推荐系统。

它要解决知乎创作中的三件事：

```text
看什么
想什么
写什么
```

对应产品主线：

```text
看到好内容
↓
形成好观点
↓
写出好表达
```

完整闭环：

```text
高质量输入筛选
↓
阅读反应记录
↓
观点种子沉淀
↓
材料浇水和观点探讨
↓
热点/新信息触发发芽
↓
写作苗圃逐步成文
↓
圆桌审稿压测
↓
发布反馈回流
```

核心原则：

1. 输入先于输出：没有好输入，就很难有好表达。
2. 观点先于文章：文章不是字数堆叠，而是观点、证据、经历和反方回应的组织。
3. AI 代理整理，不代理立场：AI 可以帮用户看清楚、想清楚、说清楚，但不能替用户决定相信什么。
4. Memory 可见可改：用户必须知道系统为什么这样推荐、这样写、这样提醒。
5. 成本可控：昂贵的搜索、发芽和写作任务必须有缓存、触发条件和用户主动操作。

## 2. 当前文档需要统一的关键判断

### 2.1 今日看什么的展示层级

输入分类必须平级展示：

```text
兴趣小类
关注流
偶遇输入
```

关注流不是某个兴趣小类的子模块，偶遇输入也不是兴趣小类下的“拓展内容”。它们都是和兴趣小类平级的输入入口。

正确结构：

```text
今日看什么
├── Agent 工程化
├── AI Coding
├── RAG / 检索
├── 后端工程
├── 程序员成长
├── 金融风控
├── 医学 AI
├── 产品设计
├── 内容创作
├── 关注流精选
└── 偶遇输入
```

用户点击某个分类后，只展示当前分类下的卡片，不在一个页面同时塞满所有分类。

### 2.2 种子库不是今日发芽入口

早期文档中出现过种子卡片上的“今日发芽”按钮，但后续产品判断已经调整。

最终理解：

- 种子库负责种子管理、继续浇水、开始写作、合并相似种子。
- 今日发芽是独立页面或独立入口，由用户主动触发一次发芽计算。
- 种子库里的单颗种子可以被写作，也可以被浇水，但不直接承担“今日发芽”全局计算入口。

### 2.3 原始来源必须可见

值得看卡片不能只展示 AI 摘要。

每张卡片都必须保留原始来源，来源可以来自多篇内容。真实 API 场景下，来源内容应由知乎热榜、知乎搜索、全网搜索、关注流、圈子、评论、故事等接口返回。

前端展示方式：

- 来源不应只跳转 mock 链接。
- 来源应在卡片内展开，固定高度滚动展示完整内容。
- `url` 或 `sourceId` 只作为可追溯标识。

### 2.4 Memory 归属

P0 阶段不拆独立 `memory-service`。

Memory 归属：

```text
profile-service
├── global memory
├── interest scoped memory
├── memory update requests
├── memory versions
└── memory injection summary
```

后续只有在出现向量检索、跨服务高频读写、审计回滚、跨产品复用时，才考虑拆出 `memory-service`。

### 2.5 历史反馈不直接更新 Memory

历史反馈模块只能生成 `memory_update_requests`。

用户必须在画像页确认、修改或拒绝后，才写入长期 Memory。

### 2.6 疑问是多轮线程

“有疑问”不是一次性按钮。

正确流程：

```text
用户提出问题
↓
Agent 基于种子、卡片来源、历史追问回答
↓
生成 SeedQuestion
↓
生成 open_question 材料
↓
生成 evidence 或 counterargument 材料
↓
用户选择：
  标记已解决：线程结束
  仍需补资料：保留为待解决问题，可继续追问
  继续追问：新问题关联 parentQuestionId
```

## 3. 系统分层

从产品语义看，系统分为五层。

```text
输入层
↓
理解层
↓
观点层
↓
表达层
↓
反馈层
```

### 3.1 输入层

负责获取内容：知乎热榜、知乎搜索、全网搜索、关注流、圈子帖子与评论、故事与知识内容、用户手动输入。

主要服务：`zhihu-adapter`、`content-service`。

### 3.2 理解层

负责把原始内容变成可读、可判断、可写的结构：摘要、原始来源解释、主要争议、可写角度、来源贡献、质量评分。

主要服务：`content-service`、`llm-service`。

### 3.3 观点层

负责沉淀用户真实反应：认同、反对、有疑问、补充、想写、继续浇水、合并相似种子。

主要服务：`seed-service`、`profile-service`、`llm-service`。

### 3.4 表达层

负责从成熟观点到文章：核心观点确认、Memory 注入、论证蓝图、初稿、圆桌审稿、定稿草案、模拟发布。

主要服务：`writing-service`、`llm-service`、`seed-service`、`profile-service`。

### 3.5 反馈层

负责把发布后的数据变成长期资产：阅读表现、点赞收藏、评论摘要、反方观点、补充材料、二次文章建议、Memory 更新建议。

主要服务：`feedback-service`、`zhihu-adapter`、`llm-service`、`profile-service`。

## 4. 核心 Agent 设计

这里的 Agent 不一定都需要独立部署成服务。P0 可以由 `llm-service` 通过不同 task type 和 prompt 模板实现。后续复杂后再拆分。

### 4.1 输入策展 Agent

目标：决定今天应该看什么。

输入：用户兴趣 Memory、全局 Memory、热榜候选、知乎搜索候选、全网搜索候选、关注流候选、偶遇输入候选。

输出：Query Plan、候选来源池、值得看卡片、推荐理由、来源贡献解释。

不做：不创建观点种子，不替用户表态，不直接生成文章。

### 4.2 来源证据 Agent

目标：保证所有 AI 摘要和观点都有来源可追溯。

输出：每个来源的原文要点、来源贡献、可引用证据、反方材料、来源风险提示。

规则：

- 热榜可作为时效背景，但不一定是强证据。
- 知乎高赞回答可作为观点材料，但要保留作者和互动信号。
- 全网搜索可补外部证据，但要标记来源权威度。
- 故事内容只能进入创意表达或偶遇输入，不作为严肃事实论证证据。

### 4.3 观点种子 Agent

目标：把阅读动作转化为观点种子。

触发：我认同、我反对、有疑问、加入种子库、基于它写一篇。

输入：WorthReadingCard、用户 reaction、用户 note、用户兴趣 Memory。

输出：IdeaSeed、初始 evidence、初始 counterargument、requiredMaterials。

关键规则：

- 同一个 `cardId` 重复操作不应创建多个种子。
- 用户立场必须保存在 seed 中。
- AI 只能辅助生成 `coreClaim`，不能覆盖用户 note。

### 4.4 疑问求证 Agent

目标：把“有疑问”变成可追问、可补资料、可结束的线程。

输入：SeedQuestion、IdeaSeed、card.originalSources、历史 question thread、已有 wateringMaterials。

输出：Agent answer、citedSourceIds、statusRecommendation、new evidence / counterargument / open_question material、followUpQuestions。

状态：

```text
answered
resolved
needs_material
```

多轮规则：

- `resolved` 后该问题线程结束。
- `needs_material` 后下次追问必须带上 parentQuestionId 和上下文。
- 新问题应新建 thread，避免把不同疑问混在一起。

### 4.5 浇水材料 Agent

目标：让种子从“有想法”变成“可写”。

四格材料板：

| 类型 | 说明 | Agent 行为 |
| --- | --- | --- |
| 事实证据 | 数据、来源、案例、原文摘要 | Agent 补证据 |
| 反方质疑 | 反对意见、漏洞、边界 | Agent 找反方 |
| 个人经验 | 用户自己的项目、观察、踩坑 | Agent 只能提问引导，不能编造 |
| 待解决问题 | 尚未回答的疑问 | Agent 问答和追问 |

成熟度公式：

```text
maturityScore =
min(96, 32 + 已采纳材料类型数 * 14 + 疑问总数 * 3 + 已解决疑问数 * 4)
```

后续可升级：增加材料质量分、来源可信度分、个人经验权重、反方回应完整度。

### 4.6 发芽 Agent

目标：判断哪颗历史种子在今天值得写。

触发：用户点击“开始今日发芽”。后续可扩展为每日一次提醒，但 P0 不自动执行。

输入：未发布种子、种子成熟度、种子材料、种子疑问、兴趣 Memory、热榜缓存、搜索缓存、关注流缓存、新信息增益。

输出：SproutOpportunity、activationScore、whyWorthWriting、suggestedTitle、suggestedAngle、suggestedMaterials。

### 4.7 写作教练 Agent

目标：从观点种子生成可编辑 writing session。

子任务：核心观点确认、文章类型选择、语气选择、论证蓝图、初稿、定稿草案。

关键规则：

- 创建 session 时必须注入兴趣 Memory。
- Memory 注入必须前端可见。
- 用户可以修改 `memoryOverride`。
- `memoryOverride` 只影响当前 session，不覆盖 profile Memory。

### 4.8 圆桌审稿 Agent 组

目标：做逻辑压测、人味检查、反方攻击和社区表达优化。

| 角色 | 职责 |
| --- | --- |
| 逻辑审稿 Agent | 检查论证跳跃、未证明前提、概念偷换 |
| 人味编辑 Agent | 检查个人经历、真实细节、AI 味 |
| 反方读者 Agent | 从反对者角度攻击漏洞 |
| 社区传播 Agent | 检查标题、开头、知乎语境 |

输出：问题列表、修改建议、严重级别、可采纳建议。

### 4.9 反馈分析 Agent

目标：把发布后的互动数据转成新资产。

输入：文章指标、评论列表、赞同/收藏/回复、圈子反馈。

输出：表现摘要、支持观点、反方观点、补充材料、二次文章建议、Memory 更新建议。

边界：不直接更新 Memory，不直接改写原文章，不自动发布二次文章。

## 5. 今日看什么推流算法

这里的“推流”不是传统推荐流，而是“创作输入流”。目标不是提高停留时长，而是提高有效创作输入质量。

### 5.1 输入分类

```text
interest: 兴趣小类
following: 关注流精选
serendipity: 偶遇输入
```

每次只展示一个分类下的内容。

### 5.2 候选召回

兴趣小类输入：interest Memory、global Memory、用户历史种子、热榜、知乎搜索、全网搜索。

Query Plan 由以下信息组成：兴趣关键词、用户近期观点关键词、反方关键词、时效关键词。

关注流输入：关注动态、关注列表、粉丝列表、用户兴趣 Memory、已有种子主题。

关注流召回重点：关注作者的新观点、和已有种子相关的讨论、评论密度高的内容、可补充材料的内容。

偶遇输入输入：热榜、全网搜索、故事 API、与兴趣相邻但不完全重合的主题。

偶遇输入召回重点：远端关联、跨域类比、新信息增益、能打破同温层但不完全随机。

### 5.3 卡片评分

基础公式：

```text
relevanceScore =
兴趣匹配 35%
+ authorityLevel / rankingScore 20%
+ VoteUpCount / CommentCount 15%
+ 争议点数量 15%
+ EditTime / 热榜排序 10%
+ 可写角度质量 5%
```

建议扩展为：

```text
cardScore =
interestFit * 0.28
+ sourceQuality * 0.18
+ freshness * 0.12
+ controversy * 0.14
+ writingPotential * 0.16
+ memoryFit * 0.08
+ diversityGain * 0.04
```

说明：

- `interestFit`：与当前兴趣 Memory 的匹配。
- `sourceQuality`：权威度、赞同、评论、来源可追溯性。
- `freshness`：热榜顺位、发布时间。
- `controversy`：是否存在明确正反观点。
- `writingPotential`：是否能生成清晰主张和可写角度。
- `memoryFit`：是否符合用户写作偏好和证据偏好。
- `diversityGain`：是否能避免当前页面内容同质化。

### 5.4 多样性和去重

同一批卡片需要做去重和多样性控制：

- 同一 `sourceId` 不重复展示。
- 同一标题相似度高的内容合并来源。
- 同一来源类型不能完全垄断页面。
- 至少保留一张高争议卡片。
- 至少保留一张可写性高的卡片。
- 偶遇输入最多展示少量，不压过兴趣主线。

后续可用 MMR：

```text
selectedScore =
lambda * relevance
- (1 - lambda) * similarityToSelected
```

P0 可以用规则排序，P1 再引入向量相似度。

### 5.5 刷新输入

刷新只影响当前分类展示，不影响用户资产。

```text
当前 categoryId
↓
读取候选池
↓
轮换 / 重排
↓
更新 categoryRefreshState
```

刷新不影响：种子库、写作 session、用户画像、历史反馈。

## 6. 发芽算法详细设计

### 6.1 发芽不是推荐文章

今日发芽不是给用户推荐“今天热门文章”，而是找：

```text
历史观点种子
×
今日新信息
×
用户 Memory
```

之间的写作机会。

### 6.2 发芽触发

P0：用户手动点击“开始今日发芽”。

P1：每日首次打开时提示是否发芽，可选只对成熟种子计算。

P2：定时提醒，但要有频控。

### 6.3 候选种子筛选

进入发芽计算的种子：

```text
status in ["water_needed", "sproutable", "high_timeliness"]
not published
maturityScore >= 45
updatedAt 不过旧，或有长期价值
```

优先种子：

- `maturityScore >= 70`
- 最近 14 天有浇水
- 和当前热点相关
- 有明确反方或未解决问题
- 和用户主兴趣匹配

### 6.4 新信息召回

每颗候选种子召回：热榜相关 topic、知乎搜索结果、全网搜索结果、关注流动态、圈子讨论。

召回时先走 Redis 短缓存。

### 6.5 Activation Score

基础公式：

```text
activationScore =
种子成熟度 25%
+ 热点相关性 25%
+ 时效性 20%
+ 新信息增益 15%
+ 写作潜力 15%
- 重复发布惩罚
- 证据不足惩罚
```

详细解释：

- 种子成熟度：来自 seed-service 的 maturityScore。
- 热点相关性：种子主题和今日热点的匹配。
- 时效性：热点是否发生在近期。
- 新信息增益：今天是否出现了过去没有的新证据、新反方、新案例。
- 写作潜力：是否能形成清晰标题、核心观点和论证蓝图。
- 重复发布惩罚：近期是否写过同主题。
- 证据不足惩罚：材料是否只有观点、没有证据。

### 6.6 发芽输出

每个机会必须解释：哪颗种子被激活、被什么热点或新信息触发、为什么现在值得写、建议标题、建议角度、还缺什么材料、是否建议立即写。

### 6.7 发芽动作

```text
补充资料
→ 写入四格浇水材料板

换个角度
→ 重新生成 suggestedAngle，不改 seed 主张

暂时不写
→ opportunity.status = dismissed

开始写作
→ 创建 writing session
```

## 7. 写作苗圃详细设计

### 7.1 写作不是一步生成

写作苗圃必须保持“用户确认逐步推进”：

```text
选择种子
↓
确认观点
↓
选择文章类型
↓
生成论证蓝图
↓
生成初稿
↓
圆桌审稿
↓
定稿草案
↓
模拟发布 / 用户发布
```

### 7.2 Memory 注入

创建 session 时：

```text
seed.interestId
↓
profile-service / memory/injection/{interestId}
↓
写入 writing_session.memoryOverride
↓
前端展示给用户
```

用户可以改 `memoryOverride`，但该修改只影响当前文章。

如果用户希望长期更新 Memory，应生成 `memory_update_request`。

### 7.3 文章类型

P0 支持：知乎回答、长文分析、经验复盘、观点评论、反驳文、想法 / pin。

不同文章类型影响：标题风格、开头方式、段落长度、证据密度、个人经验占比、反方回应位置。

### 7.4 人味检查

写作苗圃必须检查：是否缺少个人经历、是否缺少明确立场、是否只堆概念、是否没有反方回应、是否像 AI 模板文、是否没有具体案例。

### 7.5 AI 辅助声明

最终草稿应提示用户：

```text
本文使用 AI 辅助梳理论证结构，核心观点和最终表达由作者确认。
```

实际声明文案可由用户决定。

## 8. 反馈回流详细设计

### 8.1 数据来源

P0：mock 历史反馈、mock 评论摘要。

P1：圈子评论、文章评论、点赞、收藏、评论、阅读完成率等可得指标。

### 8.2 分析目标

反馈分析不是展示数据看板，而是回答：

```text
读者为什么喜欢
读者为什么反对
评论补充了什么
是否能生成二次种子
是否需要更新 Memory
```

### 8.3 回流路径

```text
评论/指标
↓
feedback-service
↓
llm-service 总结
↓
生成：
  二次文章建议
  二次种子 payload
  memory_update_requests
```

### 8.4 Memory 更新边界

必须遵守：

```text
feedback-service 不直接写 profile Memory
只创建 memory_update_request
用户 apply 后才写入
```

## 9. 服务边界总览

| 服务 | 负责 | 不负责 |
| --- | --- | --- |

## 12. 风险清单

### 12.1 产品风险

- 变成普通 AI 写作工具，而不是读写闭环。
- 推荐内容太像信息流，没有沉淀为观点。
- AI 生成太强，用户主体性变弱。
- 圆桌审稿只像形式，不真正发现问题。

### 12.2 工程风险

- 服务拆太细导致单人开发联调成本高。
- 各服务 DTO 不一致。
- 知乎 API 额度被重复请求消耗。
- LLM 输出不稳定，前端难渲染。

### 12.3 算法风险

- 偶遇输入变成随机内容。
- 发芽算法只追热点，不看种子成熟度。
- 卡片评分只看热度，不看可写性。
- Memory 注入隐藏在后台，用户无感。

### 12.4 演示风险

- Demo 链路太长，现场不稳定。
- 真实接口失败影响演示。
- 生成内容时间过长。
- AI 辅助声明和知乎社区规范没有体现。

## 13. Demo 主链路建议

黑客松演示优先展示一条闭环，不要展示所有能力。

推荐主链路：

```text
1. 进入个人画像，展示全局 Memory 和 AI Coding 兴趣 Memory
2. 今日看什么选择 AI Coding
3. 打开一张多来源卡片，展示原始来源
4. 点击有疑问，Agent 回答并写入浇水材料
5. 加入种子库，继续浇水
6. 今日发芽主动触发，生成一个发芽机会
7. 开始写作，展示 Memory 注入
8. 生成论证蓝图
9. 圆桌审稿指出缺少个人经历和反方回应
10. 定稿草案 + AI 辅助声明
11. 历史反馈生成二次种子和 Memory 更新建议
```

这条链路能表达核心差异：

```text
不是 AI 替你写
而是 AI 帮你从真实阅读中形成观点，再把观点发展成表达
```
