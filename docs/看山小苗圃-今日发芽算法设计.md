# 看山小苗圃-今日发芽算法设计

本文档定义 `sprout-service` 的目标算法、数据流、评分公式、LLM 调用点和接口验收标准。

今日发芽不是“每天随机推荐一个写作题目”，而是：

```text
历史观点种子
+ 今日新输入
+ 用户 Memory
+ 资料成熟度
→ 判断这颗种子今天是否值得写
```

核心目标：

1. 让旧种子被新热点、新材料、新反方重新激活。
2. 避免成熟度不足的种子被过早推去写作。
3. 让用户知道“为什么现在值得写”，而不是只看到一个标题。
4. 发芽结果必须可回写种子材料，并能一键进入写作苗圃。

---

## 1. 服务边界

### 1.1 sprout-service 负责

- 聚合候选种子、今日输入信号、兴趣 Memory。
- 计算 `activationScore`。
- 调用 `llm-service` 生成发芽解释、标题、角度和建议补充材料。
- 持久化 `sprout_runs` 和 `sprout_opportunities`。
- 处理发芽机会动作：补充资料、换角度、暂时不写、开始写作。

### 1.2 sprout-service 不负责

- 不直接调用知乎 API，必须通过 `content-service` 或 `zhihu-adapter` 已缓存结果。
- 不直接调用模型供应商，必须通过 `llm-service`。
- 不修改长期 Memory。
- 不发布文章。
- 不替用户决定立场，只判断“是否值得进入写作流程”。

---

## 2. 触发方式

### 2.1 手动触发

前端点击：

```text
POST /api/v1/sprout/start
```

适合 Demo 和用户主动查看。

### 2.2 内容刷新后触发

当 `content-service` 完成“今日看什么”刷新后，可异步触发：

```text
content refreshed
→ sprout-service start_run(trigger=content_refresh)
```

P0 可以不自动触发，只在用户点击“开始今日发芽”时使用最新缓存。

### 2.3 种子材料更新后触发

当用户继续浇水、Agent 补证据、Agent 找反方后，种子成熟度可能跨过阈值：

```text
seed.material_added
→ seed.maturityScore >= 70
→ sprout-service 可将该 seed 放入下一次候选集
```

P0 不需要实时推送，只需保证下一次发芽计算能读取更新后的种子。

## 3. 输入数据

### 3.1 种子输入

来源：`seed-service`

```ts
IdeaSeed {
  id: string
  title: string
  interestId: string
  status: "water_needed" | "sproutable" | "writing" | "published"
  maturityScore: number
  coreClaim: string
  userNote: string
  counterArguments: string[]
  questions: SeedQuestion[]
  wateringMaterials: WateringMaterial[]
  createdFromCardId?: string
}
```

候选条件：

```text
status in ["water_needed", "sproutable"]
maturityScore >= 55
没有正在写作中的 writingSession
最近 24 小时没有被用户 dismiss
不是已发布 seed
```

说明：

- `maturityScore >= 70` 的种子优先。
- `55 <= maturityScore < 70` 的种子仍可进入候选，但需要更强的热点/新材料信号，并在机会中提示“建议先补资料”。

### 3.2 今日输入信号

来源：`content-service` 已缓存卡片和 `zhihu-adapter` Redis 短缓存。

```text
hot_list cached result
zhihu_search cached result
global_search cached result
following moments
today WorthReadingCard[]
```

不得在 sprout-service 内直接回源知乎 API。

### 3.3 用户 Memory

来源：`profile-service`

```ts
GlobalMemory
MemorySummary for seed.interestId
```

用途：

- 判断该种子是否符合用户长期写作方向。
- 生成更贴合用户风格的发芽解释。
- 提醒写作风险，例如“缺少个人经验”“不要只讲大道理”。

### 3.4 历史反馈

来源：`feedback-service`

P0 可选。若已接入，用于避免重复写同类题目：

```text
最近 30 天已发布文章
标题相似度
读者反馈信号
二次文章建议
```

## 4. 总体流程

```text
POST /sprout/start
↓
1. 读取候选种子
↓
2. 读取今日输入信号
↓
3. 为每颗种子生成 trigger candidates
↓
4. 计算 deterministic activationScore
↓
5. 取 Top K 调用 llm-service/sprout-opportunities
↓
6. 合并 LLM 输出和确定性分数
↓
7. 写入 sprout_run / sprout_opportunity
↓
8. 返回前端
```

## 5. 候选种子筛选

### 5.1 基础过滤

```python
def is_seed_candidate(seed):
    if seed.status in ["writing", "published"]:
        return False
    if seed.maturityScore < 55:
        return False
    if seed.dismissedAt within last 24h:
        return False
    return True
```

### 5.2 必要字段补齐

如果种子缺少 `interestId`：

```text
1. 优先使用 createdFromCardId 对应 card.categoryId。
2. 否则用 title/coreClaim 和兴趣名称做文本匹配。
3. 仍无法判断，则进入 low-confidence 队列，不推荐进 Top K。
```

### 5.3 写作冷却

避免用户连续看到同类机会：

```text
same seed dismissed within 24h → 排除
same seed already writing → 排除
same coreClaim published within 30d → 降权 30
same suggestedTitle appeared in latest run → 降权 15
```

## 6. 今日输入信号生成

### 6.1 TriggerTopic

每个 trigger 来源于一个或多个内容项：

```ts
TriggerTopic {
  id: string
  sourceTypes: ("hot_list" | "zhihu_search" | "global_search" | "following" | "content_card")[]
  title: string
  summary: string
  sourceIds: string[]
  heatScore?: number
  relevanceScore?: number
  authorityScore?: number
  publishedAt?: string
  matchedKeywords: string[]
  noveltyHints: string[]
}
```

### 6.2 Trigger 来源

| 来源 | 用途 | 优先级 |
| --- | --- | --- |
| 热榜 | 时效触发、公共讨论 | 高 |
| 知乎搜索 | 站内优质讨论、同题论据 | 高 |
| 全网搜索 | 外部证据、新信息增益 | 中 |
| 关注流 | 用户关系网络、关注作者新观点 | 中 |
| 今日看什么卡片 | 已聚合并被 LLM 处理过的高质量输入 | 高 |

### 6.3 触发主题合并

同一个 seed 可能被多个输入触发，需要合并：

```text
same normalized title
OR sourceId 相同
OR title token Jaccard >= 0.55
OR LLM 判定为同一讨论主题
```

P0 先用确定性规则：

```text
中文分词/字符 bigram
+ 去停用词
+ Jaccard 相似度
+ sourceId 去重
```

LLM 只用于 Top N 候选的最终解释，不用于全量聚类。

## 7. 评分公式

最终分：

```text
activationScore =
  seedReadinessScore * 0.25
+ trendRelevanceScore * 0.25
+ timelinessScore * 0.20
+ newInformationGainScore * 0.15
+ writingPotentialScore * 0.15
- duplicatePenalty
- evidencePenalty
- riskPenalty
```

范围：`0 - 100`。

推荐阈值：

```text
score >= 78 → 强推荐
68 <= score < 78 → 可发芽
58 <= score < 68 → 建议先补资料
score < 58 → 不展示
```

### 7.1 seedReadinessScore

输入：`maturityScore`、材料类型、已解决疑问。

```text
seedReadinessScore =
  maturityScore * 0.70
+ adoptedMaterialCoverage * 20
+ resolvedQuestionBonus
```

`adoptedMaterialCoverage`：

```text
已采纳 evidence → +0.25
已采纳 counterargument → +0.25
已采纳 personal_experience → +0.30
open_question 都有 answered/resolved → +0.20
```

`resolvedQuestionBonus`：

```text
min(8, resolved_question_count * 2)
```

上限：100。

### 7.2 trendRelevanceScore

判断今日输入和种子是否相关。

```text
trendRelevanceScore =
  keywordMatch * 35
+ semanticMatch * 35
+ sourceTopicMatch * 20
+ memoryInterestMatch * 10
```

P0：

- `keywordMatch`：`title/coreClaim/materials` 与 trigger title/summary 的 token overlap。
- `semanticMatch`：无 embedding 时用 LLM 对 Top 候选返回 `fitScore`；没有 LLM 时用 `keywordMatch` 近似。
- `sourceTopicMatch`：trigger 来自同一兴趣分类或同一 `WorthReadingCard`。
- `memoryInterestMatch`：trigger 与 `MemorySummary.interestName/preferredPerspective` 匹配。

### 7.3 timelinessScore

判断“是不是现在写”。

```text
hot_list item:
  timelinessScore = heatRankScore

search/global item:
  timelinessScore = recencyScore * 0.7 + interactionFreshness * 0.3

following:
  timelinessScore = recentMomentScore
```

建议计算：

```text
热榜第 1 名 100 分，之后每位 -2，最低 40。
发布时间 24h 内 100，3 天内 80，7 天内 60，30 天内 35，更早 15。
无时间字段时默认 50。
```

### 7.4 newInformationGainScore

判断今天有没有“新材料”能让旧种子更值得写。

```text
newInformationGainScore =
  newSourceRatio * 35
+ newCounterArgument * 25
+ newEvidenceQuality * 25
+ crossSourceDiversity * 15
```

规则：

- 若 trigger sourceId 已在 seed.sources 或 seed.materials 中出现，`newSourceRatio` 降低。
- 若 trigger 能补充反方观点，`newCounterArgument` 提高。
- 若来源 `authorityLevel/rankingScore/VoteUpCount/CommentCount` 较高，`newEvidenceQuality` 提高。
- 若同时来自热榜 + 知乎搜索 / 全网搜索 / 关注流，`crossSourceDiversity` 提高。

### 7.5 writingPotentialScore

判断是否适合写成知乎内容。

```text
writingPotentialScore =
  claimClarity * 25
+ controversyStrength * 25
+ personalExperienceFit * 20
+ articleFormFit * 15
+ communityFit * 15
```

解释：

- `claimClarity`：种子是否有明确核心观点。
- `controversyStrength`：是否有正反争议。
- `personalExperienceFit`：用户是否有可补充经历。
- `articleFormFit`：能否形成回答/长文/经验复盘。
- `communityFit`：是否符合知乎讨论语境，避免纯营销或低质量 AI 味。

P0 可由规则 + LLM TopK 共同计算：

```text
规则先筛，LLM 只给 Top 8 候选补充 fitScore 和 whyWorthWriting。
```

## 8. 惩罚项

### 8.1 duplicatePenalty

```text
同 seed 最近 24h dismiss: 排除
相似标题 7d 内出现: -15
相似文章 30d 内 published: -30
同 trigger 已生成 opportunity: -10
```

### 8.2 evidencePenalty

```text
没有 evidence: -12
没有 counterargument: -8
没有 personal_experience: -6
maturityScore < 70: -10
```

### 8.3 riskPenalty

```text
医学/金融/法律等高风险话题且缺少权威来源: -15
仅热榜无补充来源: -8
LLM 判断容易变成无营养跟风: -10
可能需要 AI 辅助声明但用户未确认发布策略: -5
```

## 9. LLM 调用

### 9.1 调用时机

为了节省额度：

```text
全量候选只做规则评分
规则分 Top 8 进入 LLM
LLM 产出最多 Top 5 opportunities
```

### 9.2 任务类型

```text
POST /llm/tasks/sprout-opportunities
```

输入：

```json
{
  "taskType": "sprout-opportunities",
  "input": {
    "seed": {},
    "seedMaterials": [],
    "questions": [],
    "memory": {},
    "triggerTopics": [],
    "deterministicScores": {},
    "constraints": {
      "doNotInventPersonalExperience": true,
      "mustExplainWhyNow": true,
      "mustReturnMaterialsGap": true
    }
  },
  "promptVersion": "v1",
  "schemaVersion": "v1"
}
```

输出：

```json
{
  "opportunities": [
    {
      "seedId": "seed-xxx",
      "triggerTopic": "AI 编程工具影响程序员成长路径",
      "whyWorthWriting": "今日热榜和站内讨论都在讨论初级程序员价值，正好能激活该种子里的工程判断观点。",
      "suggestedTitle": "AI 时代，程序员真正需要训练的是什么？",
      "suggestedAngle": "从机械刷题转向问题建模和工程判断。",
      "suggestedMaterials": "补充一次你使用 AI 写代码时仍需要人工判断的项目经历。",
      "fitScore": 86,
      "materialGaps": ["personal_experience"],
      "riskWarnings": ["需要补充真实经历，避免泛泛谈趋势"]
    }
  ]
}
```

### 9.3 LLM 输出校验

必须校验：

- `seedId` 必须来自输入候选。
- `suggestedTitle` 非空，长度建议 8-40 中文字。
- `whyWorthWriting` 必须解释“为什么现在写”。
- 不得编造用户经历。
- 如果 `materialGaps` 包含 `personal_experience`，前端应提示用户补充。

LLM 失败时：

```text
使用 deterministic candidate 生成降级 opportunity
fallback=true
whyWorthWriting = 基于规则分最高的 trigger 自动生成
```


新增字段可以先进入后端，不强制前端立即展示。

## 12. 发芽机会动作

### 12.1 补充资料

接口：

```text
POST /sprout/opportunities/{opportunity_id}/supplement
```

流程：

```text
1. 读取 opportunity。
2. 如果 payload.material 存在，直接写入。
3. 否则调用 llm-service/supplement-material。
4. 返回 seedMaterial，前端/seed-service 写入四格材料板。
5. opportunity.status = supplemented。
```

建议回写类型：

```text
materialGaps 包含 evidence → evidence
materialGaps 包含 counterargument → counterargument
否则默认 evidence
```

### 12.2 换个角度

接口：

```text
POST /sprout/opportunities/{opportunity_id}/switch-angle
```

流程：

```text
1. 输入用户偏好：更犀利 / 更稳健 / 反方视角 / 经验复盘。
2. 调 llm-service/generate-writing-angles。
3. 更新 suggestedTitle / suggestedAngle。
4. 保留 score 不变，但 status = angle_changed。
```

### 12.3 暂时不写

接口：

```text
POST /sprout/opportunities/{opportunity_id}/dismiss
```

流程：

```text
1. opportunity.status = dismissed。
2. 记录 dismissedAt。
3. 同 seed 24h 内不再进入推荐。
```

### 12.4 开始写作

接口：

```text
POST /sprout/opportunities/{opportunity_id}/start-writing
```

输出：

```ts
writingHandoff {
  seedId: string
  interestId: string
  coreClaim: string
  suggestedTitle: string
  suggestedAngle: string
  suggestedMaterials: string
}
```

后续：

```text
api-gateway → writing-service 创建 WritingSession
writing-service → profile-service 读取 Memory 注入
```

## 14. 实现优先级

### P0

- 候选种子过滤。
- 从 content-service 缓存读取今日卡片和热榜信号。
- 规则评分。
- Top 5 opportunity 输出。
- opportunity actions。
- Redis 当日缓存。

### P1

- 接入 `llm-service/sprout-opportunities`。
- scoreBreakdown 暴露给前端。
- materialGaps / riskWarnings。
- 关注流触发信号。

### P2

- 历史反馈驱动的重复发布惩罚。
- 更复杂的主题聚类。
- 用户级个性化阈值。
- 多天趋势追踪。

## 15. 验收用例

1. `maturityScore < 55` 的种子不产生机会。
2. `published` / `writing` 种子不产生机会。
3. 同一个 seed 被 dismiss 后 24h 内不再推荐。
4. 热榜变化后，相关 seed 的 `timelinessScore` 变化。
5. 新来源进入后，`newInformationGainScore` 上升。
6. 缺少 evidence/counterargument 的 seed 有 `evidencePenalty`。
7. LLM 失败时仍返回 fallback opportunity。
8. `start-writing` 返回完整 handoff。
9. `supplement` 返回可写入四格浇水板的 material。
10. 同输入二次调用命中 Redis 缓存。
