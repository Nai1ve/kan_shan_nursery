# 看山小苗圃 - 实际部署清单与一键脚本

更新时间：2026-05-14

本文只描述当前项目在 Docker Compose 部署时**实际会读取到的配置**，用于线上部署、排错和交给其他模型/人工执行。

---

## 1. 最重要的结论

当前 Docker 镜像不会自动复制本地的：

```text
services/config.yaml
```

因此线上容器内的主要配置来源是：

```text
infra/.env
↓
infra/docker-compose.yml environment
↓
packages/shared-python/kanshan_shared/config.py 的 load_config()
```

也就是说，线上改 `services/config.yaml` 通常不会影响容器，除非你额外挂载它并设置：

```bash
KANSHAN_CONFIG_PATH=/path/to/config.yaml
```

当前推荐不要走挂载配置文件，统一使用：

```text
infra/.env
```

这样最稳定，也最容易排查。

## 2. 当前代码实际读取的配置项

### 2.1 全局配置

由 `load_config()` 读取：

| 环境变量 | 用途 | 必填 |
| --- | --- | --- |
| `PROVIDER_MODE` | `mock` / `live`，控制知乎 API 是否回源 | 是 |
| `STORAGE_BACKEND` | `memory` / `postgres` | 是 |
| `DATABASE_URL` | PostgreSQL 连接串 | postgres 模式必填 |
| `REDIS_URL` | Redis 连接串 | 推荐必填 |
| `CORS_ORIGINS` | api-gateway CORS 白名单 | 跨域访问时必填 |
| `OAUTH_STATE_SECRET` | OAuth state 签名密钥 | 生产必填 |

### 2.2 前端配置

前端有两类配置：

| 变量 | 读取位置 | 用途 |
| --- | --- | --- |
| `NEXT_PUBLIC_KANSHAN_BACKEND_MODE` | 前端构建期 | 固定为 `gateway` |
| `NEXT_PUBLIC_KANSHAN_GATEWAY_URL` | 前端构建期 / 浏览器端 | 浏览器直接请求 gateway 的公网地址 |
| `NEXT_PUBLIC_ZHIHU_ADAPTER_URL` | 前端构建期 | 当前正式 OAuth 主链路不直接依赖 |
| `NEXT_PUBLIC_ZHIHU_OAUTH_AUTO_START` | 前端构建期 | `1` 自动拉起知乎授权，`0` 只手动触发；本地建议 `0` |
| `KANSHAN_GATEWAY_URL` | Next.js rewrites | 前端容器内 `/api/*` 代理目标 |

关键点：

```text
NEXT_PUBLIC_KANSHAN_GATEWAY_URL = 浏览器能访问的地址
KANSHAN_GATEWAY_URL = frontend 容器能访问的 api-gateway 地址
```

当前 Docker Compose 中应为：

```text
NEXT_PUBLIC_KANSHAN_GATEWAY_URL=http://113.44.217.169:8000
KANSHAN_GATEWAY_URL=http://api-gateway:8000
```

否则 OAuth callback 页面里的相对请求 `/api/v1/auth/zhihu/callback` 可能会被 Next 代理到错误地址。

### 2.3 服务间 URL

以下变量必须在容器内使用 Docker 服务名，不能用 `127.0.0.1`：

```bash
PROFILE_SERVICE_URL=http://profile-service:8010
CONTENT_SERVICE_URL=http://content-service:8020
SEED_SERVICE_URL=http://seed-service:8030
SPROUT_SERVICE_URL=http://sprout-service:8040
WRITING_SERVICE_URL=http://writing-service:8050
FEEDBACK_SERVICE_URL=http://feedback-service:8060
ZHIHU_ADAPTER_URL=http://zhihu-adapter:8070
LLM_SERVICE_URL=http://llm-service:8080
```

特别是：

```text
profile-service 必须拿到 ZHIHU_ADAPTER_URL=http://zhihu-adapter:8070
```

否则会出现：

```text
ConnectionRefusedError: [Errno 111] Connection refused
```

原因是容器内 `127.0.0.1:8070` 指向 profile-service 自己，不是 zhihu-adapter。

### 2.4 知乎配置

`zhihu-adapter` 实际读取：

| 环境变量 | 用途 |
| --- | --- |
| `ZHIHU_APP_KEY` | Community Client app key |
| `ZHIHU_APP_SECRET` | Community Client app secret |
| `ZHIHU_COMMUNITY_BASE_URL` | 默认 `https://openapi.zhihu.com` |
| `ZHIHU_DEFAULT_RING_ID` | 默认发布圈子 |
| `ZHIHU_OAUTH_APP_ID` | OAuth app id |
| `ZHIHU_OAUTH_APP_KEY` | OAuth app key |
| `ZHIHU_OAUTH_REDIRECT_URI` | 知乎开放平台登记的回调地址 |
| `ZHIHU_OAUTH_BASE_URL` | 默认 `https://openapi.zhihu.com` |
| `ZHIHU_ACCESS_TOKEN` | 可选，预置 OAuth token |
| `ZHIHU_ACCESS_SECRET` | Data Platform access secret |
| `ZHIHU_DATA_PLATFORM_BASE_URL` | 默认 `https://developer.zhihu.com` |
| `LLM_DEFAULT_MODEL` | 知乎直答默认模型 |

OAuth 回调地址当前必须是前端 callback 页：

```text
http://113.44.217.169:3000/auth/zhihu/callback
```

如果使用正式域名和 HTTPS，则改为：

```text
https://<domain>/auth/zhihu/callback
```

该值必须和知乎开放平台登记值完全一致。

### 2.5 LLM 配置

`llm-service` 实际读取：

| 环境变量 | 用途 |
| --- | --- |
| `LLM_PROVIDER_MODE` | `mock` / `zhihu` / `openai_compat` |
| `LLM_DEFAULT_MODEL` | 默认模型 |
| `LLM_CACHE_BACKEND` | 推荐 `redis` |
| `OPENAI_COMPAT_BASE_URL` | OpenAI-compatible base URL |
| `OPENAI_COMPAT_API_KEY` | OpenAI-compatible API key |
| `OPENAI_COMPAT_MODEL` | OpenAI-compatible 模型 |

注意：

```text
代码实际读取 OPENAI_COMPAT_*。
旧的 LLM_API_BASE_URL / LLM_API_KEY 只作为部署脚本兼容变量，不是 load_config() 的主读取项。
```

## 3. infra/.env 最小可运行清单

以当前服务器 IP 部署为例：

```bash
# 前端与 API
KANSHAN_GATEWAY_URL=http://113.44.217.169:8000
KANSHAN_ZHIHU_URL=http://113.44.217.169:8070
CORS_ORIGINS=http://113.44.217.169:3000,http://113.44.217.169:8000
NEXT_PUBLIC_ZHIHU_OAUTH_AUTO_START=1

# 运行模式
PROVIDER_MODE=live
STORAGE_BACKEND=postgres

# 数据库
POSTGRES_DB=kanshan
POSTGRES_USER=kanshan
POSTGRES_PASSWORD=<强密码>
DATABASE_URL=postgresql+psycopg://kanshan:<强密码>@postgres:5432/kanshan

# Redis
REDIS_URL=redis://redis:6379/0

# 知乎 Community / Data Platform
ZHIHU_APP_KEY=<community-app-key>
ZHIHU_APP_SECRET=<community-app-secret>
ZHIHU_ACCESS_SECRET=<developer-access-secret>
ZHIHU_COMMUNITY_BASE_URL=https://openapi.zhihu.com
ZHIHU_DATA_PLATFORM_BASE_URL=https://developer.zhihu.com
ZHIHU_DEFAULT_RING_ID=<ring-id>

# 知乎 OAuth
ZHIHU_OAUTH_APP_ID=<oauth-app-id>
ZHIHU_OAUTH_APP_KEY=<oauth-app-key>
ZHIHU_OAUTH_REDIRECT_URI=http://113.44.217.169:3000/auth/zhihu/callback
ZHIHU_OAUTH_BASE_URL=https://openapi.zhihu.com
OAUTH_STATE_SECRET=<随机长密钥>

# LLM
LLM_PROVIDER_MODE=openai_compat
LLM_DEFAULT_MODEL=zhida-thinking-1p5
LLM_CACHE_BACKEND=redis
OPENAI_COMPAT_BASE_URL=<openai-compatible-base-url>
OPENAI_COMPAT_API_KEY=<api-key>
OPENAI_COMPAT_MODEL=<model>

# 端口
FRONTEND_PORT=3000
API_GATEWAY_PORT=8000

```

如果暂时不用自有 LLM：

```bash

OPENAI_COMPAT_API_KEY=
OPENAI_COMPAT_MODEL=gpt-4o-mini
```

## 4. 一键部署脚本

脚本路径：

```text
scripts/deploy.sh
```

常用命令：

```bash
# 进入项目目录
cd /opt/kanshan

# 第一次部署，或修改了前端 / env 后
bash scripts/deploy.sh

# 只检查 infra/.env，不启动
bash scripts/deploy.sh --check

# 不重新构建，只重启容器
bash scripts/deploy.sh --no-build

# 查看已启动容器和 OAuth 链路
bash scripts/deploy.sh --doctor

# 查看日志
bash scripts/deploy.sh --logs

# 查看部署备份
bash scripts/deploy.sh --list-backups

# 回滚到最近一次部署前版本
bash scripts/deploy.sh --rollback latest

# 回滚到指定备份
bash scripts/deploy.sh --rollback deploy-20260514123000

# 停止
bash scripts/deploy.sh --down
```

脚本会做这些事：

```text
1. 检查 infra/.env 是否存在。
2. 检查 live 模式下知乎 OAuth / Data Platform 必填项。
3. 兼容旧变量：LLM_API_BASE_URL → OPENAI_COMPAT_BASE_URL。
4. 在 output/deploy-backups 下备份当前线上版本。
5. 先启动 postgres / redis。
6. 执行 scripts/sync_db_schema.sh --docker，同步旧数据库卷缺失字段。
7. docker compose build + up -d。
8. 等待 9 个后端服务健康。
9. 检查 profile-service 容器内是否能访问 http://zhihu-adapter:8070/health。
10. 请求 /api/v1/auth/zhihu/authorize，确认 OAuth 授权入口可用。
```

### 4.1 部署备份与一键回滚

每次执行以下命令前，脚本都会自动备份当前线上版本：

```bash
bash scripts/deploy.sh
bash scripts/deploy.sh --no-build
```

备份目录：

```text
output/deploy-backups/deploy-YYYYMMDDHHMMSS/
```

每个备份包含：

```text
metadata.env          当前 git commit、分支、是否有未提交变更
infra.env             当前线上 infra/.env 副本
compose-ps.txt        当前 compose 容器状态
compose-images.txt    当前 compose 镜像状态
images.tsv            当前运行容器镜像的备份标签
git-status.txt        当前 git 状态
working-tree.patch    未提交工作区 diff，仅用于人工排查
staged.patch          staged diff，仅用于人工排查
```

查看备份：

```bash
bash scripts/deploy.sh --list-backups
```

回滚到最近一次部署前版本：

```bash
bash scripts/deploy.sh --rollback latest
```

回滚到指定版本：

```bash
bash scripts/deploy.sh --rollback deploy-YYYYMMDDHHMMSS
```

回滚会执行：

```text
1. 恢复备份里的 infra/.env。
2. git checkout --detach 到备份记录的 commit。
3. 优先把备份镜像标签恢复到 compose 原镜像名，并 docker compose up -d --no-build。
4. 如果备份镜像不存在，则从备份 commit 重新 build。
5. 等待服务健康并执行 OAuth doctor。
```

限制：

```text
1. 回滚不删除数据库新增字段；当前 schema 同步是幂等加字段，设计上允许旧版本忽略多余字段。
2. 如果备份时线上工作区本身有未提交改动，脚本会保存 patch，但一键回滚默认不自动应用 patch。
3. 回滚后仓库处于 detached HEAD，后续重新部署新版本前建议执行 git checkout main && git pull。
```

## 5. 部署后验收

### 5.1 服务健康

```bash
curl -s http://113.44.217.169:8000/health
curl -s http://113.44.217.169:8070/health
curl -s http://113.44.217.169:8010/health
```

### 5.2 OAuth 授权入口

```bash
curl -s http://113.44.217.169:8000/api/v1/auth/zhihu/authorize
```

返回内容中应该包含：

```text
https://openapi.zhihu.com/authorize
redirect_uri=http%3A%2F%2F113.44.217.169%3A3000%2Fauth%2Fzhihu%2Fcallback
```

### 5.3 容器内网络

```bash
docker compose --env-file infra/.env -f infra/docker-compose.yml exec -T profile-service python - <<'PY'
import urllib.request
print(urllib.request.urlopen("http://zhihu-adapter:8070/health", timeout=5).read().decode())
PY
```

必须返回：

```json
{"status":"ok","service":"zhihu-adapter"}
```

### 5.4 浏览器 OAuth

```text
1. 打开 http://113.44.217.169:3000
2. 进入用户入口
3. 点击知乎授权
4. 知乎授权后回到 http://113.44.217.169:3000/auth/zhihu/callback
5. 页面跳到 /oauth/zhihu/success
6. 主页面收到 ticket 并进入 LLM 配置或工作台
```

### 5.5 数据库绑定记录

```bash
docker exec -it kanshan-postgres psql -U kanshan -d kanshan -c \
"select user_id, zhihu_uid, binding_status, bound_at, expired_at from profile.zhihu_bindings order by bound_at desc limit 5;"
```

应看到：

```text
binding_status = bound
```

### 5.6 旧数据库结构热修

如果线上是从旧版本数据库卷升级上来的，可能会出现：

```text
sqlalchemy.exc.ProgrammingError: column users.setup_state does not exist
```

这是因为 `profile.users` 表已经存在，PostgreSQL 初始化脚本不会再次执行。不要直接 `down -v` 删除数据，先执行当前幂等同步脚本：

```bash
bash scripts/sync_db_schema.sh --docker
```

`scripts/deploy.sh` 现在会在每次部署时自动执行该同步脚本。手动执行后，可重启相关服务：

```bash
docker compose --env-file infra/.env -f infra/docker-compose.yml restart profile-service api-gateway
```

## 6. 常见错误

| 错误 | 原因 | 修复 |
| --- | --- | --- |
| `ConnectionRefusedError 127.0.0.1:8070` | 容器内服务间 URL 错了 | `profile-service` 必须使用 `ZHIHU_ADAPTER_URL=http://zhihu-adapter:8070` |
| OAuth 授权入口 500 | `ZHIHU_OAUTH_*` 缺失或 zhihu-adapter 不通 | 运行 `bash scripts/deploy.sh --doctor` |
| 授权后无法访问页面 | 知乎 redirect URI 配错 | 改为前端地址 `/auth/zhihu/callback` |
| callback 页面调用 `/api` 失败 | Next rewrite 目标错 | frontend 容器必须有 `KANSHAN_GATEWAY_URL=http://api-gateway:8000` |
| 浏览器请求 `http://api-gateway:8000` | `NEXT_PUBLIC_KANSHAN_GATEWAY_URL` 配成了 Docker 内网地址 | 改成公网地址并重新 build frontend |
| LLM openai_compat 没注册 | 只填了 `LLM_API_*` | 填 `OPENAI_COMPAT_BASE_URL` 和 `OPENAI_COMPAT_API_KEY` |
| `column users.setup_state does not exist` | 旧 PostgreSQL 卷没有新字段 | 执行 `bash scripts/sync_db_schema.sh --docker`，不要先删除数据库卷 |

## 7. 当前部署建议

当前你使用 IP + 端口方式部署时，推荐先用：

```text
前端：http://113.44.217.169:3000
Gateway：http://113.44.217.169:8000
OAuth redirect_uri：http://113.44.217.169:3000/auth/zhihu/callback
```

等主流程跑通后，再切到正式域名 + HTTPS：

```text
前端：https://<domain>
Gateway：https://<domain>/api/*
OAuth redirect_uri：https://<domain>/auth/zhihu/callback
```

切换域名后必须：

```text
1. 修改知乎开放平台 redirect URI。
2. 修改 infra/.env 中 KANSHAN_GATEWAY_URL / CORS_ORIGINS / ZHIHU_OAUTH_REDIRECT_URI。
3. 重新执行 bash scripts/deploy.sh，让 frontend 重新 build。
```
